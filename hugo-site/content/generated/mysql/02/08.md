            <h2><a rel="bookmark" href="/mysql/02/08.html">第08章：事务与并发控制 ⭐⭐⭐⭐⭐</a></h2>
            <div class="meta"><span>MySQL教程 / 02</span></div>

          <div class="vditor-reset post__content"><h1>第08章：事务与并发控制 ⭐⭐⭐⭐⭐</h1>
<blockquote>
<p>事务是数据库的核心特性，理解事务和MVCC是成为MySQL专家的必备知识</p>
</blockquote>
<h2>8.1 事务概述</h2>
<h3>8.1.1 什么是事务</h3>
<p><strong>事务（Transaction）</strong>：一组SQL语句的集合，要么全部成功，要么全部失败。</p>
<p><strong>经典案例：银行转账</strong></p>
<pre><code class="language-sql">-- 转账操作必须是一个事务
START TRANSACTION;
<p>-- 步骤1：A账户扣款
UPDATE accounts SET balance = balance - 100 WHERE user_id = 1;</p>
<p>-- 步骤2：B账户加款
UPDATE accounts SET balance = balance + 100 WHERE user_id = 2;</p>
<p>-- 提交事务
COMMIT;
</code></pre></p>
<p>如果步骤1成功，步骤2失败，会导致钱丢失！事务保证两个操作要么都成功，要么都失败。</p>
<hr>
<h2>8.2 ACID特性</h2>
<p>事务必须满足ACID四大特性：</p>
<h3>8.2.1 原子性（Atomicity）</h3>
<p><strong>定义：</strong> 事务是最小的执行单位，不可分割。要么全部成功，要么全部失败。</p>
<p><strong>实现机制：</strong> undo log（回滚日志）</p>
<p><strong>示例：</strong></p>
<pre><code class="language-sql">START TRANSACTION;
<p>INSERT INTO users (username) VALUES ('user1');
INSERT INTO users (username) VALUES ('user2');
-- 这里发生错误
INSERT INTO users (username) VALUES (NULL);  -- 违反NOT NULL约束</p>
<p>ROLLBACK;  -- 回滚，前面的INSERT也会撤销
</code></pre></p>
<h3>8.2.2 一致性（Consistency）</h3>
<p><strong>定义：</strong> 事务执行前后，数据库从一个一致性状态转换到另一个一致性状态。</p>
<p><strong>示例：</strong></p>
<pre><code class="language-sql">-- 转账前：A=1000, B=500, 总额=1500
START TRANSACTION;
UPDATE accounts SET balance = balance - 100 WHERE user_id = 1;  -- A=900
UPDATE accounts SET balance = balance + 100 WHERE user_id = 2;  -- B=600
COMMIT;
-- 转账后：A=900, B=600, 总额=1500（保持一致）
</code></pre>
<h3>8.2.3 隔离性（Isolation）</h3>
<p><strong>定义：</strong> 多个事务并发执行时，一个事务的执行不应影响其他事务。</p>
<p><strong>问题：</strong> 如果没有隔离性，会出现脏读、不可重复读、幻读等问题。</p>
<p><strong>实现机制：</strong> 锁机制 + MVCC（多版本并发控制）</p>
<h3>8.2.4 持久性（Durability）</h3>
<p><strong>定义：</strong> 事务一旦提交，对数据库的改变是永久性的。</p>
<p><strong>实现机制：</strong> redo log（重做日志）</p>
<p><strong>示例：</strong></p>
<pre><code class="language-sql">START TRANSACTION;
INSERT INTO users (username) VALUES ('user1');
COMMIT;
<p>-- 即使此时MySQL崩溃，重启后数据仍然存在
</code></pre></p>
<hr>
<h2>8.3 事务的使用</h2>
<h3>8.3.1 开启事务</h3>
<pre><code class="language-sql">-- 方法1
START TRANSACTION;
<p>-- 方法2
BEGIN;</p>
<p>-- 方法3：开启只读事务
START TRANSACTION READ ONLY;</p>
<p>-- 方法4：开启读写事务
START TRANSACTION READ WRITE;
</code></pre></p>
<h3>8.3.2 提交事务</h3>
<pre><code class="language-sql">COMMIT;
</code></pre>
<h3>8.3.3 回滚事务</h3>
<pre><code class="language-sql">ROLLBACK;
</code></pre>
<h3>8.3.4 保存点</h3>
<pre><code class="language-sql">START TRANSACTION;
<p>INSERT INTO users (username) VALUES ('user1');</p>
<p>-- 设置保存点
SAVEPOINT sp1;</p>
<p>INSERT INTO users (username) VALUES ('user2');</p>
<p>-- 回滚到保存点（只撤销user2）
ROLLBACK TO sp1;</p>
<p>INSERT INTO users (username) VALUES ('user3');</p>
<p>COMMIT;
-- 结果：插入了user1和user3
</code></pre></p>
<h3>8.3.5 自动提交</h3>
<pre><code class="language-sql">-- 查看自动提交状态
SHOW VARIABLES LIKE 'autocommit';
<p>-- 关闭自动提交
SET autocommit = 0;</p>
<p>-- 此时每个语句都需要手动COMMIT
INSERT INTO users (username) VALUES ('test');
COMMIT;</p>
<p>-- 开启自动提交（默认）
SET autocommit = 1;
</code></pre></p>
<hr>
<h2>8.4 事务隔离级别 ⭐⭐⭐⭐⭐</h2>
<h3>8.4.1 并发问题</h3>
<p><strong>1. 脏读（Dirty Read）</strong></p>
<p>事务A读取了事务B未提交的数据，如果B回滚，A读到的就是脏数据。</p>
<pre><code class="language-sql">-- 时间线：
-- 事务A                          事务B
START TRANSACTION;               START TRANSACTION;
                                 UPDATE users SET age = 30 WHERE id = 1;
SELECT age FROM users WHERE id = 1;  -- 读到30（脏数据）
                                 ROLLBACK;  -- B回滚了
-- A读到的30是无效的
</code></pre>
<p><strong>2. 不可重复读（Non-Repeatable Read）</strong></p>
<p>事务A多次读取同一数据，结果不一致（因为事务B修改并提交了）。</p>
<pre><code class="language-sql">-- 时间线：
-- 事务A                          事务B
START TRANSACTION;               
SELECT age FROM users WHERE id = 1;  -- 读到25
                                 START TRANSACTION;
                                 UPDATE users SET age = 30 WHERE id = 1;
                                 COMMIT;
SELECT age FROM users WHERE id = 1;  -- 读到30（不一致）
COMMIT;
</code></pre>
<p><strong>3. 幻读（Phantom Read）</strong></p>
<p>事务A多次查询，结果集的行数不一致（因为事务B插入或删除了数据）。</p>
<pre><code class="language-sql">-- 时间线：
-- 事务A                          事务B
START TRANSACTION;               
SELECT COUNT(*) FROM users WHERE age &gt; 20;  -- 返回10行
                                 START TRANSACTION;
                                 INSERT INTO users (age) VALUES (25);
                                 COMMIT;
SELECT COUNT(*) FROM users WHERE age &gt; 20;  -- 返回11行（幻读）
COMMIT;
</code></pre>

<h3>8.4.3 查看和设置隔离级别</h3>
<pre><code class="language-sql">-- 查看全局隔离级别
SELECT @@global.transaction_isolation;
<p>-- 查看当前会话隔离级别
SELECT @@transaction_isolation;
-- 或
SHOW VARIABLES LIKE 'transaction_isolation';</p>
<p>-- 设置全局隔离级别
SET GLOBAL transaction_isolation = 'READ-COMMITTED';</p>
<p>-- 设置当前会话隔离级别
SET SESSION transaction_isolation = 'READ-COMMITTED';</p>
<p>-- 设置下一个事务的隔离级别
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;
</code></pre></p>
<h3>8.4.4 隔离级别详解</h3>
<h4>1. READ UNCOMMITTED（读未提交）</h4>
<p><strong>特点：</strong> 可以读取未提交的数据（脏读）</p>
<p><strong>演示脏读：</strong></p>
<pre><code class="language-sql">-- 终端1（事务A）
SET SESSION transaction_isolation = 'READ-UNCOMMITTED';
START TRANSACTION;
SELECT balance FROM accounts WHERE user_id = 1;  -- 1000
<p>-- 终端2（事务B）
START TRANSACTION;
UPDATE accounts SET balance = 500 WHERE user_id = 1;
-- 注意：没有COMMIT</p>
<p>-- 终端1（事务A）
SELECT balance FROM accounts WHERE user_id = 1;  -- 500（脏读！）</p>
<p>-- 终端2（事务B）
ROLLBACK;  -- 回滚</p>
<p>-- 终端1（事务A）
SELECT balance FROM accounts WHERE user_id = 1;  -- 1000（之前读到的500是脏数据）
</code></pre></p>
<p><strong>使用场景：</strong> 几乎不使用</p>
<h4>2. READ COMMITTED（读已提交）</h4>
<p><strong>特点：</strong> 只能读取已提交的数据，避免脏读</p>
<p><strong>演示不可重复读：</strong></p>
<pre><code class="language-sql">-- 终端1（事务A）
SET SESSION transaction_isolation = 'READ-COMMITTED';
START TRANSACTION;
SELECT balance FROM accounts WHERE user_id = 1;  -- 1000
<p>-- 终端2（事务B）
START TRANSACTION;
UPDATE accounts SET balance = 500 WHERE user_id = 1;
COMMIT;  -- 提交</p>
<p>-- 终端1（事务A）
SELECT balance FROM accounts WHERE user_id = 1;  -- 500（不可重复读）
COMMIT;
</code></pre></p>
<p><strong>使用场景：</strong> Oracle、PostgreSQL默认级别</p>
<h4>3. REPEATABLE READ（可重复读）⭐ MySQL默认</h4>
<p><strong>特点：</strong> 同一事务内多次读取结果一致</p>
<p><strong>演示：</strong></p>
<pre><code class="language-sql">-- 终端1（事务A）
SET SESSION transaction_isolation = 'REPEATABLE-READ';
START TRANSACTION;
SELECT balance FROM accounts WHERE user_id = 1;  -- 1000
<p>-- 终端2（事务B）
START TRANSACTION;
UPDATE accounts SET balance = 500 WHERE user_id = 1;
COMMIT;</p>
<p>-- 终端1（事务A）
SELECT balance FROM accounts WHERE user_id = 1;  -- 仍然是1000（可重复读）
COMMIT;</p>
<p>-- 终端1（新事务）
SELECT balance FROM accounts WHERE user_id = 1;  -- 500（新事务读到最新值）
</code></pre></p>
<p><strong>MySQL的REPEATABLE READ解决了幻读问题！</strong>（通过Next-Key Lock）</p>
<h4>4. SERIALIZABLE（串行化）</h4>
<p><strong>特点：</strong> 最高隔离级别，完全串行化执行</p>
<p><strong>演示：</strong></p>
<pre><code class="language-sql">-- 终端1（事务A）
SET SESSION transaction_isolation = 'SERIALIZABLE';
START TRANSACTION;
SELECT * FROM users WHERE id = 1;
<p>-- 终端2（事务B）
START TRANSACTION;
UPDATE users SET age = 30 WHERE id = 1;  -- 阻塞，等待事务A提交</p>
<p>-- 终端1（事务A）
COMMIT;  -- 提交后，事务B才能执行</p>
<p>-- 终端2（事务B）
-- UPDATE执行成功
COMMIT;
</code></pre></p>
<p><strong>使用场景：</strong> 对数据一致性要求极高的场景</p>
<hr>
<h2>8.5 MVCC多版本并发控制 ⭐⭐⭐⭐⭐</h2>
<h3>8.5.1 MVCC概述</h3>
<p><strong>MVCC（Multi-Version Concurrency Control）</strong>：多版本并发控制</p>
<p><strong>核心思想：</strong> 为每个事务提供数据的一个快照，不同事务看到不同版本的数据。</p>
<p><strong>优点：</strong></p>
<ul>
<li>读不加锁，写不阻塞读</li>
<li>提高并发性能</li>
<li>解决脏读、不可重复读问题</li>
</ul>
<p><strong>适用范围：</strong> InnoDB存储引擎，READ COMMITTED和REPEATABLE READ隔离级别</p>
<h3>8.5.2 MVCC实现原理</h3>
<h4>1. 隐藏列</h4>
<p>InnoDB为每行记录添加了3个隐藏列：</p>
<table>
<thead>
<tr>
<th>隐藏列</th>
<th>长度</th>
<th>说明</th>
</tr>
</thead>
<tbody>
<tr>
<td><strong>DB_TRX_ID</strong></td>
<td>6字节</td>
<td>最后修改该行的事务ID</td>
</tr>
<tr>
<td><strong>DB_ROLL_PTR</strong></td>
<td>7字节</td>
<td>回滚指针，指向undo log</td>
</tr>
<tr>
<td><strong>DB_ROW_ID</strong></td>
<td>6字节</td>
<td>行ID（如果没有主键）</td>
</tr>
</tbody>
</table>
<p><strong>示例：</strong></p>
<pre><code class="language-sql">-- 用户看到的表
CREATE TABLE users (
    id INT PRIMARY KEY,
    name VARCHAR(50),
    age INT
);
<p>-- InnoDB实际存储的结构
-- id | name | age | DB_TRX_ID | DB_ROLL_PTR | DB_ROW_ID
</code></pre></p>
<h4>2. undo log版本链</h4>
<p>每次修改数据时，旧版本会保存到undo log中，形成版本链。</p>
<p><strong>示例：</strong></p>
<pre><code class="language-sql">-- 初始数据：id=1, name='张三', age=25
<p>-- 事务100修改
UPDATE users SET age = 26 WHERE id = 1;
-- 当前版本：id=1, name='张三', age=26, DB_TRX_ID=100
-- undo log：id=1, name='张三', age=25, DB_TRX_ID=初始</p>
<p>-- 事务200修改
UPDATE users SET age = 27 WHERE id = 1;
-- 当前版本：id=1, name='张三', age=27, DB_TRX_ID=200
-- undo log链：
--   版本2：age=26, DB_TRX_ID=100
--   版本1：age=25, DB_TRX_ID=初始
</code></pre></p>
<p><strong>版本链结构：</strong></p>
<pre><code>当前版本 → undo log版本2 → undo log版本1 → ...
age=27      age=26           age=25
TRX_ID=200  TRX_ID=100       TRX_ID=初始
</code></pre>
<h4>3. Read View（读视图）</h4>
<p><strong>Read View</strong>：事务开始时创建的一致性视图，决定能看到哪些版本的数据。</p>
<p><strong>Read View包含：</strong></p>
<ul>
<li><code>m_ids</code>：当前活跃的事务ID列表</li>
<li><code>min_trx_id</code>：最小活跃事务ID</li>
<li><code>max_trx_id</code>：下一个要分配的事务ID</li>
<li><code>creator_trx_id</code>：当前事务ID</li>
</ul>
<p><strong>可见性判断规则：</strong></p>
<pre><code>如果 DB_TRX_ID &lt; min_trx_id：
    可见（已提交的旧事务）
<p>如果 DB_TRX_ID &gt;= max_trx_id：
不可见（未来的事务）</p>
<p>如果 min_trx_id &lt;= DB_TRX_ID &lt; max_trx_id：
如果 DB_TRX_ID 在 m_ids 中：
不可见（未提交的事务）
否则：
可见（已提交的事务）</p>
<p>如果 DB_TRX_ID == creator_trx_id：
可见（自己的修改）
</code></pre></p>
<h3>8.5.3 MVCC工作流程</h3>
<p><strong>场景：</strong> 两个事务并发执行</p>
<pre><code class="language-sql">-- 初始数据
INSERT INTO users (id, name, age) VALUES (1, '张三', 25);
-- DB_TRX_ID = 50
<p>-- 时间线：
-- 事务A (TRX_ID=100)              事务B (TRX_ID=200)
START TRANSACTION;
SELECT age FROM users WHERE id=1; -- 读到25
START TRANSACTION;
UPDATE users SET age=30 WHERE id=1;
-- DB_TRX_ID = 200
SELECT age FROM users WHERE id=1; -- 读到25（MVCC）
COMMIT;
SELECT age FROM users WHERE id=1; -- 读到25（REPEATABLE READ）
COMMIT;</p>
<p>-- 新事务
START TRANSACTION;
SELECT age FROM users WHERE id=1; -- 读到30
</code></pre></p>
<p><strong>详细分析：</strong></p>
<ol>
<li>
<p><strong>事务A第一次查询（age=25）</strong></p>
<ul>
<li>创建Read View：m_ids=[100], min_trx_id=100, max_trx_id=101</li>
<li>当前版本：DB_TRX_ID=50 &lt; min_trx_id=100 → 可见</li>
<li>返回：age=25</li>
</ul>
</li>
<li>
<p><strong>事务B更新（age=30）</strong></p>
<ul>
<li>修改数据：DB_TRX_ID=200</li>
<li>创建undo log：保存旧版本（age=25, DB_TRX_ID=50）</li>
</ul>
</li>
<li>
<p><strong>事务A第二次查询（age=25）</strong></p>
<ul>
<li>使用之前的Read View：m_ids=[100], min_trx_id=100, max_trx_id=101</li>
<li>当前版本：DB_TRX_ID=200 &gt;= max_trx_id=101 → 不可见</li>
<li>沿着undo log链查找：
<ul>
<li>版本1：DB_TRX_ID=50 &lt; min_trx_id=100 → 可见</li>
</ul>
</li>
<li>返回：age=25（可重复读）</li>
</ul>
</li>
<li>
<p><strong>事务B提交后，事务A第三次查询（age=25）</strong></p>
<ul>
<li>仍使用之前的Read View（REPEATABLE READ）</li>
<li>返回：age=25</li>
</ul>
</li>
<li>
<p><strong>新事务查询（age=30）</strong></p>
<ul>
<li>创建新的Read View：m_ids=[], min_trx_id=201, max_trx_id=201</li>
<li>当前版本：DB_TRX_ID=200 &lt; min_trx_id=201 → 可见</li>
<li>返回：age=30</li>
</ul>
</li>
</ol>
<h3>8.5.4 READ COMMITTED vs REPEATABLE READ</h3>
<p><strong>READ COMMITTED：</strong></p>
<ul>
<li>每次SELECT都创建新的Read View</li>
<li>能读到其他事务已提交的数据</li>
<li>不可重复读</li>
</ul>
<p><strong>REPEATABLE READ：</strong></p>
<ul>
<li>事务开始时创建Read View，之后一直使用</li>
<li>读到的数据始终一致</li>
<li>可重复读</li>
</ul>
<p><strong>演示：</strong></p>
<pre><code class="language-sql">-- READ COMMITTED
-- 事务A                          事务B
SET SESSION transaction_isolation = 'READ-COMMITTED';
START TRANSACTION;
SELECT age FROM users WHERE id=1; -- 25（创建Read View 1）
                                  START TRANSACTION;
                                  UPDATE users SET age=30 WHERE id=1;
                                  COMMIT;
SELECT age FROM users WHERE id=1; -- 30（创建Read View 2，读到已提交数据）
<p>-- REPEATABLE READ
-- 事务A                          事务B
SET SESSION transaction_isolation = 'REPEATABLE-READ';
START TRANSACTION;
SELECT age FROM users WHERE id=1; -- 25（创建Read View）
START TRANSACTION;
UPDATE users SET age=30 WHERE id=1;
COMMIT;
SELECT age FROM users WHERE id=1; -- 25（使用同一个Read View）
</code></pre></p>
<h3>8.5.5 当前读 vs 快照读</h3>
<p><strong>快照读（Snapshot Read）：</strong></p>
<ul>
<li>普通的SELECT语句</li>
<li>读取的是快照版本（通过MVCC）</li>
<li>不加锁</li>
</ul>
<pre><code class="language-sql">SELECT * FROM users WHERE id = 1;
</code></pre>
<p><strong>当前读（Current Read）：</strong></p>
<ul>
<li>SELECT ... FOR UPDATE</li>
<li>SELECT ... LOCK IN SHARE MODE</li>
<li>INSERT、UPDATE、DELETE</li>
<li>读取的是最新版本</li>
<li>加锁</li>
</ul>
<pre><code class="language-sql">-- 当前读（加排他锁）
SELECT * FROM users WHERE id = 1 FOR UPDATE;
<p>-- 当前读（加共享锁）
SELECT * FROM users WHERE id = 1 LOCK IN SHARE MODE;</p>
<p>-- 当前读
UPDATE users SET age = 30 WHERE id = 1;
DELETE FROM users WHERE id = 1;
</code></pre></p>
<p><strong>演示：</strong></p>
<pre><code class="language-sql">-- 事务A                          事务B
START TRANSACTION;
SELECT age FROM users WHERE id=1; -- 25（快照读）
                                  START TRANSACTION;
                                  UPDATE users SET age=30 WHERE id=1;
                                  COMMIT;
SELECT age FROM users WHERE id=1; -- 25（快照读，MVCC）
SELECT age FROM users WHERE id=1 FOR UPDATE; -- 30（当前读）
COMMIT;
</code></pre>
<h3>8.5.6 MVCC解决的问题</h3>
<p><strong>1. 解决脏读</strong></p>
<ul>
<li>通过Read View，只能读到已提交的事务</li>
<li>未提交事务的修改不可见</li>
</ul>
<p><strong>2. 解决不可重复读（REPEATABLE READ）</strong></p>
<ul>
<li>事务内使用同一个Read View</li>
<li>读到的数据始终一致</li>
</ul>
<p><strong>3. 提高并发性能</strong></p>
<ul>
<li>读不加锁</li>
<li>写不阻塞读</li>
<li>多个事务可以并发读取</li>
</ul>

<h2>8.6 实战案例</h2>
<h3>案例1：转账事务</h3>
<pre><code class="language-sql">-- 创建账户表
CREATE TABLE accounts (
    id INT PRIMARY KEY AUTO_INCREMENT,
    user_id INT NOT NULL UNIQUE,
    balance DECIMAL(10, 2) NOT NULL DEFAULT 0,
    version INT NOT NULL DEFAULT 0  -- 乐观锁版本号
) ENGINE=InnoDB;
<p>-- 插入测试数据
INSERT INTO accounts (user_id, balance) VALUES (1, 1000.00), (2, 500.00);</p>
<p>-- 转账：用户1给用户2转账100元
START TRANSACTION;</p>
<p>-- 检查余额
SELECT balance FROM accounts WHERE user_id = 1 FOR UPDATE;  -- 当前读，加锁</p>
<p>-- 扣款
UPDATE accounts SET balance = balance - 100 WHERE user_id = 1 AND balance &gt;= 100;</p>
<p>-- 检查是否扣款成功
IF ROW_COUNT() = 0 THEN
ROLLBACK;
SELECT '余额不足' AS message;
ELSE
-- 加款
UPDATE accounts SET balance = balance + 100 WHERE user_id = 2;
COMMIT;
SELECT '转账成功' AS message;
END IF;
</code></pre></p>
<h3>案例2：秒杀系统</h3>
<pre><code class="language-sql">-- 创建商品表
CREATE TABLE products (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(100),
    stock INT NOT NULL DEFAULT 0,
    version INT NOT NULL DEFAULT 0
) ENGINE=InnoDB;
<p>-- 插入测试数据
INSERT INTO products (name, stock) VALUES ('iPhone 15', 100);</p>
<p>-- 方法1：悲观锁（FOR UPDATE）
START TRANSACTION;</p>
<p>-- 查询库存（加排他锁）
SELECT stock FROM products WHERE id = 1 FOR UPDATE;</p>
<p>-- 减库存
UPDATE products SET stock = stock - 1 WHERE id = 1 AND stock &gt; 0;</p>
<p>-- 检查是否成功
IF ROW_COUNT() = 0 THEN
ROLLBACK;
SELECT '库存不足' AS message;
ELSE
-- 创建订单
INSERT INTO orders (product_id, user_id) VALUES (1, 100);
COMMIT;
SELECT '秒杀成功' AS message;
END IF;</p>
<p>-- 方法2：乐观锁（版本号）
START TRANSACTION;</p>
<p>-- 查询库存和版本号
SELECT stock, version FROM products WHERE id = 1;
-- 假设：stock=10, version=5</p>
<p>-- 减库存（带版本号检查）
UPDATE products
SET stock = stock - 1, version = version + 1
WHERE id = 1 AND version = 5 AND stock &gt; 0;</p>
<p>-- 检查是否成功
IF ROW_COUNT() = 0 THEN
ROLLBACK;
SELECT '秒杀失败，请重试' AS message;
ELSE
-- 创建订单
INSERT INTO orders (product_id, user_id) VALUES (1, 100);
COMMIT;
SELECT '秒杀成功' AS message;
END IF;
</code></pre></p>
<h3>案例3：防止超卖</h3>
<pre><code class="language-sql">-- 错误示例：可能超卖
START TRANSACTION;
SELECT stock FROM products WHERE id = 1;  -- 快照读，stock=10
-- 此时其他事务可能已经减库存
UPDATE products SET stock = stock - 1 WHERE id = 1;  -- 可能变成负数
COMMIT;
<p>-- 正确示例1：使用FOR UPDATE
START TRANSACTION;
SELECT stock FROM products WHERE id = 1 FOR UPDATE;  -- 当前读，加锁
UPDATE products SET stock = stock - 1 WHERE id = 1 AND stock &gt; 0;
COMMIT;</p>
<p>-- 正确示例2：直接UPDATE带条件
START TRANSACTION;
UPDATE products SET stock = stock - 1 WHERE id = 1 AND stock &gt; 0;
IF ROW_COUNT() = 0 THEN
ROLLBACK;
ELSE
COMMIT;
END IF;
</code></pre></p>
<h3>案例4：分布式事务（两阶段提交）</h3>
<pre><code class="language-sql">-- 场景：跨数据库转账
<p>-- 数据库1：扣款
START TRANSACTION;
UPDATE accounts SET balance = balance - 100 WHERE user_id = 1;
-- 准备提交（不真正提交）
XA PREPARE 'trx1';</p>
<p>-- 数据库2：加款
START TRANSACTION;
UPDATE accounts SET balance = balance + 100 WHERE user_id = 2;
-- 准备提交
XA PREPARE 'trx2';</p>
<p>-- 如果两个都准备成功，则提交
XA COMMIT 'trx1';
XA COMMIT 'trx2';</p>
<p>-- 如果任何一个失败，则回滚
XA ROLLBACK 'trx1';
XA ROLLBACK 'trx2';
</code></pre></p>

<h2>8.7 性能优化建议</h2>
<h3>8.7.1 选择合适的隔离级别</h3>
<pre><code class="language-sql">-- 对一致性要求不高的场景
SET SESSION transaction_isolation = 'READ-COMMITTED';
<p>-- 对一致性要求高的场景（默认）
SET SESSION transaction_isolation = 'REPEATABLE-READ';</p>
<p>-- 极高一致性要求
SET SESSION transaction_isolation = 'SERIALIZABLE';
</code></pre></p>
<h3>8.7.2 缩短事务时间</h3>
<pre><code class="language-sql">-- ❌ 错误：事务时间过长
START TRANSACTION;
SELECT * FROM users;  -- 查询
-- 执行复杂的业务逻辑（耗时）
-- 调用外部API（耗时）
UPDATE users SET status = 1 WHERE id = 1;
COMMIT;
<p>-- ✅ 正确：只在必要时使用事务
-- 先执行业务逻辑
-- 再开启事务
START TRANSACTION;
UPDATE users SET status = 1 WHERE id = 1;
COMMIT;
</code></pre></p>
<h3>8.7.3 避免大事务</h3>
<pre><code class="language-sql">-- ❌ 错误：一次更新100万行
START TRANSACTION;
UPDATE users SET status = 1 WHERE create_time &lt; '2024-01-01';  -- 100万行
COMMIT;
<p>-- ✅ 正确：分批更新
WHILE (SELECT COUNT(*) FROM users WHERE create_time &lt; '2024-01-01' AND status = 0) &gt; 0 DO
START TRANSACTION;
UPDATE users SET status = 1
WHERE create_time &lt; '2024-01-01' AND status = 0
LIMIT 1000;
COMMIT;
SELECT SLEEP(0.1);  -- 暂停一下
END WHILE;
</code></pre></p>
<h3>8.7.4 使用乐观锁代替悲观锁</h3>
<pre><code class="language-sql">-- 悲观锁：并发低
START TRANSACTION;
SELECT * FROM products WHERE id = 1 FOR UPDATE;  -- 加锁
UPDATE products SET stock = stock - 1 WHERE id = 1;
COMMIT;
<p>-- 乐观锁：并发高
START TRANSACTION;
SELECT stock, version FROM products WHERE id = 1;
UPDATE products SET stock = stock - 1, version = version + 1
WHERE id = 1 AND version = #{old_version};
IF ROW_COUNT() = 0 THEN
ROLLBACK;  -- 版本冲突，重试
ELSE
COMMIT;
END IF;
</code></pre></p>
