          <div class="vditor-reset post__content"><h1>第06章：索引原理与优化 ⭐⭐⭐⭐⭐</h1>
<blockquote>
<p>索引是MySQL性能优化的核心，必须深入理解</p>
</blockquote>
<h2>6.1 索引的概念</h2>
<h3>6.1.1 什么是索引</h3>
<p>索引是一种数据结构，用于帮助MySQL高效地查找数据。</p>
<p><strong>类比：</strong></p>
<ul>
<li>没有索引：像在字典中逐页查找一个单词（全表扫描）</li>
<li>有索引：像使用字典的目录快速定位（索引查找）</li>
</ul>
<h3>6.1.2 索引的优缺点</h3>
<p><strong>优点：</strong></p>
<ul>
<li>✅ 大大加快数据查询速度</li>
<li>✅ 加速表与表之间的连接</li>
<li>✅ 减少分组和排序的时间</li>
<li>✅ 使用覆盖索引可以避免回表</li>
</ul>
<p><strong>缺点：</strong></p>
<ul>
<li>❌ 占用额外的磁盘空间</li>
<li>❌ 降低INSERT、UPDATE、DELETE的速度</li>
<li>❌ 需要维护成本</li>
</ul>
<p><strong>何时使用索引：</strong></p>
<ul>
<li>✅ WHERE条件字段</li>
<li>✅ ORDER BY字段</li>
<li>✅ GROUP BY字段</li>
<li>✅ JOIN的ON字段</li>
<li>✅ 经常查询的字段</li>
</ul>
<p><strong>何时不使用索引：</strong></p>
<ul>
<li>❌ 数据量很小的表（几百行）</li>
<li>❌ 频繁更新的字段</li>
<li>❌ 区分度很低的字段（如性别）</li>
<li>❌ WHERE条件中用不到的字段</li>
</ul>
<hr>
<h2>6.2 索引的数据结构</h2>
<h3>6.2.1 为什么使用B+Tree</h3>
<p><strong>常见数据结构对比：</strong></p>
<table>
<thead>
<tr>
<th>数据结构</th>
<th>查询时间复杂度</th>
<th>缺点</th>
</tr>
</thead>
<tbody>
<tr>
<td>数组</td>
<td>O(n)</td>
<td>查询慢</td>
</tr>
<tr>
<td>链表</td>
<td>O(n)</td>
<td>查询慢</td>
</tr>
<tr>
<td>哈希表</td>
<td>O(1)</td>
<td>不支持范围查询、排序</td>
</tr>
<tr>
<td>二叉搜索树</td>
<td>O(log n)</td>
<td>可能退化为链表</td>
</tr>
<tr>
<td>平衡二叉树（AVL）</td>
<td>O(log n)</td>
<td>树太高，磁盘I/O次数多</td>
</tr>
<tr>
<td>红黑树</td>
<td>O(log n)</td>
<td>树太高，磁盘I/O次数多</td>
</tr>
<tr>
<td><strong>B+Tree</strong></td>
<td><strong>O(log n)</strong></td>
<td>✅ 树矮、范围查询快</td>
</tr>
</tbody>
</table>
<p><strong>为什么不用哈希表？</strong></p>
<pre><code class="language-sql">-- 哈希表适合等值查询
SELECT * FROM users WHERE id = 100;  -- ✅ 快
<p>-- 但不支持范围查询
SELECT * FROM users WHERE id &gt; 100;  -- ❌ 慢
SELECT * FROM users ORDER BY id;     -- ❌ 慢
</code></pre></p>
<p><strong>为什么不用二叉树？</strong></p>
<pre><code>假设100万条数据：
- 二叉树高度：log₂(1000000) ≈ 20层
- 需要20次磁盘I/O
<p>B+Tree（假设每个节点1000个key）：</p>
<ul>
<li>B+Tree高度：log₁₀₀₀(1000000) ≈ 3层</li>
<li>只需要3次磁盘I/O
</code></pre></li>
</ul>
<h3>6.2.2 B+Tree结构详解</h3>
<p><strong>B+Tree特点：</strong></p>
<ol>
<li>所有数据都存储在叶子节点</li>
<li>叶子节点之间有指针连接（双向链表）</li>
<li>非叶子节点只存储索引，不存储数据</li>
<li>树的高度很低（通常3-4层）</li>
</ol>
<p><strong>B+Tree示例：</strong></p>
<pre><code>                    [50, 100]           ← 根节点（非叶子节点）
                   /    |    \
                  /     |     \
          [10,20,30] [60,70] [110,120]  ← 非叶子节点
           /  |  |  \
          /   |  |   \
    [1-9][10-19][20-29][30-39] ... [110-119][120-129]  ← 叶子节点（存储数据）
      ↔     ↔      ↔      ↔              ↔       ↔     ← 双向链表
</code></pre>
<p><strong>查询过程：</strong></p>
<pre><code class="language-sql">-- 查询 id=25 的记录
SELECT * FROM users WHERE id = 25;
<p>-- 查询过程：
-- 1. 从根节点开始：25 &lt; 50，走左边
-- 2. 到达 [10,20,30]：20 &lt; 25 &lt; 30，走中间
-- 3. 到达叶子节点 [20-29]，找到 id=25 的数据
-- 总共3次磁盘I/O
</code></pre></p>
<p><strong>范围查询：</strong></p>
<pre><code class="language-sql">-- 查询 id 在 20-60 之间的记录
SELECT * FROM users WHERE id BETWEEN 20 AND 60;
<p>-- 查询过程：
-- 1. 找到 id=20 的叶子节点
-- 2. 通过叶子节点的链表指针，顺序扫描到 id=60
-- 非常高效！
</code></pre></p>
<h3>6.2.3 InnoDB的B+Tree</h3>
<p><strong>InnoDB的两种索引：</strong></p>
<p><strong>1. 聚簇索引（Clustered Index）：</strong></p>
<ul>
<li>叶子节点存储完整的行数据</li>
<li>一个表只有一个聚簇索引</li>
<li>通常是主键索引</li>
</ul>
<p><strong>结构：</strong></p>
<pre><code>聚簇索引（主键索引）：
                [50, 100]
               /    |    \
        [10,20,30] [60,70] [110,120]
         /  |  |  \
    [id=1,  [id=10, [id=20, [id=30,
     name=  name=   name=   name=
     age=   age=    age=    age=
     ...]   ...]    ...]    ...]
     ↔      ↔       ↔       ↔
</code></pre>
<p><strong>2. 二级索引（Secondary Index）：</strong></p>
<ul>
<li>叶子节点存储索引列的值 + 主键值</li>
<li>一个表可以有多个二级索引</li>
</ul>
<p><strong>结构：</strong></p>
<pre><code>二级索引（name索引）：
                ['李', '王']
               /     |     \
        ['张三','张四'] ['李四','李五'] ['王五','王六']
         /      |      \
    [name='张三', [name='张四', [name='李四',
     id=1]        id=5]         id=10]
     ↔            ↔             ↔
</code></pre>
<p><strong>回表查询：</strong></p>
<pre><code class="language-sql">-- 使用二级索引查询
SELECT * FROM users WHERE name = '张三';
<p>-- 查询过程：
-- 1. 在name索引中找到 name='张三'，得到 id=1
-- 2. 回到聚簇索引，根据 id=1 查找完整数据
-- 这就是&quot;回表&quot;
</code></pre></p>
<p><strong>覆盖索引（避免回表）：</strong></p>
<pre><code class="language-sql">-- 创建联合索引
CREATE INDEX idx_name_age ON users(name, age);
<p>-- 只查询索引中的字段，不需要回表
SELECT name, age FROM users WHERE name = '张三';
-- Extra: Using index（覆盖索引）
</code></pre></p>
<h3>6.2.4 MyISAM的B+Tree</h3>
<p><strong>MyISAM的索引：</strong></p>
<ul>
<li>所有索引都是非聚簇索引</li>
<li>叶子节点存储索引列的值 + 数据文件的地址指针</li>
<li>主键索引和二级索引结构相同</li>
</ul>
<p><strong>结构：</strong></p>
<pre><code>MyISAM索引：
                [50, 100]
               /    |    \
        [10,20,30] [60,70] [110,120]
         /  |  |  \
    [id=1,  [id=10, [id=20, [id=30,
     addr]   addr]   addr]   addr]
     ↔       ↔       ↔       ↔
<p>addr: 数据文件中的物理地址
</code></pre></p>
<hr>
<h2>6.3 索引类型</h2>
<h3>6.3.1 按功能分类</h3>
<h4>1. 主键索引（PRIMARY KEY）</h4>
<p><strong>特点：</strong></p>
<ul>
<li>唯一且不能为NULL</li>
<li>一个表只能有一个主键</li>
<li>InnoDB中是聚簇索引</li>
</ul>
<p><strong>创建：</strong></p>
<pre><code class="language-sql">-- 创建表时指定
CREATE TABLE users (
    id INT PRIMARY KEY,
    name VARCHAR(50)
);
<p>-- 或
CREATE TABLE users (
id INT,
name VARCHAR(50),
PRIMARY KEY (id)
);</p>
<p>-- 添加主键
ALTER TABLE users ADD PRIMARY KEY (id);</p>
<p>-- 删除主键
ALTER TABLE users DROP PRIMARY KEY;
</code></pre></p>
<h4>2. 唯一索引（UNIQUE）</h4>
<p><strong>特点：</strong></p>
<ul>
<li>列值必须唯一，但可以有NULL</li>
<li>一个表可以有多个唯一索引</li>
</ul>
<p><strong>创建：</strong></p>
<pre><code class="language-sql">-- 创建表时指定
CREATE TABLE users (
    id INT PRIMARY KEY,
    email VARCHAR(100) UNIQUE
);
<p>-- 创建唯一索引
CREATE UNIQUE INDEX idx_email ON users(email);</p>
<p>-- 或
ALTER TABLE users ADD UNIQUE INDEX idx_email (email);</p>
<p>-- 删除唯一索引
DROP INDEX idx_email ON users;
</code></pre></p>
<h4>3. 普通索引（INDEX）</h4>
<p><strong>特点：</strong></p>
<ul>
<li>最基本的索引</li>
<li>没有任何限制</li>
</ul>
<p><strong>创建：</strong></p>
<pre><code class="language-sql">-- 创建索引
CREATE INDEX idx_name ON users(name);
<p>-- 或
ALTER TABLE users ADD INDEX idx_name (name);</p>
<p>-- 删除索引
DROP INDEX idx_name ON users;
</code></pre></p>
<h4>4. 全文索引（FULLTEXT）</h4>
<p><strong>特点：</strong></p>
<ul>
<li>用于全文搜索</li>
<li>只支持CHAR、VARCHAR、TEXT类型</li>
<li>InnoDB在MySQL 5.6+支持</li>
</ul>
<p><strong>创建：</strong></p>
<pre><code class="language-sql">-- 创建全文索引
CREATE FULLTEXT INDEX idx_content ON articles(content);
<p>-- 使用全文索引
SELECT * FROM articles
WHERE MATCH(content) AGAINST('MySQL' IN NATURAL LANGUAGE MODE);
</code></pre></p>
<h3>6.3.2 按字段数量分类</h3>
<h4>1. 单列索引</h4>
<pre><code class="language-sql">CREATE INDEX idx_name ON users(name);
</code></pre>
<h4>2. 联合索引（组合索引）⭐⭐⭐⭐⭐</h4>
<p><strong>创建：</strong></p>
<pre><code class="language-sql">-- 创建联合索引
CREATE INDEX idx_name_age_city ON users(name, age, city);
</code></pre>
<p><strong>最左前缀原则：</strong></p>
<pre><code class="language-sql">-- 索引：idx_name_age_city (name, age, city)
<p>-- ✅ 可以使用索引
WHERE name = '张三'
WHERE name = '张三' AND age = 20
WHERE name = '张三' AND age = 20 AND city = '北京'
WHERE name = '张三' AND city = '北京'  -- 只使用name</p>
<p>-- ❌ 不能使用索引
WHERE age = 20
WHERE city = '北京'
WHERE age = 20 AND city = '北京'
</code></pre></p>
<p><strong>索引顺序的选择：</strong></p>
<pre><code class="language-sql">-- 原则：区分度高的字段放前面
<p>-- 假设：
-- name: 10000个不同值（区分度高）
-- age: 100个不同值（区分度中）
-- city: 10个不同值（区分度低）</p>
<p>-- 推荐顺序：
CREATE INDEX idx_name_age_city ON users(name, age, city);</p>
<p>-- 不推荐：
CREATE INDEX idx_city_age_name ON users(city, age, name);
</code></pre></p></div>
