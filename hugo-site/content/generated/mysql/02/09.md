          <div class="vditor-reset post__content"><h1>第09章：锁机制 ⭐⭐⭐⭐⭐</h1>
<blockquote>
<p>锁是数据库并发控制的核心机制，理解锁机制是成为MySQL专家的必备技能</p>
</blockquote>
<h2>9.1 锁概述</h2>
<h3>9.1.1 为什么需要锁</h3>
<p><strong>问题：</strong> 多个事务同时修改同一数据，会导致数据不一致。</p>
<p><strong>示例：</strong></p>
<pre><code class="language-sql">-- 初始：stock = 10
<p>-- 事务A                          事务B
SELECT stock FROM products WHERE id=1;  -- 10
SELECT stock FROM products WHERE id=1;  -- 10
UPDATE products SET stock=9 WHERE id=1;
UPDATE products SET stock=9 WHERE id=1;
-- 结果：stock=9（错误！应该是8）
</code></pre></p>
<p><strong>解决方案：</strong> 使用锁机制，保证同一时间只有一个事务能修改数据。</p>
<h3>9.1.2 锁的分类</h3>
<p><strong>按锁的粒度分类：</strong></p>
<ul>
<li><strong>表锁（Table Lock）</strong>：锁定整张表</li>
<li><strong>行锁（Row Lock）</strong>：锁定某一行</li>
<li><strong>页锁（Page Lock）</strong>：锁定一页数据（BDB引擎）</li>
</ul>
<p><strong>按锁的类型分类：</strong></p>
<ul>
<li><strong>共享锁（S锁，Shared Lock）</strong>：读锁，多个事务可以同时持有</li>
<li><strong>排他锁（X锁，Exclusive Lock）</strong>：写锁，只有一个事务可以持有</li>
</ul>
<p><strong>按锁的模式分类：</strong></p>
<ul>
<li><strong>乐观锁</strong>：不真正加锁，通过版本号控制</li>
<li><strong>悲观锁</strong>：真正加锁，阻塞其他事务</li>
</ul>
<p><strong>InnoDB特有的锁：</strong></p>
<ul>
<li><strong>意向锁（Intention Lock）</strong>：表级锁，辅助行锁</li>
<li><strong>记录锁（Record Lock）</strong>：锁定单个行记录</li>
<li><strong>间隙锁（Gap Lock）</strong>：锁定索引记录之间的间隙</li>
<li><strong>临键锁（Next-Key Lock）</strong>：记录锁 + 间隙锁</li>
</ul>
<hr>
<h2>9.2 表锁</h2>
<h3>9.2.1 表锁概述</h3>
<p><strong>特点：</strong></p>
<ul>
<li>锁定整张表</li>
<li>开销小，加锁快</li>
<li>不会出现死锁</li>
<li>锁粒度大，并发度低</li>
<li>MyISAM默认使用表锁</li>
</ul>
<h3>9.2.2 表锁的使用</h3>
<p><strong>加锁：</strong></p>
<pre><code class="language-sql">-- 加读锁（共享锁）
LOCK TABLES users READ;
<p>-- 加写锁（排他锁）
LOCK TABLES users WRITE;</p>
<p>-- 同时锁定多个表
LOCK TABLES users READ, orders WRITE;
</code></pre></p>
<p><strong>释放锁：</strong></p>
<pre><code class="language-sql">-- 释放所有表锁
UNLOCK TABLES;
</code></pre>
<p><strong>演示：</strong></p>
<pre><code class="language-sql">-- 终端1
LOCK TABLES users READ;
SELECT * FROM users;  -- 可以读
UPDATE users SET age = 30 WHERE id = 1;  -- 报错：表被锁定
UNLOCK TABLES;
<p>-- 终端1
LOCK TABLES users WRITE;
SELECT * FROM users;  -- 可以读
UPDATE users SET age = 30 WHERE id = 1;  -- 可以写</p>
<p>-- 终端2
SELECT * FROM users;  -- 阻塞，等待终端1释放锁
</code></pre></p>
<h3>9.2.3 表锁兼容性</h3>
<table>
<thead>
<tr>
<th>当前锁</th>
<th>请求读锁</th>
<th>请求写锁</th>
</tr>
</thead>
<tbody>
<tr>
<td><strong>无锁</strong></td>
<td>✅ 允许</td>
<td>✅ 允许</td>
</tr>
<tr>
<td><strong>读锁</strong></td>
<td>✅ 允许</td>
<td>❌ 阻塞</td>
</tr>
<tr>
<td><strong>写锁</strong></td>
<td>❌ 阻塞</td>
<td>❌ 阻塞</td>
</tr>
</tbody>
</table>
<h3>9.2.4 查看表锁</h3>
<pre><code class="language-sql">-- 查看表锁状态
SHOW OPEN TABLES WHERE In_use &gt; 0;
<p>-- 查看表锁等待情况
SHOW STATUS LIKE 'Table_locks%';
-- Table_locks_immediate：立即获得锁的次数
-- Table_locks_waited：需要等待的次数
</code></pre></p>
<hr>
<h2>9.3 行锁</h2>
<h3>9.3.1 行锁概述</h3>
<p><strong>特点：</strong></p>
<ul>
<li>锁定单行或多行</li>
<li>开销大，加锁慢</li>
<li>可能出现死锁</li>
<li>锁粒度小，并发度高</li>
<li>InnoDB默认使用行锁</li>
</ul>
<p><strong>重要：</strong> 行锁是针对索引加的锁，如果没有索引，会升级为表锁！</p>
<h3>9.3.2 共享锁（S锁）</h3>
<p><strong>加锁方式：</strong></p>
<pre><code class="language-sql">-- 方法1：SELECT ... LOCK IN SHARE MODE（MySQL 5.7）
SELECT * FROM users WHERE id = 1 LOCK IN SHARE MODE;
<p>-- 方法2：SELECT ... FOR SHARE（MySQL 8.0）
SELECT * FROM users WHERE id = 1 FOR SHARE;
</code></pre></p>
<p><strong>特点：</strong></p>
<ul>
<li>允许其他事务读取（加S锁）</li>
<li>阻塞其他事务写入（加X锁）</li>
</ul>
<p><strong>演示：</strong></p>
<pre><code class="language-sql">-- 终端1（事务A）
START TRANSACTION;
SELECT * FROM users WHERE id = 1 LOCK IN SHARE MODE;  -- 加S锁
<p>-- 终端2（事务B）
START TRANSACTION;
SELECT * FROM users WHERE id = 1 LOCK IN SHARE MODE;  -- 成功（S锁兼容）
UPDATE users SET age = 30 WHERE id = 1;  -- 阻塞（等待S锁释放）</p>
<p>-- 终端1（事务A）
COMMIT;  -- 释放S锁</p>
<p>-- 终端2（事务B）
-- UPDATE执行成功
COMMIT;
</code></pre></p>
<h3>9.3.3 排他锁（X锁）</h3>
<p><strong>加锁方式：</strong></p>
<pre><code class="language-sql">-- 方法1：SELECT ... FOR UPDATE
SELECT * FROM users WHERE id = 1 FOR UPDATE;
<p>-- 方法2：UPDATE、DELETE、INSERT自动加X锁
UPDATE users SET age = 30 WHERE id = 1;
DELETE FROM users WHERE id = 1;
INSERT INTO users (name) VALUES ('test');
</code></pre></p>
<p><strong>特点：</strong></p>
<ul>
<li>阻塞其他事务读取（加S锁）</li>
<li>阻塞其他事务写入（加X锁）</li>
</ul>
<p><strong>演示：</strong></p>
<pre><code class="language-sql">-- 终端1（事务A）
START TRANSACTION;
SELECT * FROM users WHERE id = 1 FOR UPDATE;  -- 加X锁
<p>-- 终端2（事务B）
START TRANSACTION;
SELECT * FROM users WHERE id = 1;  -- 成功（快照读，不加锁）
SELECT * FROM users WHERE id = 1 LOCK IN SHARE MODE;  -- 阻塞（等待X锁释放）
SELECT * FROM users WHERE id = 1 FOR UPDATE;  -- 阻塞（等待X锁释放）
UPDATE users SET age = 30 WHERE id = 1;  -- 阻塞（等待X锁释放）</p>
<p>-- 终端1（事务A）
COMMIT;  -- 释放X锁</p>
<p>-- 终端2（事务B）
-- 所有阻塞的操作执行成功
COMMIT;
</code></pre></p>

<h2>9.4 意向锁</h2>
<h3>9.4.1 意向锁概述</h3>
<p><strong>问题：</strong> 如果要加表锁，需要检查是否有行锁，效率低。</p>
<p><strong>解决方案：</strong> 意向锁（表级锁），表示事务想要在表的某些行上加锁。</p>
<p><strong>类型：</strong></p>
<ul>
<li><strong>意向共享锁（IS锁）</strong>：表示事务想要在某些行上加S锁</li>
<li><strong>意向排他锁（IX锁）</strong>：表示事务想要在某些行上加X锁</li>
</ul>
<h3>9.4.2 意向锁的作用</h3>
<p><strong>自动加锁：</strong></p>
<pre><code class="language-sql">-- 加行级S锁时，自动加表级IS锁
SELECT * FROM users WHERE id = 1 LOCK IN SHARE MODE;
<p>-- 加行级X锁时，自动加表级IX锁
SELECT * FROM users WHERE id = 1 FOR UPDATE;
</code></pre></p>

<ul>

<h2>9.5 记录锁、间隙锁、临键锁 ⭐⭐⭐⭐⭐</h2>
<h3>9.5.1 记录锁（Record Lock）</h3>
<p><strong>定义：</strong> 锁定单个索引记录。</p>
<p><strong>示例：</strong></p>
<pre><code class="language-sql">-- 锁定id=1的记录
SELECT * FROM users WHERE id = 1 FOR UPDATE;
</code></pre>
<p><strong>特点：</strong></p>
<ul>
<li>只锁定匹配的索引记录</li>
<li>不锁定间隙</li>
</ul>
<h3>9.5.2 间隙锁（Gap Lock）</h3>
<p><strong>定义：</strong> 锁定索引记录之间的间隙，防止其他事务插入数据。</p>
<p><strong>作用：</strong> 解决幻读问题。</p>
<p><strong>示例：</strong></p>
<pre><code class="language-sql">-- 假设表中有id: 1, 5, 10
<p>-- 锁定(5, 10)之间的间隙
SELECT * FROM users WHERE id &gt; 5 AND id &lt; 10 FOR UPDATE;</p>
<p>-- 此时其他事务无法插入id=6, 7, 8, 9的记录
INSERT INTO users (id, name) VALUES (7, 'test');  -- 阻塞
</code></pre></p>
<p><strong>间隙锁的范围：</strong></p>
<pre><code class="language-sql">-- 假设表中有id: 1, 5, 10, 15
<p>-- 查询id=7（不存在）
SELECT * FROM users WHERE id = 7 FOR UPDATE;
-- 锁定间隙：(5, 10)</p>
<p>-- 查询id&gt;5
SELECT * FROM users WHERE id &gt; 5 FOR UPDATE;
-- 锁定间隙：(5, 10), (10, 15), (15, +∞)</p>
<p>-- 查询id&lt;10
SELECT * FROM users WHERE id &lt; 10 FOR UPDATE;
-- 锁定间隙：(-∞, 1), (1, 5), (5, 10)
</code></pre></p>
<h3>9.5.3 临键锁（Next-Key Lock）</h3>
<p><strong>定义：</strong> 记录锁 + 间隙锁，锁定索引记录及其前面的间隙。</p>
<p><strong>格式：</strong> (左开右闭区间]</p>
<p><strong>示例：</strong></p>
<pre><code class="language-sql">-- 假设表中有id: 1, 5, 10, 15
<p>-- 查询id=5
SELECT * FROM users WHERE id = 5 FOR UPDATE;
-- 临键锁：(1, 5]
-- 包含：间隙锁(1, 5) + 记录锁[5]</p>
<p>-- 查询id&gt;=5 AND id&lt;15
SELECT * FROM users WHERE id &gt;= 5 AND id &lt; 15 FOR UPDATE;
-- 临键锁：(1, 5], (5, 10], (10, 15)
</code></pre></p>
<p><strong>InnoDB默认使用Next-Key Lock（REPEATABLE READ）</strong></p>
<h3>9.5.4 锁定范围详解</h3>
<p><strong>准备测试数据：</strong></p>
<pre><code class="language-sql">CREATE TABLE users (
    id INT PRIMARY KEY,
    name VARCHAR(50),
    age INT,
    KEY idx_age (age)
) ENGINE=InnoDB;
<p>INSERT INTO users VALUES
(1, 'user1', 10),
(5, 'user5', 20),
(10, 'user10', 30),
(15, 'user15', 40);
</code></pre></p>
<p><strong>场景1：等值查询（记录存在）</strong></p>
<pre><code class="language-sql">-- 查询id=5
SELECT * FROM users WHERE id = 5 FOR UPDATE;
-- 锁定：记录锁[5]（主键等值查询，退化为记录锁）
</code></pre>
<p><strong>场景2：等值查询（记录不存在）</strong></p>
<pre><code class="language-sql">-- 查询id=7（不存在）
SELECT * FROM users WHERE id = 7 FOR UPDATE;
-- 锁定：间隙锁(5, 10)
</code></pre>
<p><strong>场景3：范围查询</strong></p>
<pre><code class="language-sql">-- 查询id&gt;5 AND id&lt;=10
SELECT * FROM users WHERE id &gt; 5 AND id &lt;= 10 FOR UPDATE;
-- 锁定：
--   间隙锁(5, 10)
--   记录锁[10]
--   间隙锁(10, 15)
-- 合并：临键锁(5, 15)
</code></pre>
<p><strong>场景4：非唯一索引</strong></p>
<pre><code class="language-sql">-- 查询age=20
SELECT * FROM users WHERE age = 20 FOR UPDATE;
-- 锁定：
--   临键锁(10, 20]（age索引）
--   间隙锁(20, 30)（age索引）
--   记录锁[5]（主键索引）
</code></pre>
<p><strong>场景5：无索引（全表扫描）</strong></p>
<pre><code class="language-sql">-- name没有索引
SELECT * FROM users WHERE name = 'user5' FOR UPDATE;
-- 锁定：所有记录 + 所有间隙（相当于表锁）
</code></pre>

<h2>9.6 死锁 ⭐⭐⭐⭐⭐</h2>
<h3>9.6.1 什么是死锁</h3>
<p><strong>死锁（Deadlock）</strong>：两个或多个事务互相等待对方释放锁，导致永久阻塞。</p>
<p><strong>经典示例：</strong></p>
<pre><code class="language-sql">-- 事务A                          事务B
START TRANSACTION;               START TRANSACTION;
UPDATE users SET age=30 WHERE id=1;  -- 锁定id=1
                                 UPDATE users SET age=40 WHERE id=2;  -- 锁定id=2
UPDATE users SET age=30 WHERE id=2;  -- 等待id=2的锁
                                 UPDATE users SET age=40 WHERE id=1;  -- 等待id=1的锁
-- 死锁！
</code></pre>
<p><strong>死锁的四个必要条件：</strong></p>
<ol>
<li><strong>互斥</strong>：资源不能共享</li>
<li><strong>持有并等待</strong>：持有资源的同时等待其他资源</li>
<li><strong>不可剥夺</strong>：资源不能被强制释放</li>
<li><strong>循环等待</strong>：形成环形等待链</li>
</ol>
<h3>9.6.2 死锁检测和处理</h3>
<p><strong>InnoDB的死锁处理：</strong></p>
<ol>
<li>自动检测死锁</li>
<li>选择一个事务回滚（回滚代价小的）</li>
<li>返回错误：<code>ERROR 1213: Deadlock found when trying to get lock</code></li>
</ol>
<p><strong>查看死锁日志：</strong></p>
<pre><code class="language-sql">-- 查看最近一次死锁信息
SHOW ENGINE INNODB STATUS\G
<p>-- 在输出中查找 LATEST DETECTED DEADLOCK 部分
</code></pre></p>
<p><strong>死锁日志示例：</strong></p>
<pre><code>------------------------

<h3>9.6.3 死锁案例分析</h3>
<p><strong>案例1：更新顺序不一致</strong></p>
<pre><code class="language-sql">-- 事务A
START TRANSACTION;
UPDATE users SET age = 30 WHERE id = 1;
UPDATE users SET age = 30 WHERE id = 2;
COMMIT;
<p>-- 事务B
START TRANSACTION;
UPDATE users SET age = 40 WHERE id = 2;  -- 与事务A顺序相反
UPDATE users SET age = 40 WHERE id = 1;
COMMIT;</p>
<p>-- 解决方案：统一更新顺序
-- 都按照id从小到大的顺序更新
</code></pre></p>
<p><strong>案例2：间隙锁导致的死锁</strong></p>
<pre><code class="language-sql">-- 假设表中有id: 1, 5, 10
<p>-- 事务A
START TRANSACTION;
SELECT * FROM users WHERE id = 3 FOR UPDATE;  -- 锁定间隙(1, 5)</p>
<p>-- 事务B
START TRANSACTION;
SELECT * FROM users WHERE id = 4 FOR UPDATE;  -- 锁定间隙(1, 5)，等待</p>
<p>-- 事务A
INSERT INTO users (id, name) VALUES (4, 'test');  -- 等待事务B</p>
<p>-- 死锁！
</code></pre></p>
<p><strong>案例3：索引失效导致的死锁</strong></p>
<pre><code class="language-sql">-- 事务A
START TRANSACTION;
UPDATE users SET age = 30 WHERE name = 'user1';  -- name没有索引，锁全表
<p>-- 事务B
START TRANSACTION;
UPDATE users SET age = 40 WHERE name = 'user2';  -- 等待全表锁</p>
<p>-- 解决方案：在name上创建索引
CREATE INDEX idx_name ON users(name);
</code></pre></p>
<h3>9.6.4 避免死锁的方法</h3>
<p><strong>1. 统一加锁顺序</strong></p>
<pre><code class="language-sql">-- ❌ 错误：顺序不一致
-- 事务A：先锁1，再锁2
-- 事务B：先锁2，再锁1
<p>-- ✅ 正确：统一顺序
-- 所有事务都按照id从小到大的顺序加锁
START TRANSACTION;
SELECT * FROM users WHERE id IN (1, 2, 5) ORDER BY id FOR UPDATE;
UPDATE ...
COMMIT;
</code></pre></p>
<p><strong>2. 尽量使用索引</strong></p>
<pre><code class="language-sql">-- ❌ 错误：没有索引，锁全表
UPDATE users SET age = 30 WHERE name = 'user1';
<p>-- ✅ 正确：有索引，只锁一行
CREATE INDEX idx_name ON users(name);
UPDATE users SET age = 30 WHERE name = 'user1';
</code></pre></p>
<p><strong>3. 缩短事务时间</strong></p>
<pre><code class="language-sql">-- ❌ 错误：事务时间过长
START TRANSACTION;
SELECT * FROM users WHERE id = 1 FOR UPDATE;
-- 执行复杂的业务逻辑（耗时）
UPDATE users SET age = 30 WHERE id = 1;
COMMIT;
<p>-- ✅ 正确：缩短事务时间
-- 先执行业务逻辑
START TRANSACTION;
SELECT * FROM users WHERE id = 1 FOR UPDATE;
UPDATE users SET age = 30 WHERE id = 1;
COMMIT;
</code></pre></p>
<p><strong>4. 降低隔离级别</strong></p>
<pre><code class="language-sql">-- REPEATABLE READ可能产生间隙锁死锁
SET SESSION transaction_isolation = 'READ-COMMITTED';
-- READ COMMITTED不使用间隙锁
</code></pre>
<p><strong>5. 使用乐观锁</strong></p>
<pre><code class="language-sql">-- 悲观锁：可能死锁
START TRANSACTION;
SELECT * FROM products WHERE id = 1 FOR UPDATE;
UPDATE products SET stock = stock - 1 WHERE id = 1;
COMMIT;
<p>-- 乐观锁：不会死锁
START TRANSACTION;
SELECT stock, version FROM products WHERE id = 1;
UPDATE products SET stock = stock - 1, version = version + 1
WHERE id = 1 AND version = #{old_version};
COMMIT;
</code></pre></p>
<p><strong>6. 设置锁等待超时</strong></p>
<pre><code class="language-sql">-- 查看锁等待超时时间（默认50秒）
SHOW VARIABLES LIKE 'innodb_lock_wait_timeout';
<p>-- 设置超时时间（秒）
SET innodb_lock_wait_timeout = 10;
</code></pre></p>

<h2>9.7 查看锁信息</h2>
<h3>9.7.1 查看当前锁</h3>
<p><strong>MySQL 5.7：</strong></p>
<pre><code class="language-sql">-- 查看InnoDB锁等待
SELECT * FROM information_schema.INNODB_LOCKS;
<p>-- 查看锁等待关系
SELECT * FROM information_schema.INNODB_LOCK_WAITS;</p>
<p>-- 查看事务信息
SELECT * FROM information_schema.INNODB_TRX;
</code></pre></p>
<p><strong>MySQL 8.0：</strong></p>
<pre><code class="language-sql">-- 查看数据锁
SELECT * FROM performance_schema.data_locks;
<p>-- 查看锁等待
SELECT * FROM performance_schema.data_lock_waits;</p>
<p>-- 查看事务信息
SELECT * FROM information_schema.INNODB_TRX;
</code></pre></p>
<h3>9.7.2 分析锁等待</h3>
<pre><code class="language-sql">-- 查看正在等待锁的事务
SELECT
    r.trx_id AS waiting_trx_id,
    r.trx_mysql_thread_id AS waiting_thread,
    r.trx_query AS waiting_query,
    b.trx_id AS blocking_trx_id,
    b.trx_mysql_thread_id AS blocking_thread,
    b.trx_query AS blocking_query
FROM information_schema.INNODB_LOCK_WAITS w
INNER JOIN information_schema.INNODB_TRX b ON b.trx_id = w.blocking_trx_id
INNER JOIN information_schema.INNODB_TRX r ON r.trx_id = w.requesting_trx_id;
</code></pre>
<h3>9.7.3 杀死阻塞的事务</h3>
<pre><code class="language-sql">-- 查看所有连接
SHOW PROCESSLIST;
<p>-- 杀死指定连接
KILL 123;  -- 123是thread_id
</code></pre></p>

<h2>9.8 实战案例</h2>
<h3>案例1：秒杀系统（悲观锁）</h3>
<pre><code class="language-sql">-- 创建商品表
CREATE TABLE products (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(100),
    stock INT NOT NULL DEFAULT 0
) ENGINE=InnoDB;
<p>-- 插入测试数据
INSERT INTO products (name, stock) VALUES ('iPhone 15', 100);</p>
<p>-- 秒杀逻辑（悲观锁）
START TRANSACTION;</p>
<p>-- 查询库存并加锁
SELECT stock FROM products WHERE id = 1 FOR UPDATE;</p>
<p>-- 检查库存
IF stock &gt; 0 THEN
-- 减库存
UPDATE products SET stock = stock - 1 WHERE id = 1;</p>
<pre><code>-- 创建订单
INSERT INTO orders (product_id, user_id, create_time)
VALUES (1, 100, NOW());
<p>COMMIT;
SELECT '秒杀成功' AS message;
</code></pre></p>
<p>ELSE
ROLLBACK;
SELECT '库存不足' AS message;
END IF;
</code></pre></p>
<h3>案例2：秒杀系统（乐观锁）</h3>
<pre><code class="language-sql">-- 创建商品表（带版本号）
CREATE TABLE products (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(100),
    stock INT NOT NULL DEFAULT 0,
    version INT NOT NULL DEFAULT 0
) ENGINE=InnoDB;
<p>-- 秒杀逻辑（乐观锁）
START TRANSACTION;</p>
<p>-- 查询库存和版本号
SELECT stock, version FROM products WHERE id = 1;
-- 假设：stock=10, version=5</p>
<p>-- 减库存（带版本号检查）
UPDATE products
SET stock = stock - 1, version = version + 1
WHERE id = 1 AND version = 5 AND stock &gt; 0;</p>
<p>-- 检查是否更新成功
IF ROW_COUNT() = 0 THEN
ROLLBACK;
SELECT '秒杀失败，请重试' AS message;
ELSE
-- 创建订单
INSERT INTO orders (product_id, user_id, create_time)
VALUES (1, 100, NOW());</p>
<pre><code>COMMIT;
SELECT '秒杀成功' AS message;
</code></pre>
<p>END IF;
</code></pre></p>
<h3>案例3：转账（防止死锁）</h3>
<pre><code class="language-sql">-- 创建账户表
CREATE TABLE accounts (
    id INT PRIMARY KEY AUTO_INCREMENT,
    user_id INT NOT NULL UNIQUE,
    balance DECIMAL(10, 2) NOT NULL DEFAULT 0
) ENGINE=InnoDB;
<p>-- 转账逻辑（统一加锁顺序）
START TRANSACTION;</p>
<p>-- 按照user_id从小到大的顺序加锁
SET @from_user = 1;
SET @to_user = 2;
SET @amount = 100;</p>
<p>-- 确保加锁顺序一致
IF @from_user &lt; @to_user THEN
SELECT balance FROM accounts WHERE user_id = @from_user FOR UPDATE;
SELECT balance FROM accounts WHERE user_id = @to_user FOR UPDATE;
ELSE
SELECT balance FROM accounts WHERE user_id = @to_user FOR UPDATE;
SELECT balance FROM accounts WHERE user_id = @from_user FOR UPDATE;
END IF;</p>
<p>-- 扣款
UPDATE accounts SET balance = balance - @amount
WHERE user_id = @from_user AND balance &gt;= @amount;</p>
<p>IF ROW_COUNT() = 0 THEN
ROLLBACK;
SELECT '余额不足' AS message;
ELSE
-- 加款
UPDATE accounts SET balance = balance + @amount
WHERE user_id = @to_user;</p>
<pre><code>COMMIT;
SELECT '转账成功' AS message;
</code></pre>
<p>END IF;
</code></pre></p>
<h3>案例4：防止重复插入</h3>
<pre><code class="language-sql">-- 方法1：使用唯一索引
CREATE TABLE users (
    id INT PRIMARY KEY AUTO_INCREMENT,
    username VARCHAR(50) NOT NULL UNIQUE
) ENGINE=InnoDB;
<p>INSERT INTO users (username) VALUES ('zhangsan');
-- 重复插入会报错</p>
<p>-- 方法2：使用INSERT IGNORE
INSERT IGNORE INTO users (username) VALUES ('zhangsan');
-- 不会报错，也不会插入</p>
<p>-- 方法3：使用ON DUPLICATE KEY UPDATE
INSERT INTO users (username) VALUES ('zhangsan')
ON DUPLICATE KEY UPDATE username = 'zhangsan';</p>
<p>-- 方法4：使用SELECT ... FOR UPDATE
START TRANSACTION;
SELECT id FROM users WHERE username = 'zhangsan' FOR UPDATE;
IF NOT FOUND THEN
INSERT INTO users (username) VALUES ('zhangsan');
END IF;
COMMIT;
</code></pre></p>

<h2>9.9 性能优化建议</h2>
<h3>9.9.1 减少锁冲突</h3>
<p><strong>1. 使用合适的索引</strong></p>
<pre><code class="language-sql">-- ❌ 错误：没有索引，锁全表
UPDATE users SET age = 30 WHERE name = 'user1';
<p>-- ✅ 正确：有索引，只锁一行
CREATE INDEX idx_name ON users(name);
UPDATE users SET age = 30 WHERE name = 'user1';
</code></pre></p>
<p><strong>2. 缩小锁的范围</strong></p>
<pre><code class="language-sql">-- ❌ 错误：锁定范围过大
SELECT * FROM users WHERE age &gt; 20 FOR UPDATE;
<p>-- ✅ 正确：精确锁定
SELECT * FROM users WHERE id = 1 FOR UPDATE;
</code></pre></p>
<p><strong>3. 缩短锁的时间</strong></p>
<pre><code class="language-sql">-- ❌ 错误：持锁时间过长
START TRANSACTION;
SELECT * FROM users WHERE id = 1 FOR UPDATE;
-- 执行复杂的业务逻辑（耗时）
UPDATE users SET age = 30 WHERE id = 1;
COMMIT;
<p>-- ✅ 正确：缩短持锁时间
-- 先执行业务逻辑
START TRANSACTION;
SELECT * FROM users WHERE id = 1 FOR UPDATE;
UPDATE users SET age = 30 WHERE id = 1;
COMMIT;
</code></pre></p>
<h3>9.9.2 选择合适的锁策略</h3>
<p><strong>悲观锁 vs 乐观锁：</strong></p>
<table>
<thead>
<tr>
<th>场景</th>
<th>推荐策略</th>
</tr>
</thead>
<tbody>
<tr>
<td>冲突频繁</td>
<td>悲观锁</td>
</tr>
<tr>
<td>冲突较少</td>
<td>乐观锁</td>
</tr>
<tr>
<td>读多写少</td>
<td>乐观锁</td>
</tr>
<tr>
<td>写多读少</td>
<td>悲观锁</td>
</tr>
</tbody>
</table>
<h3>9.9.3 监控锁等待</h3>
<pre><code class="language-sql">-- 查看锁等待统计
SHOW STATUS LIKE 'Innodb_row_lock%';
<p>-- Innodb_row_lock_current_waits：当前等待锁的数量
-- Innodb_row_lock_time：总等待时间（毫秒）
-- Innodb_row_lock_time_avg：平均等待时间（毫秒）
-- Innodb_row_lock_time_max：最大等待时间（毫秒）
-- Innodb_row_lock_waits：总等待次数
</code></pre></p>
