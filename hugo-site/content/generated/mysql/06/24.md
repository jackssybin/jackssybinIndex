<h2>24.1 高可用概述</h2>
<h3>24.1.1 什么是高可用？</h3>
<p><strong>定义：</strong></p>
<ul>
<li>High Availability（HA）</li>
<li>系统能够持续运行，减少停机时间</li>
<li>目标：99.9%、99.99%、99.999%可用性</li>
</ul>
<p><strong>可用性计算：</strong></p>
<pre><code>可用性 = (总时间 - 停机时间) / 总时间 × 100%
<p>99.9%   → 年停机时间：8.76小时
99.99%  → 年停机时间：52.56分钟
99.999% → 年停机时间：5.26分钟
</code></pre></p>
<h3>24.1.2 为什么需要高可用？</h3>
<p><strong>问题场景：</strong></p>
<ul>
<li>❌ 主库故障，业务中断</li>
<li>❌ 硬件故障，数据丢失</li>
<li>❌ 人为误操作，数据损坏</li>
</ul>
<p><strong>高可用的目标：</strong></p>
<ul>
<li>✅ 故障自动检测</li>
<li>✅ 故障自动切换</li>
<li>✅ 数据不丢失</li>
<li>✅ 业务不中断</li>
</ul>

<h2>24.2 MHA（Master High Availability）⭐⭐⭐⭐</h2>
<h3>24.2.1 MHA概述</h3>
<p><strong>什么是MHA？</strong></p>
<ul>
<li>Master High Availability Manager</li>
<li>日本DeNA公司开发</li>
<li>自动主从故障切换工具</li>
</ul>
<p><strong>MHA架构：</strong></p>
<pre><code>MHA Manager（管理节点）
    ↓ 监控
主库 → 从库1
    → 从库2
    → 从库3
<p>故障切换流程：</p>
<ol>
<li>检测主库故障</li>
<li>选择新主库</li>
<li>应用差异日志</li>
<li>切换到新主库</li>
<li>配置其他从库
</code></pre></li>
</ol>
<p><strong>MHA组件：</strong></p>
<ul>
<li><strong>MHA Manager</strong>：管理节点，监控和故障切换</li>
<li><strong>MHA Node</strong>：数据节点，在每个MySQL服务器上安装</li>
</ul>
<h3>24.2.2 MHA安装</h3>
<p><strong>环境准备：</strong></p>
<pre><code class="language-bash"># 服务器规划
192.168.1.10  master   主库
192.168.1.11  slave1   从库1（候选主库）
192.168.1.12  slave2   从库2
192.168.1.13  manager  MHA管理节点
<h1>配置SSH免密登录（所有节点）</h1>
<p>ssh-keygen -t rsa
ssh-copy-id <a href="mailto:root@192.168.1.10">root@192.168.1.10</a>
ssh-copy-id <a href="mailto:root@192.168.1.11">root@192.168.1.11</a>
ssh-copy-id <a href="mailto:root@192.168.1.12">root@192.168.1.12</a>
ssh-copy-id <a href="mailto:root@192.168.1.13">root@192.168.1.13</a>
</code></pre></p>
<p><strong>安装MHA Node（所有MySQL节点）：</strong></p>
<pre><code class="language-bash"># 安装依赖
yum install perl-DBD-MySQL perl-Config-Tiny perl-Log-Dispatch perl-Parallel-ForkManager
<h1>下载MHA Node</h1>
<p>wget <a href="https://github.com/yoshinorim/mha4mysql-node/releases/download/v0.58/mha4mysql-node-0.58-0.el7.centos.noarch.rpm">https://github.com/yoshinorim/mha4mysql-node/releases/download/v0.58/mha4mysql-node-0.58-0.el7.centos.noarch.rpm</a></p>
<h1>安装</h1>
<p>rpm -ivh mha4mysql-node-0.58-0.el7.centos.noarch.rpm</p>
<h1>验证</h1>
<p>ls /usr/bin/</p>
<h1>应该看到：</h1>

<h1>filter_mysqlbinlog</h1>
<h1>purge_relay_logs</h1>
<p></code></pre></p>
<p><strong>安装MHA Manager（管理节点）：</strong></p>
<pre><code class="language-bash"># 先安装MHA Node
rpm -ivh mha4mysql-node-0.58-0.el7.centos.noarch.rpm
<h1>下载MHA Manager</h1>

<h1>安装</h1>
<p>rpm -ivh mha4mysql-manager-0.58-0.el7.centos.noarch.rpm</p>

<p>ls /usr/bin/</p>

<h1>masterha_check_repl</h1>
<h1>masterha_manager</h1>
<h1>masterha_master_switch</h1>
<h1>masterha_check_status</h1>
<h1>masterha_stop</h1>
<p></code></pre></p>
<h3>24.2.3 MHA配置</h3>
<p><strong>创建MHA配置文件（/etc/mha/app1.cnf）：</strong></p>

<p>user=mha

<h1>复制用户</h1>

<p>ssh_user=root</p>

<p>secondary_check_script=/usr/local/bin/masterha_secondary_check -s 192.168.1.11 -s 192.168.1.12</p>

<p>[server1]
hostname=192.168.1.10
port=3306</p>

<p>[server3]
hostname=192.168.1.12
port=3306</p>
<h1>不作为候选主库</h1>
<p>no_master=1
</code></pre></p>
<p><strong>创建MySQL监控用户：</strong></p>
<pre><code class="language-sql">-- 在所有MySQL节点上执行

GRANT ALL PRIVILEGES ON *.* TO 'mha'@'%';
FLUSH PRIVILEGES;
</code></pre>

<h3>24.2.4 MHA使用</h3>
<p><strong>检查SSH连接：</strong></p>
<pre><code class="language-bash">masterha_check_ssh --conf=/etc/mha/app1.cnf
<h1>输出：</h1>
<h1>All SSH connection tests passed successfully.</h1>
<p></code></pre></p>
<p><strong>检查复制状态：</strong></p>
<pre><code class="language-bash">masterha_check_repl --conf=/etc/mha/app1.cnf

<h1>MySQL Replication Health is OK.</h1>
<p></code></pre></p>
<p><strong>启动MHA Manager：</strong></p>
<pre><code class="language-bash"># 前台启动（测试用）
masterha_manager --conf=/etc/mha/app1.cnf
<h1>后台启动</h1>
<p>nohup masterha_manager --conf=/etc/mha/app1.cnf &gt; /var/log/mha/app1/manager.log 2&gt;&amp;1 &amp;</p>

<p>masterha_check_status --conf=/etc/mha/app1.cnf</p>
<h1>输出：</h1>

<p></code></pre></p>
<p><strong>手动故障切换：</strong></p>
<pre><code class="language-bash"># 在线切换（主库正常）

  --new_master_host=192.168.1.11 --new_master_port=3306 \
  --orig_master_is_new_slave --running_updates_limit=10000
<h1>故障切换（主库故障）</h1>
<p>masterha_master_switch --conf=/etc/mha/app1.cnf --master_state=dead <br>
--dead_master_host=192.168.1.10 --dead_master_port=3306 <br>
--new_master_host=192.168.1.11 --new_master_port=3306
</code></pre></p>
<p><strong>停止MHA Manager：</strong></p>
<pre><code class="language-bash">masterha_stop --conf=/etc/mha/app1.cnf
</code></pre>

<h2>24.3 MGR（MySQL Group Replication）⭐⭐⭐⭐⭐</h2>
<h3>24.3.1 MGR概述</h3>
<p><strong>什么是MGR？</strong></p>
<ul>
<li>MySQL Group Replication</li>
<li>MySQL 5.7.17+官方提供</li>
<li>基于Paxos协议的多主复制</li>
<li>自动故障检测和切换</li>
</ul>
<p><strong>MGR特点：</strong></p>
<ul>
<li>✅ 自动故障检测</li>
<li>✅ 自动故障切换</li>
<li>✅ 数据强一致性</li>
<li>✅ 支持单主和多主模式</li>
<li>✅ 官方支持</li>
</ul>
<p><strong>MGR架构：</strong></p>
<pre><code>单主模式（推荐）：
节点1（主）← 写
节点2（从）← 读
节点3（从）← 读
<p>多主模式：
节点1（主）← 读写
节点2（主）← 读写
节点3（主）← 读写
</code></pre></p>
<h3>24.3.2 MGR安装配置</h3>
<p><strong>环境准备：</strong></p>
<pre><code class="language-bash"># 服务器规划
192.168.1.10  node1  主节点
192.168.1.11  node2  从节点
192.168.1.12  node3  从节点
<h1>MySQL版本要求：5.7.17+或8.0+</h1>
<p></code></pre></p>
<p><strong>配置节点1（/etc/my.cnf）：</strong></p>
<pre><code class="language-ini">[mysqld]
# 基本配置
server_id = 1
gtid_mode = ON
enforce_gtid_consistency = ON
binlog_format = ROW
log_bin = mysql-bin
log_slave_updates = ON
binlog_checksum = NONE
<h1>MGR配置</h1>
<p>plugin_load_add = 'group_replication.so'
transaction_write_set_extraction = XXHASH64</p>
<h1>组复制配置</h1>
<p>loose-group_replication_group_name = &quot;aaaaaaaa-aaaa-aaaa-aaaa-aaaaaaaaaaaa&quot;
loose-group_replication_start_on_boot = OFF
loose-group_replication_local_address = &quot;192.168.1.10:33061&quot;
loose-group_replication_group_seeds = &quot;192.168.1.10:33061,192.168.1.11:33061,192.168.1.12:33061&quot;
loose-group_replication_bootstrap_group = OFF</p>
<h1>单主模式</h1>
<p>loose-group_replication_single_primary_mode = ON
loose-group_replication_enforce_update_everywhere_checks = OFF</p>
<h1>多主模式（可选）</h1>
<h1>loose-group_replication_single_primary_mode = OFF</h1>
<h1>loose-group_replication_enforce_update_everywhere_checks = ON</h1>
<h1>IP白名单</h1>
<p>loose-group_replication_ip_whitelist = &quot;192.168.1.0/24&quot;
</code></pre></p>
<p><strong>配置节点2和节点3：</strong></p>
<pre><code class="language-ini"># 只需修改以下参数
server_id = 2  # 节点2
server_id = 3  # 节点3
<p>loose-group_replication_local_address = &quot;192.168.1.11:33061&quot;  # 节点2
loose-group_replication_local_address = &quot;192.168.1.12:33061&quot;  # 节点3
</code></pre></p>
<p><strong>创建复制用户（所有节点）：</strong></p>
<pre><code class="language-sql">-- 创建复制用户
SET SQL_LOG_BIN=0;
CREATE USER 'repl'@'%' IDENTIFIED BY 'repl_password';
GRANT REPLICATION SLAVE ON *.* TO 'repl'@'%';
FLUSH PRIVILEGES;
SET SQL_LOG_BIN=1;
<p>-- 配置复制通道
CHANGE MASTER TO MASTER_USER='repl', MASTER_PASSWORD='repl_password'
FOR CHANNEL 'group_replication_recovery';
</code></pre></p>
<p><strong>启动MGR（节点1）：</strong></p>
<pre><code class="language-sql">-- 安装插件
INSTALL PLUGIN group_replication SONAME 'group_replication.so';
<p>-- 启动组复制（第一个节点需要bootstrap）
SET GLOBAL group_replication_bootstrap_group=ON;
START GROUP_REPLICATION;
SET GLOBAL group_replication_bootstrap_group=OFF;</p>
<p>-- 查看状态
SELECT * FROM performance_schema.replication_group_members;
</code></pre></p>
<p><strong>启动MGR（节点2和节点3）：</strong></p>
<pre><code class="language-sql">-- 安装插件
INSTALL PLUGIN group_replication SONAME 'group_replication.so';
<p>-- 启动组复制
START GROUP_REPLICATION;</p>
<p>-- 查看状态
SELECT * FROM performance_schema.replication_group_members;
</code></pre></p>
<h3>24.3.3 MGR监控</h3>
<p><strong>查看组成员：</strong></p>
<pre><code class="language-sql">SELECT * FROM performance_schema.replication_group_members;
<p>-- 输出：
-- CHANNEL_NAME              MEMBER_ID                            MEMBER_HOST  MEMBER_PORT  MEMBER_STATE
-- group_replication_applier aaaaaaaa-aaaa-aaaa-aaaa-aaaaaaaaaaaa 192.168.1.10 3306         ONLINE
-- group_replication_applier bbbbbbbb-bbbb-bbbb-bbbb-bbbbbbbbbbbb 192.168.1.11 3306         ONLINE
-- group_replication_applier cccccccc-cccc-cccc-cccc-cccccccccccc 192.168.1.12 3306         ONLINE
</code></pre></p>
<p><strong>查看主节点：</strong></p>
<pre><code class="language-sql">-- 单主模式下查看主节点
SELECT MEMBER_HOST, MEMBER_PORT, MEMBER_STATE, MEMBER_ROLE
FROM performance_schema.replication_group_members;
<p>-- MEMBER_ROLE = PRIMARY：主节点
-- MEMBER_ROLE = SECONDARY：从节点
</code></pre></p>
<p><strong>查看复制状态：</strong></p>
<pre><code class="language-sql">SHOW STATUS LIKE 'group_replication%';
</code></pre>
<h3>24.3.4 MGR故障切换</h3>
<p><strong>自动故障切换：</strong></p>
<pre><code class="language-bash"># 模拟主节点故障
# 在节点1上执行
systemctl stop mysqld
<h1>在节点2或节点3上查看</h1>
<p>SELECT * FROM performance_schema.replication_group_members;</p>
<h1>结果：</h1>
<h1>节点1状态变为UNREACHABLE或消失</h1>
<h1>自动选举新主节点（节点2或节点3）</h1>
<p></code></pre></p>
<p><strong>手动切换主节点：</strong></p>
<pre><code class="language-sql">-- 在当前主节点上执行
SELECT group_replication_set_as_primary('新主节点的MEMBER_ID');
<p>-- 示例
SELECT group_replication_set_as_primary('bbbbbbbb-bbbb-bbbb-bbbb-bbbbbbbbbbbb');
</code></pre></p>

<h3>24.5.2 选择建议</h3>
<p><strong>小型项目（&lt;10台服务器）：</strong></p>
<ul>
<li>推荐：主从复制 + MHA</li>
<li>理由：简单、成本低</li>
</ul>
<p><strong>中型项目（10-50台服务器）：</strong></p>
<ul>
<li>推荐：MGR单主模式</li>
<li>理由：自动切换、数据一致性好</li>
</ul>
<p><strong>大型项目（&gt;50台服务器）：</strong></p>
<ul>
<li>推荐：MGR + ProxySQL + 分库分表</li>
<li>理由：高可用、高性能、可扩展</li>
</ul>
<p><strong>对数据一致性要求极高：</strong></p>
<ul>
<li>推荐：MGR</li>
<li>理由：基于Paxos协议，强一致性</li>
</ul>
<p><strong>对切换速度要求高：</strong></p>
<ul>
<li>推荐：MGR</li>
<li>理由：切换速度快（10-30秒）</li>
</ul>

   - 提高可用性
<ol start="2">

</ul>
</li>

</ul>
</li>
<li>
<p>定期演练</p>
<ul>
<li>定期进行故障切换演练</li>
<li>验证高可用方案</li>
</ul>
</li>
<li>
<p>数据备份</p>
<ul>

</ul>
</li>
<li>

</code></pre></li>
</ul>
</li>

<h3>24.6.2 常见问题</h3>
<p><strong>问题1：脑裂</strong></p>
<pre><code>场景：网络分区导致多个主库
<p>解决方案：</p>
<ol>
<li>使用奇数个节点（3、5、7）</li>
<li>使用仲裁节点</li>
<li>使用Paxos/Raft协议（MGR）
</code></pre></li>
</ol>
<p><strong>问题2：数据丢失</strong></p>
<pre><code>场景：主库故障，binlog未同步到从库
<p>解决方案：</p>
<ol>
<li>使用半同步复制</li>
<li>使用MGR</li>
<li>设置sync_binlog=1
</code></pre></li>
</ol>
<p><strong>问题3：切换时间长</strong></p>
<pre><code>场景：故障切换需要几分钟
<p>解决方案：</p>
<ol>
<li>使用MGR（切换快）</li>
<li>优化MHA配置</li>
<li>减少binlog大小
</code></pre></li>
</ol>
<p><strong>问题4：从库延迟</strong></p>
<pre><code>场景：从库延迟导致切换后数据不一致
<p>解决方案：</p>
<ol>
<li>监控主从延迟</li>
<li>使用并行复制</li>
<li>优化从库性能
</code></pre></li>
</ol>
