          <div class="vditor-reset post__content"><h1>第23章：读写分离</h1>
<blockquote>
<p>读写分离是提高MySQL并发能力的重要手段</p>
</blockquote>
<h2>23.1 读写分离概述</h2>
<h3>23.1.1 什么是读写分离？</h3>
<p><strong>定义：</strong></p>
<ul>
<li>写操作（INSERT、UPDATE、DELETE）→ 主库</li>
<li>读操作（SELECT）→ 从库</li>
<li>分散数据库压力，提高并发能力</li>
</ul>
<p><strong>架构图：</strong></p>
<pre><code>        应用程序
           ↓
      读写分离中间件
       ↙        ↘
    写操作      读操作
      ↓          ↓
    主库    →   从库1
            →   从库2
            →   从库3
</code></pre>
<h3>23.1.2 为什么需要读写分离？</h3>
<p><strong>问题场景：</strong></p>
<ul>
<li>❌ 读操作占比80%-90%</li>
<li>❌ 单台数据库压力大</li>
<li>❌ 查询慢，影响业务</li>
</ul>
<p><strong>读写分离的优势：</strong></p>
<ul>
<li>✅ 分散读压力（多个从库）</li>
<li>✅ 提高并发能力</li>
<li>✅ 提高查询性能</li>
<li>✅ 提高可用性（主库故障，从库可读）</li>
</ul>
<p><strong>适用场景：</strong></p>
<ul>
<li>✅ 读多写少（读写比 &gt; 3:1）</li>
<li>✅ 高并发查询</li>
<li>✅ 报表查询</li>
<li>✅ 数据分析</li>
</ul>
<hr>
<h2>23.2 读写分离实现方式</h2>
<h3>23.2.1 应用层实现</h3>
<p><strong>原理：</strong></p>
<ul>
<li>应用程序判断SQL类型</li>
<li>写操作连接主库</li>
<li>读操作连接从库</li>
</ul>
<p><strong>示例代码（Java）：</strong></p>
<pre><code class="language-java">public class DataSourceRouter {
    private DataSource masterDataSource;  // 主库
    private List&lt;DataSource&gt; slaveDataSources;  // 从库列表
<pre><code>public DataSource getDataSource(String sql) {
    if (isWriteOperation(sql)) {
        return masterDataSource;  // 写操作用主库
    } else {
        return getSlaveDataSource();  // 读操作用从库
    }
}
<p>private boolean isWriteOperation(String sql) {
String upperSql = sql.trim().toUpperCase();
return upperSql.startsWith(&amp;quot;INSERT&amp;quot;) ||
upperSql.startsWith(&amp;quot;UPDATE&amp;quot;) ||
upperSql.startsWith(&amp;quot;DELETE&amp;quot;);
}</p>
<p>private DataSource getSlaveDataSource() {
// 负载均衡：随机选择一个从库
int index = new Random().nextInt(slaveDataSources.size());
return slaveDataSources.get(index);
}
</code></pre></p>
<p>}
</code></pre></p>
<p><strong>优点：</strong></p>
<ul>
<li>✅ 灵活，可自定义规则</li>
<li>✅ 性能好（无中间件）</li>
<li>✅ 可以精确控制</li>
</ul>
<p><strong>缺点：</strong></p>
<ul>
<li>❌ 代码侵入性强</li>
<li>❌ 维护成本高</li>
<li>❌ 需要处理主从延迟</li>
</ul>
<hr>
<h3>23.2.2 中间件实现</h3>
<p><strong>常用中间件：</strong></p>
<ul>
<li><strong>MySQL Router</strong>：MySQL官方</li>
<li><strong>ProxySQL</strong>：功能强大（推荐）⭐⭐⭐⭐⭐</li>
<li><strong>MaxScale</strong>：MariaDB开发</li>
<li><strong>MyCat</strong>：国产，功能丰富</li>
<li><strong>Atlas</strong>：360开源</li>
</ul>
<p><strong>中间件架构：</strong></p>
<pre><code>应用程序
   ↓
中间件（ProxySQL）
   ↓
主库 + 从库
</code></pre>
<p><strong>优点：</strong></p>
<ul>
<li>✅ 对应用透明</li>
<li>✅ 无代码侵入</li>
<li>✅ 统一管理</li>
<li>✅ 支持负载均衡</li>
</ul>
<p><strong>缺点：</strong></p>
<ul>
<li>❌ 增加一层网络开销</li>
<li>❌ 中间件本身可能成为瓶颈</li>
<li>❌ 需要维护中间件</li>
</ul>

<h2>23.3 ProxySQL实战 ⭐⭐⭐⭐⭐</h2>

</code></pre></p>
<h3>23.3.2 ProxySQL配置</h3>
<pre><code class="language-sql">-- 连接ProxySQL管理端口
mysql -u admin -padmin -h 127.0.0.1 -P6032
<p>-- 1. 添加MySQL服务器
INSERT INTO mysql_servers(hostgroup_id, hostname, port)
VALUES (1, '192.168.1.10', 3306);  -- 主库，hostgroup_id=1</p>
<p>INSERT INTO mysql_servers(hostgroup_id, hostname, port)
VALUES (2, '192.168.1.11', 3306);  -- 从库1，hostgroup_id=2</p>
<p>INSERT INTO mysql_servers(hostgroup_id, hostname, port)
VALUES (2, '192.168.1.12', 3306);  -- 从库2，hostgroup_id=2</p>
<p>-- 加载到运行时
LOAD MYSQL SERVERS TO RUNTIME;</p>
<p>-- 保存到磁盘
SAVE MYSQL SERVERS TO DISK;</p>
<p>-- 查看服务器列表
SELECT * FROM mysql_servers;</p>
<p>-- 2. 配置监控账号
INSERT INTO mysql_users(username, password, default_hostgroup)
VALUES ('monitor', 'monitor_password', 1);</p>
<p>LOAD MYSQL USERS TO RUNTIME;
SAVE MYSQL USERS TO DISK;</p>
<p>-- 3. 配置应用账号
INSERT INTO mysql_users(username, password, default_hostgroup, transaction_persistent)
VALUES ('app_user', 'app_password', 1, 1);
-- default_hostgroup=1：默认使用主库
-- transaction_persistent=1：事务内的所有语句都路由到同一个hostgroup</p>
<p>LOAD MYSQL USERS TO RUNTIME;
SAVE MYSQL USERS TO DISK;</p>
<p>-- 4. 配置读写分离规则
-- 写操作路由到hostgroup 1（主库）
INSERT INTO mysql_query_rules(rule_id, active, match_pattern, destination_hostgroup, apply)
VALUES (1, 1, '^SELECT.*FOR UPDATE$', 1, 1);</p>
<p>INSERT INTO mysql_query_rules(rule_id, active, match_pattern, destination_hostgroup, apply)
VALUES (2, 1, '^SELECT.*LOCK IN SHARE MODE$', 1, 1);</p>
<p>-- 读操作路由到hostgroup 2（从库）
INSERT INTO mysql_query_rules(rule_id, active, match_pattern, destination_hostgroup, apply)
VALUES (10, 1, '^SELECT', 2, 1);</p>
<p>-- 写操作路由到hostgroup 1（主库）
INSERT INTO mysql_query_rules(rule_id, active, match_pattern, destination_hostgroup, apply)
VALUES (20, 1, '^INSERT', 1, 1);</p>
<p>INSERT INTO mysql_query_rules(rule_id, active, match_pattern, destination_hostgroup, apply)
VALUES (21, 1, '^UPDATE', 1, 1);</p>
<p>INSERT INTO mysql_query_rules(rule_id, active, match_pattern, destination_hostgroup, apply)
VALUES (22, 1, '^DELETE', 1, 1);</p>
<p>-- 加载规则
LOAD MYSQL QUERY RULES TO RUNTIME;
SAVE MYSQL QUERY RULES TO DISK;</p>
<p>-- 查看规则
SELECT rule_id, active, match_pattern, destination_hostgroup
FROM mysql_query_rules
ORDER BY rule_id;</p>
<p>-- 5. 配置健康检查
UPDATE global_variables
SET variable_value='monitor'
WHERE variable_name='mysql-monitor_username';</p>
<p>UPDATE global_variables
SET variable_value='monitor_password'
WHERE variable_name='mysql-monitor_password';</p>
<p>UPDATE global_variables
SET variable_value='2000'
WHERE variable_name='mysql-monitor_connect_interval';</p>
<p>UPDATE global_variables
SET variable_value='200000'
WHERE variable_name='mysql-monitor_ping_interval';</p>
<p>LOAD MYSQL VARIABLES TO RUNTIME;
SAVE MYSQL VARIABLES TO DISK;
</code></pre></p>

<h2>23.4 主从延迟问题 ⭐⭐⭐⭐⭐</h2>
<h3>23.4.1 什么是主从延迟？</h3>
<p><strong>定义：</strong></p>
<ul>
<li>主库执行的事务，从库还没有执行</li>
<li>从库的数据落后于主库</li>
</ul>
<p><strong>查看主从延迟：</strong></p>
<pre><code class="language-sql">-- 在从库上执行
SHOW SLAVE STATUS\G
<p>-- 关注字段：
-- Seconds_Behind_Master：延迟秒数
--   0：无延迟
--   NULL：复制未运行
--   &gt;0：有延迟
</code></pre></p>
<h3>23.4.2 主从延迟的原因</h3>
<p><strong>1. 主库写入压力大</strong></p>
<pre><code>主库TPS：10000
从库TPS：5000
→ 从库跟不上主库
</code></pre>
<p><strong>2. 从库硬件性能差</strong></p>
<pre><code>主库：SSD
从库：HDD
→ 从库I/O慢
</code></pre>
<p><strong>3. 大事务</strong></p>
<pre><code class="language-sql">-- 主库执行大事务
UPDATE users SET status = 1;  -- 更新100万行
-- 从库需要很长时间才能执行完
</code></pre>
<p><strong>4. 从库负载高</strong></p>
<pre><code>从库同时承担：
- 复制
- 查询
→ 资源竞争
</code></pre>
<p><strong>5. 网络延迟</strong></p>
<pre><code>主库和从库在不同机房
→ binlog传输慢
</code></pre>
<h3>23.4.3 主从延迟的影响</h3>
<p><strong>问题场景：</strong></p>
<pre><code class="language-sql">-- 1. 用户注册
INSERT INTO users(name, email) VALUES ('张三', 'test@example.com');
-- 主库执行成功
<p>-- 2. 立即查询（读从库）
SELECT * FROM users WHERE email = 'test@example.com';
-- 从库还没有同步，查询不到
-- 用户看到&quot;注册失败&quot;
</code></pre></p>
<p><strong>影响：</strong></p>
<ul>
<li>❌ 数据不一致</li>
<li>❌ 用户体验差</li>
<li>❌ 业务逻辑错误</li>
</ul>
<h3>23.4.4 解决主从延迟</h3>
<p><strong>方案1：强制读主库</strong></p>
<pre><code class="language-java">// 写操作后，立即读主库
public void register(User user) {
    // 1. 写入主库
    masterDao.insert(user);
<pre><code>// 2. 读主库（不读从库）
User savedUser = masterDao.selectByEmail(user.getEmail());
</code></pre>
<p>}
</code></pre></p>
<p><strong>方案2：延迟读取</strong></p>
<pre><code class="language-java">// 写操作后，延迟一段时间再读
public void register(User user) {
    // 1. 写入主库
    masterDao.insert(user);
<pre><code>// 2. 等待100ms
Thread.sleep(100);
<p>// 3. 读从库
User savedUser = slaveDao.selectByEmail(user.getEmail());
</code></pre></p>
<p>}
</code></pre></p>
<p><strong>方案3：缓存</strong></p>
<pre><code class="language-java">// 写操作后，写入缓存
public void register(User user) {
    // 1. 写入主库
    masterDao.insert(user);
<pre><code>// 2. 写入缓存
cache.put(&amp;quot;user:&amp;quot; + user.getEmail(), user);
<p>// 3. 读缓存
User savedUser = cache.get(&amp;quot;user:&amp;quot; + user.getEmail());
</code></pre></p>
<p>}
</code></pre></p>
<p><strong>方案4：半同步复制</strong></p>
<pre><code class="language-sql">-- 主库等待至少一个从库确认后才返回
-- 详见第22章：主从复制
</code></pre>
<p><strong>方案5：并行复制</strong></p>
<pre><code class="language-sql">-- 从库使用多线程并行复制
-- MySQL 5.7+支持
[mysqld]
slave_parallel_type = LOGICAL_CLOCK
slave_parallel_workers = 4
</code></pre>
<p><strong>方案6：读写都用主库</strong></p>
<pre><code class="language-java">// 对于重要业务，读写都用主库
public void transfer(int fromId, int toId, int amount) {
    // 1. 读主库
    Account from = masterDao.selectById(fromId);
    Account to = masterDao.selectById(toId);
<pre><code>// 2. 写主库
masterDao.update(from);
masterDao.update(to);
</code></pre>
<p>}
</code></pre></p>

<h2>23.5 负载均衡策略</h2>
<h3>23.5.1 轮询（Round Robin）</h3>
<pre><code class="language-java">public class RoundRobinLoadBalancer {
    private List&lt;DataSource&gt; slaves;
    private AtomicInteger index = new AtomicInteger(0);
<pre><code>public DataSource getDataSource() {
    int i = index.getAndIncrement() % slaves.size();
    return slaves.get(i);
}
</code></pre>
<p>}
</code></pre></p>
<p><strong>特点：</strong></p>
<ul>
<li>✅ 简单</li>
<li>✅ 均匀分配</li>
<li>❌ 不考虑服务器性能</li>
</ul>
<hr>
<h3>23.5.2 随机（Random）</h3>
<pre><code class="language-java">public class RandomLoadBalancer {
    private List&lt;DataSource&gt; slaves;
    private Random random = new Random();
<pre><code>public DataSource getDataSource() {
    int i = random.nextInt(slaves.size());
    return slaves.get(i);
}
</code></pre>
<p>}
</code></pre></p>
<p><strong>特点：</strong></p>
<ul>
<li>✅ 简单</li>
<li>✅ 分布均匀（长期）</li>
<li>❌ 短期可能不均匀</li>
</ul>
<hr>
<h3>23.5.3 加权轮询（Weighted Round Robin）</h3>
<pre><code class="language-java">public class WeightedRoundRobinLoadBalancer {
    private List&lt;Server&gt; servers;  // Server包含DataSource和weight
<pre><code>public DataSource getDataSource() {
    // 根据权重选择
    // 权重高的服务器被选中的概率大
}
</code></pre>
<p>}</p>
<p>// 配置示例
Server slave1 = new Server(dataSource1, 3);  // 权重3
Server slave2 = new Server(dataSource2, 1);  // 权重1
// slave1被选中的概率是slave2的3倍
</code></pre></p>
<p><strong>特点：</strong></p>
<ul>
<li>✅ 考虑服务器性能</li>
<li>✅ 灵活</li>
<li>❌ 配置复杂</li>
</ul>
<hr>
<h3>23.5.4 最少连接（Least Connections）</h3>
<pre><code class="language-java">public class LeastConnectionsLoadBalancer {
    private List&lt;DataSource&gt; slaves;
<pre><code>public DataSource getDataSource() {
    // 选择当前连接数最少的服务器
    DataSource selected = null;
    int minConnections = Integer.MAX_VALUE;
<pre><code>for (DataSource ds : slaves) {
    int connections = getActiveConnections(ds);
    if (connections &amp;amp;lt; minConnections) {
        minConnections = connections;
        selected = ds;
    }
}
<p>return selected;
</code></pre></p>
<p>}
</code></pre></p>
<p>}
</code></pre></p>
<p><strong>特点：</strong></p>
<ul>
<li>✅ 动态负载均衡</li>
<li>✅ 适合长连接</li>
<li>❌ 需要监控连接数</li>
</ul>
<hr>
<h2>23.6 读写分离最佳实践</h2>
<h3>23.6.1 设计原则</h3>
<pre><code>1. 读写分离对应用透明
   - 使用中间件
   - 或使用框架（如MyBatis）
<ol start="2">
<li>
<p>处理主从延迟</p>
<ul>
<li>重要业务读主库</li>
<li>或使用缓存</li>
</ul>
</li>
<li>
<p>监控主从延迟</p>
<ul>
<li>延迟过大时告警</li>
<li>延迟过大时切换到主库</li>
</ul>
</li>
<li>
<p>从库只读</p>
<ul>
<li>设置read_only=1</li>
<li>防止误写入</li>
</ul>
</li>
<li>
<p>负载均衡</p>
<ul>
<li>多个从库分散压力</li>
<li>根据性能设置权重</li>
</ul>
</li>
<li>
<p>故障切换</p>
<ul>
<li>从库故障时自动摘除</li>
<li>从库恢复后自动加入
</code></pre></li>
</ul>
</li>
</ol>
<h3>23.6.2 常见错误</h3>
<pre><code class="language-sql">-- ❌ 错误1：所有查询都读从库
-- 问题：主从延迟导致数据不一致
-- 解决：重要查询读主库
<p>-- ❌ 错误2：从库可写
-- 问题：从库数据与主库不一致
-- 解决：设置read_only=1</p>
<p>-- ❌ 错误3：不监控主从延迟
-- 问题：延迟过大时仍然读从库
-- 解决：监控Seconds_Behind_Master</p>
<p>-- ❌ 错误4：事务内读从库
-- 问题：事务内的读写可能不一致
-- 解决：事务内都读主库</p>
<p>-- ❌ 错误5：不处理从库故障
-- 问题：从库故障时查询失败
-- 解决：自动摘除故障从库
</code></pre></p>
<h3>23.6.3 监控指标</h3>
<pre><code class="language-sql">-- 1. 主从延迟
SHOW SLAVE STATUS\G
-- Seconds_Behind_Master
<p>-- 2. 从库负载
SHOW PROCESSLIST;
-- 查看正在执行的查询数量</p>
<p>-- 3. 查询分布
-- 统计主库和从库的查询数量
-- 理想比例：主库20%，从库80%</p>
<p>-- 4. 从库状态
SHOW SLAVE STATUS\G
-- Slave_IO_Running: Yes
-- Slave_SQL_Running: Yes</p>
<p>-- 5. 复制错误
SHOW SLAVE STATUS\G
-- Last_Error
</code></pre></p>
