          <div class="vditor-reset post__content"><h1>第22章：主从复制 ⭐⭐⭐⭐⭐</h1>
<blockquote>
<p>主从复制是MySQL高可用架构的基础，是成为MySQL专家的必备技能</p>
</blockquote>
<h2>22.1 主从复制概述</h2>
<h3>22.1.1 什么是主从复制</h3>
<p><strong>主从复制（Replication）</strong>：将主库（Master）的数据自动同步到从库（Slave）。</p>
<p><strong>架构图：</strong></p>
<pre><code>        ┌─────────┐
        │  Master │
        │  (主库)  │
        └────┬────┘
             │ binlog
        ┌────┴────┐
        │         │
   ┌────▼───┐ ┌──▼─────┐
   │ Slave1 │ │ Slave2 │
   │ (从库) │ │ (从库) │
   └────────┘ └────────┘
</code></pre>
<h3>22.1.2 主从复制的作用</h3>
<p><strong>1. 读写分离</strong></p>
<ul>
<li>主库：处理写操作（INSERT、UPDATE、DELETE）</li>
<li>从库：处理读操作（SELECT）</li>
<li>提高并发性能</li>
</ul>
<p><strong>2. 数据备份</strong></p>
<ul>
<li>从库作为实时备份</li>
<li>主库故障时可以快速切换</li>
</ul>
<p><strong>3. 高可用</strong></p>
<ul>
<li>主库故障时，从库可以升级为主库</li>
<li>实现故障转移</li>
</ul>
<p><strong>4. 数据分析</strong></p>
<ul>
<li>在从库上执行复杂查询</li>
<li>不影响主库性能</li>
</ul>
<h3>22.1.3 主从复制原理 ⭐⭐⭐⭐⭐</h3>
<p><strong>三个线程：</strong></p>
<ol>
<li>
<p><strong>主库：Binlog Dump线程</strong></p>
<ul>
<li>读取binlog</li>
<li>发送给从库</li>
</ul>
</li>
<li>
<p><strong>从库：I/O线程</strong></p>
<ul>
<li>接收主库的binlog</li>
<li>写入relay log（中继日志）</li>
</ul>
</li>
<li>
<p><strong>从库：SQL线程</strong></p>
<ul>
<li>读取relay log</li>
<li>执行SQL语句</li>
</ul>
</li>
</ol>
<p><strong>复制流程：</strong></p>
<pre><code>主库                                从库
┌──────────────┐              ┌──────────────┐
│  执行SQL语句  │              │              │
└──────┬───────┘              │              │
       │                      │              │
┌──────▼───────┐              │              │
│  写入binlog  │              │              │
└──────┬───────┘              │              │
       │                      │              │
┌──────▼───────┐    binlog    ┌──────▼───────┐
│ Binlog Dump  ├─────────────►│  I/O Thread  │
│   Thread     │              │              │
└──────────────┘              └──────┬───────┘
                                     │
                              ┌──────▼───────┐
                              │ 写入relay log│
                              └──────┬───────┘
                                     │
                              ┌──────▼───────┐
                              │  SQL Thread  │
                              │  执行SQL语句  │
                              └──────────────┘
</code></pre>
<p><strong>详细步骤：</strong></p>
<ol>
<li>主库执行SQL语句（INSERT、UPDATE、DELETE）</li>
<li>主库将变更写入binlog</li>
<li>从库的I/O线程连接主库，请求binlog</li>
<li>主库的Binlog Dump线程读取binlog，发送给从库</li>
<li>从库的I/O线程接收binlog，写入relay log</li>
<li>从库的SQL线程读取relay log，执行SQL语句</li>
<li>从库数据与主库保持一致</li>
</ol>
<hr>
<h2>22.2 配置主从复制</h2>
<h3>22.2.1 环境准备</h3>
<p><strong>服务器规划：</strong></p>
<ul>
<li>主库：192.168.1.100（Master）</li>
<li>从库1：192.168.1.101（Slave1）</li>
<li>从库2：192.168.1.102（Slave2）</li>
</ul>
<p><strong>版本要求：</strong></p>
<ul>
<li>MySQL 5.7.x（主从版本最好一致）</li>
<li>从库版本 &gt;= 主库版本</li>
</ul>
<h3>22.2.2 配置主库（Master）</h3>
<p><strong>1. 修改配置文件（my.cnf）</strong></p>
<pre><code class="language-ini">[mysqld]
# 服务器ID（唯一）
server-id=1
<h1>开启binlog</h1>
<p>log-bin=mysql-bin</p>
<h1>binlog格式（推荐ROW）</h1>
<p>binlog_format=ROW</p>
<h1>需要同步的数据库（可选）</h1>
<p>binlog-do-db=mydb</p>
<h1>不需要同步的数据库（可选）</h1>
<p>binlog-ignore-db=mysql
binlog-ignore-db=information_schema
binlog-ignore-db=performance_schema
binlog-ignore-db=sys</p>
<h1>binlog过期时间（天）</h1>
<p>expire_logs_days=7</p>
<h1>每次事务提交都写入binlog</h1>
<p>sync_binlog=1</p>
<h1>InnoDB设置</h1>
<p>innodb_flush_log_at_trx_commit=1
</code></pre></p>
<p><strong>2. 重启MySQL</strong></p>
<pre><code class="language-bash">systemctl restart mysqld
</code></pre>
<p><strong>3. 创建复制用户</strong></p>
<pre><code class="language-sql">-- 创建复制用户
CREATE USER 'repl'@'192.168.1.%' IDENTIFIED BY 'Repl@123456';
<p>-- 授予复制权限
GRANT REPLICATION SLAVE ON <em>.</em> TO 'repl'@'192.168.1.%';</p>
<p>-- 刷新权限
FLUSH PRIVILEGES;
</code></pre></p>
<p><strong>4. 查看主库状态</strong></p>
<pre><code class="language-sql">SHOW MASTER STATUS;
</code></pre>
<p>输出示例：</p>
<pre><code>+------------------+----------+--------------+------------------+
| File             | Position | Binlog_Do_DB | Binlog_Ignore_DB |
+------------------+----------+--------------+------------------+
| mysql-bin.000001 |      154 | mydb         | mysql,...        |
+------------------+----------+--------------+------------------+
</code></pre>
<p><strong>记录：</strong> File和Position，配置从库时需要用到。</p>
<p><strong>5. 锁定主库（可选，用于全量备份）</strong></p>
<pre><code class="language-sql">-- 锁定所有表（只读）
FLUSH TABLES WITH READ LOCK;
<p>-- 备份数据
-- mysqldump -uroot -p --all-databases &gt; /backup/all.sql</p>
<p>-- 解锁
UNLOCK TABLES;
</code></pre></p>
<h3>22.2.3 配置从库（Slave）</h3>
<p><strong>1. 修改配置文件（my.cnf）</strong></p>
<pre><code class="language-ini">[mysqld]
# 服务器ID（唯一，不能与主库相同）
server-id=2
<h1>开启binlog（可选，用于级联复制）</h1>
<p>log-bin=mysql-bin</p>
<h1>relay log</h1>
<p>relay-log=mysql-relay-bin</p>
<h1>只读模式（推荐）</h1>
<p>read_only=1</p>
<h1>超级用户也只读（推荐）</h1>
<p>super_read_only=1</p>
<h1>需要同步的数据库（可选）</h1>
<p>replicate-do-db=mydb</p>
<h1>不需要同步的数据库（可选）</h1>
<p>replicate-ignore-db=mysql
replicate-ignore-db=information_schema
replicate-ignore-db=performance_schema
replicate-ignore-db=sys</p>
<h1>跳过错误（慎用）</h1>
<h1>slave-skip-errors=1062,1032</h1>
<p></code></pre></p>
<p><strong>2. 重启MySQL</strong></p>
<pre><code class="language-bash">systemctl restart mysqld
</code></pre>
<p><strong>3. 配置主从关系</strong></p>
<pre><code class="language-sql">-- 停止从库（如果已经启动）
STOP SLAVE;
<p>-- 配置主库信息
CHANGE MASTER TO
MASTER_HOST='192.168.1.100',        -- 主库IP
MASTER_PORT=3306,                   -- 主库端口
MASTER_USER='repl',                 -- 复制用户
MASTER_PASSWORD='Repl@123456',      -- 复制密码
MASTER_LOG_FILE='mysql-bin.000001', -- binlog文件名
MASTER_LOG_POS=154;                 -- binlog位置</p>
<p>-- 启动从库
START SLAVE;
</code></pre></p>
<p><strong>4. 查看从库状态</strong></p>
<pre><code class="language-sql">SHOW SLAVE STATUS\G
</code></pre>
<p><strong>重要字段：</strong></p>
<pre><code>Slave_IO_Running: Yes          -- I/O线程运行状态
Slave_SQL_Running: Yes         -- SQL线程运行状态
Seconds_Behind_Master: 0       -- 主从延迟（秒）
Last_IO_Error:                 -- I/O错误信息
Last_SQL_Error:                -- SQL错误信息
Master_Log_File: mysql-bin.000001
Read_Master_Log_Pos: 154
Relay_Log_File: mysql-relay-bin.000002
Relay_Log_Pos: 320
</code></pre>
<p><strong>判断复制是否正常：</strong></p>
<ul>
<li><code>Slave_IO_Running: Yes</code></li>
<li><code>Slave_SQL_Running: Yes</code></li>
<li><code>Seconds_Behind_Master: 0</code> 或很小的值</li>
</ul>
<h3>22.2.4 测试主从复制</h3>
<p><strong>在主库执行：</strong></p>
<pre><code class="language-sql">-- 创建数据库
CREATE DATABASE test_repl;
<p>-- 使用数据库
USE test_repl;</p>
<p>-- 创建表
CREATE TABLE users (
id INT PRIMARY KEY AUTO_INCREMENT,
name VARCHAR(50),
create_time TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);</p>
<p>-- 插入数据
INSERT INTO users (name) VALUES ('user1'), ('user2'), ('user3');</p>
<p>-- 查询数据
SELECT * FROM users;
</code></pre></p>
<p><strong>在从库查询：</strong></p>
<pre><code class="language-sql">-- 使用数据库
USE test_repl;
<p>-- 查询数据（应该能看到主库插入的数据）
SELECT * FROM users;
</code></pre></p>
<p><strong>如果从库能查到数据，说明主从复制配置成功！</strong> ✅</p>
<hr>
<h2>22.3 复制格式</h2>
<h3>22.3.1 三种复制格式</h3>
<p><strong>1. STATEMENT（语句复制）</strong></p>
<pre><code class="language-ini">binlog_format=STATEMENT
</code></pre>
<p><strong>特点：</strong></p>
<ul>
<li>记录SQL语句</li>
<li>binlog文件小</li>
<li>可能导致主从数据不一致</li>
</ul>
<p><strong>示例：</strong></p>
<pre><code class="language-sql">-- 主库执行
UPDATE users SET create_time = NOW() WHERE id &lt; 100;
<p>-- binlog记录
UPDATE users SET create_time = NOW() WHERE id &lt; 100;</p>
<p>-- 问题：从库执行时NOW()的值可能不同
</code></pre></p>
<p><strong>2. ROW（行复制）⭐ 推荐</strong></p>
<pre><code class="language-ini">binlog_format=ROW
</code></pre>
<p><strong>特点：</strong></p>
<ul>
<li>记录每一行的变化</li>
<li>binlog文件大</li>
<li>数据一致性好</li>
</ul>
<p><strong>示例：</strong></p>
<pre><code class="language-sql">-- 主库执行
UPDATE users SET create_time = NOW() WHERE id &lt; 100;
<p>-- binlog记录（伪代码）
UPDATE users SET create_time = '2024-11-11 10:00:00' WHERE id = 1;
UPDATE users SET create_time = '2024-11-11 10:00:00' WHERE id = 2;
...
</code></pre></p>
<p><strong>3. MIXED（混合复制）</strong></p>
<pre><code class="language-ini">binlog_format=MIXED
</code></pre>
<p><strong>特点：</strong></p>
<ul>
<li>默认使用STATEMENT</li>
<li>遇到不确定的函数（NOW()、UUID()等）使用ROW</li>
<li>兼顾性能和一致性</li>
</ul>
<h3>22.3.2 选择建议</h3>
<table>
<thead>
<tr>
<th>场景</th>
<th>推荐格式</th>
</tr>
</thead>
<tbody>
<tr>
<td><strong>生产环境</strong></td>
<td>ROW ⭐</td>
</tr>
<tr>
<td><strong>数据一致性要求高</strong></td>
<td>ROW</td>
</tr>
<tr>
<td><strong>binlog文件大小敏感</strong></td>
<td>STATEMENT 或 MIXED</td>
</tr>
<tr>
<td><strong>需要审计SQL语句</strong></td>
<td>STATEMENT</td>
</tr>
</tbody>
</table>
<p><strong>修改复制格式：</strong></p>
<pre><code class="language-sql">-- 查看当前格式
SHOW VARIABLES LIKE 'binlog_format';
<p>-- 修改格式（全局）
SET GLOBAL binlog_format = 'ROW';</p>
<p>-- 修改格式（当前会话）
SET SESSION binlog_format = 'ROW';
</code></pre></p>
<hr>
<h2>22.4 GTID复制 ⭐⭐⭐⭐⭐</h2>
<h3>22.4.1 什么是GTID</h3>
<p><strong>GTID（Global Transaction Identifier）</strong>：全局事务标识符</p>
<p><strong>格式：</strong></p>
<pre><code>GTID = source_id:transaction_id
</code></pre>
<p><strong>示例：</strong></p>
<pre><code>3E11FA47-71CA-11E1-9E33-C80AA9429562:1
3E11FA47-71CA-11E1-9E33-C80AA9429562:2
3E11FA47-71CA-11E1-9E33-C80AA9429562:3
</code></pre>
<p><strong>优点：</strong></p>
<ul>
<li>每个事务有唯一标识</li>
<li>主从切换更简单</li>
<li>不需要指定binlog文件和位置</li>
<li>自动跳过已执行的事务</li>
</ul>
<h3>22.4.2 配置GTID复制</h3>
<p><strong>主库配置（my.cnf）：</strong></p>
<pre><code class="language-ini">[mysqld]
server-id=1
log-bin=mysql-bin
binlog_format=ROW
<h1>开启GTID</h1>
<p>gtid_mode=ON
enforce_gtid_consistency=ON</p>
<h1>binlog设置</h1>
<p>log_slave_updates=ON
</code></pre></p>
<p><strong>从库配置（my.cnf）：</strong></p>
<pre><code class="language-ini">[mysqld]
server-id=2
log-bin=mysql-bin
binlog_format=ROW
<h1>开启GTID</h1>
<p>gtid_mode=ON
enforce_gtid_consistency=ON</p>
<h1>relay log</h1>
<p>relay-log=mysql-relay-bin
log_slave_updates=ON
</code></pre></p>
<p><strong>配置主从关系：</strong></p>
<pre><code class="language-sql">-- 停止从库
STOP SLAVE;
<p>-- 配置主库（使用GTID，不需要指定binlog文件和位置）
CHANGE MASTER TO
MASTER_HOST='192.168.1.100',
MASTER_PORT=3306,
MASTER_USER='repl',
MASTER_PASSWORD='Repl@123456',
MASTER_AUTO_POSITION=1;  -- 使用GTID自动定位</p>
<p>-- 启动从库
START SLAVE;</p>
<p>-- 查看状态
SHOW SLAVE STATUS\G
</code></pre></p>
<p><strong>查看GTID信息：</strong></p>
<pre><code class="language-sql">-- 查看已执行的GTID
SHOW GLOBAL VARIABLES LIKE 'gtid_executed';
<p>-- 查看已清除的GTID
SHOW GLOBAL VARIABLES LIKE 'gtid_purged';
</code></pre></p>

<h2>22.5 主从延迟</h2>
<h3>22.5.1 什么是主从延迟</h3>
<p><strong>主从延迟</strong>：从库执行relay log的速度慢于主库写入binlog的速度。</p>
<p><strong>查看延迟：</strong></p>
<pre><code class="language-sql">SHOW SLAVE STATUS\G
<p>-- 关键字段
Seconds_Behind_Master: 5  -- 延迟5秒
</code></pre></p>
<h3>22.5.2 主从延迟的原因</h3>
<p><strong>1. 主库写入压力大</strong></p>
<ul>
<li>主库TPS（每秒事务数）过高</li>
<li>从库单线程回放跟不上</li>
</ul>
<p><strong>2. 从库性能差</strong></p>
<ul>
<li>从库硬件配置低</li>
<li>从库有大量查询</li>
</ul>
<p><strong>3. 网络延迟</strong></p>
<ul>
<li>主从之间网络带宽不足</li>
<li>网络抖动</li>
</ul>
<p><strong>4. 大事务</strong></p>
<ul>
<li>主库执行大事务（如批量更新100万行）</li>
<li>从库需要很长时间回放</li>
</ul>
<p><strong>5. 锁等待</strong></p>
<ul>
<li>从库有长时间运行的查询</li>
<li>阻塞SQL线程执行</li>
</ul>
<h3>22.5.3 解决主从延迟</h3>
<p><strong>1. 使用并行复制（MySQL 5.7+）</strong></p>
<pre><code class="language-ini">[mysqld]
# 开启并行复制
slave_parallel_type=LOGICAL_CLOCK
slave_parallel_workers=4  -- 4个并行线程
<h1>或使用DATABASE级别并行</h1>
<p>slave_parallel_type=DATABASE
slave_parallel_workers=4
</code></pre></p>
<p><strong>2. 升级从库硬件</strong></p>
<ul>
<li>使用SSD硬盘</li>
<li>增加内存</li>
<li>使用更快的CPU</li>
</ul>
<p><strong>3. 优化网络</strong></p>
<ul>
<li>使用专用网络</li>
<li>增加带宽</li>
<li>主从部署在同一机房</li>
</ul>
<p><strong>4. 避免大事务</strong></p>
<pre><code class="language-sql">-- ❌ 错误：大事务
START TRANSACTION;
UPDATE users SET status = 1 WHERE create_time &lt; '2024-01-01';  -- 100万行
COMMIT;
<p>-- ✅ 正确：分批执行
WHILE (SELECT COUNT(*) FROM users WHERE create_time &lt; '2024-01-01' AND status = 0) &gt; 0 DO
UPDATE users SET status = 1
WHERE create_time &lt; '2024-01-01' AND status = 0
LIMIT 1000;
SELECT SLEEP(0.1);
END WHILE;
</code></pre></p>
<p><strong>5. 从库只读</strong></p>
<pre><code class="language-ini">[mysqld]
read_only=1
super_read_only=1
</code></pre>
<p><strong>6. 监控和报警</strong></p>
<pre><code class="language-sql">-- 监控脚本
SELECT
    CASE
        WHEN Seconds_Behind_Master &gt; 60 THEN 'CRITICAL'
        WHEN Seconds_Behind_Master &gt; 30 THEN 'WARNING'
        ELSE 'OK'
    END AS status,
    Seconds_Behind_Master
FROM (SELECT * FROM information_schema.SLAVE_STATUS) AS s;
</code></pre>

<h2>22.6 主从故障处理</h2>
<h3>22.6.1 从库I/O线程停止</h3>
<p><strong>查看错误：</strong></p>
<pre><code class="language-sql">SHOW SLAVE STATUS\G
<p>-- 查看错误信息
Last_IO_Error: error connecting to master 'repl@192.168.1.100:3306'
</code></pre></p>
<p><strong>常见原因：</strong></p>
<ol>
<li>网络问题</li>
<li>主库宕机</li>
<li>复制用户密码错误</li>
<li>防火墙阻止</li>
</ol>
<p><strong>解决方案：</strong></p>
<pre><code class="language-sql">-- 检查网络
ping 192.168.1.100
<p>-- 检查主库状态
mysql -h192.168.1.100 -urepl -p</p>
<p>-- 重新配置主从
STOP SLAVE;
CHANGE MASTER TO ...
START SLAVE;
</code></pre></p>
<h3>22.6.2 从库SQL线程停止</h3>
<p><strong>查看错误：</strong></p>
<pre><code class="language-sql">SHOW SLAVE STATUS\G
<p>-- 查看错误信息
Last_SQL_Error: Error 'Duplicate entry '1' for key 'PRIMARY'' on query
</code></pre></p>
<p><strong>常见原因：</strong></p>
<ol>
<li>主键冲突</li>
<li>从库被误写入数据</li>
<li>binlog损坏</li>
</ol>
<p><strong>解决方案1：跳过错误（慎用）</strong></p>
<pre><code class="language-sql">-- 跳过1个事务
STOP SLAVE;
SET GLOBAL sql_slave_skip_counter = 1;
START SLAVE;
<p>-- 或在配置文件中跳过特定错误
[mysqld]
slave-skip-errors=1062,1032  -- 1062:主键冲突, 1032:记录不存在
</code></pre></p>
<p><strong>解决方案2：重新同步</strong></p>
<pre><code class="language-sql">-- 1. 在主库锁表并备份
FLUSH TABLES WITH READ LOCK;
mysqldump -uroot -p --all-databases --master-data=2 &gt; /backup/all.sql
UNLOCK TABLES;
<p>-- 2. 在从库导入
mysql -uroot -p &lt; /backup/all.sql</p>
<p>-- 3. 重新配置主从
STOP SLAVE;
CHANGE MASTER TO ...
START SLAVE;
</code></pre></p>
<p><strong>解决方案3：使用GTID跳过</strong></p>
<pre><code class="language-sql">-- 查看错误的GTID
SHOW SLAVE STATUS\G
-- Retrieved_Gtid_Set: 3E11FA47-71CA-11E1-9E33-C80AA9429562:1-5
-- Executed_Gtid_Set: 3E11FA47-71CA-11E1-9E33-C80AA9429562:1-3
<p>-- 跳过GTID
STOP SLAVE;
SET GTID_NEXT='3E11FA47-71CA-11E1-9E33-C80AA9429562:4';
BEGIN;
COMMIT;
SET GTID_NEXT='AUTOMATIC';
START SLAVE;
</code></pre></p>
<h3>22.6.3 主从数据不一致</h3>
<p><strong>检查数据一致性：</strong></p>
<pre><code class="language-bash"># 使用pt-table-checksum
pt-table-checksum --host=192.168.1.100 --user=root --password=xxx
<h1>修复数据不一致</h1>
<p>pt-table-sync --execute --sync-to-master h=192.168.1.101,u=root,p=xxx
</code></pre></p>

<h2>22.7 主从切换</h2>
<h3>22.7.1 计划内切换</h3>
<p><strong>场景：</strong> 主库需要维护，计划切换到从库</p>
<p><strong>步骤：</strong></p>
<p><strong>1. 停止主库写入</strong></p>
<pre><code class="language-sql">-- 在主库设置只读
SET GLOBAL read_only = 1;
SET GLOBAL super_read_only = 1;
</code></pre>
<p><strong>2. 等待从库同步完成</strong></p>
<pre><code class="language-sql">-- 在从库查看延迟
SHOW SLAVE STATUS\G
-- 等待 Seconds_Behind_Master = 0
</code></pre>
<p><strong>3. 停止从库复制</strong></p>
<pre><code class="language-sql">-- 在从库执行
STOP SLAVE;
RESET SLAVE ALL;
</code></pre>
<p><strong>4. 将从库提升为主库</strong></p>
<pre><code class="language-sql">-- 在从库执行
SET GLOBAL read_only = 0;
SET GLOBAL super_read_only = 0;
</code></pre>
<p><strong>5. 配置其他从库指向新主库</strong></p>
<pre><code class="language-sql">-- 在其他从库执行
STOP SLAVE;
CHANGE MASTER TO
    MASTER_HOST='192.168.1.101',  -- 新主库IP
    MASTER_PORT=3306,
    MASTER_USER='repl',
    MASTER_PASSWORD='Repl@123456',
    MASTER_AUTO_POSITION=1;  -- 使用GTID
START SLAVE;
</code></pre>
<p><strong>6. 应用程序切换到新主库</strong></p>
<h3>22.7.2 故障切换</h3>
<p><strong>场景：</strong> 主库宕机，紧急切换</p>
<p><strong>使用MHA（Master High Availability）自动切换：</strong></p>
<pre><code class="language-bash"># 安装MHA
yum install mha4mysql-manager mha4mysql-node
<h1>配置MHA</h1>
<p>vim /etc/mha/app1.cnf</p>
<h1>启动MHA监控</h1>
<p>nohup masterha_manager --conf=/etc/mha/app1.cnf &amp;</p>
<h1>主库宕机时，MHA自动切换</h1>
<p></code></pre></p>
<p><strong>手动切换步骤：</strong></p>
<ol>
<li>选择一个从库作为新主库（数据最新的）</li>
<li>停止该从库的复制</li>
<li>将其提升为主库</li>
<li>配置其他从库指向新主库</li>
<li>修复旧主库，作为新从库</li>
</ol>

</code></pre>

server-id=2
