<h2>28.1 故障排查概述</h2>
<h3>28.1.1 常见故障类型</h3>
<p><strong>性能故障：</strong></p>
<ul>
<li>查询慢</li>
<li>CPU使用率高</li>
<li>磁盘I/O高</li>
<li>内存不足</li>
</ul>
<p><strong>可用性故障：</strong></p>
<ul>
<li>MySQL服务停止</li>
<li>无法连接</li>
<li>主从复制中断</li>
<li>数据损坏</li>
</ul>
<p><strong>数据故障：</strong></p>
<ul>
<li>数据丢失</li>
<li>数据不一致</li>
<li>误删除数据</li>
</ul>
<hr>
<h2>28.2 查询慢故障排查 ⭐⭐⭐⭐⭐</h2>
<h3>28.2.1 问题现象</h3>
<pre><code>用户反馈：查询很慢，页面加载超过10秒
</code></pre>
<h3>28.2.2 排查步骤</h3>
<p><strong>步骤1：查看当前查询</strong></p>
<pre><code class="language-sql">SHOW FULL PROCESSLIST;
<p>-- 发现：
-- Id: 123, Time: 15, State: Sending data
-- Info: SELECT * FROM orders WHERE user_id = 12345
</code></pre></p>
<p><strong>步骤2：分析执行计划</strong></p>
<pre><code class="language-sql">EXPLAIN SELECT * FROM orders WHERE user_id = 12345;
<p>-- 结果：
-- type: ALL
-- rows: 1000000
-- Extra: Using where
-- 问题：全表扫描，没有索引
</code></pre></p>
<p><strong>步骤3：检查索引</strong></p>
<pre><code class="language-sql">SHOW INDEX FROM orders;
<p>-- 发现：user_id字段没有索引
</code></pre></p>
<p><strong>步骤4：添加索引</strong></p>
<pre><code class="language-sql">CREATE INDEX idx_user_id ON orders(user_id);
</code></pre>
<p><strong>步骤5：验证</strong></p>
<pre><code class="language-sql">EXPLAIN SELECT * FROM orders WHERE user_id = 12345;
<p>-- 结果：
-- type: ref
-- key: idx_user_id
-- rows: 100
-- 性能提升1000倍
</code></pre></p>
<hr>
<h2>28.3 CPU使用率高故障排查 ⭐⭐⭐⭐⭐</h2>
<h3>28.3.1 问题现象</h3>
<pre><code>监控告警：MySQL服务器CPU使用率100%
</code></pre>
<h3>28.3.2 排查步骤</h3>
<p><strong>步骤1：查看系统负载</strong></p>
<pre><code class="language-bash">top
<h1>发现：mysqld进程CPU使用率100%</h1>
<p></code></pre></p>
<p><strong>步骤2：查看当前查询</strong></p>
<pre><code class="language-sql">SHOW FULL PROCESSLIST;
<p>-- 发现大量查询：
-- SELECT * FROM users WHERE name LIKE '%张%'
</code></pre></p>
<p><strong>步骤3：分析问题</strong></p>
<pre><code class="language-sql">EXPLAIN SELECT * FROM users WHERE name LIKE '%张%';
<p>-- 结果：
-- type: ALL
-- rows: 10000000
-- 问题：LIKE '%张%'无法使用索引，全表扫描
</code></pre></p>
<p><strong>步骤4：优化方案</strong></p>
<pre><code class="language-sql">-- 方案1：使用全文索引
ALTER TABLE users ADD FULLTEXT INDEX ft_name(name);
SELECT * FROM users WHERE MATCH(name) AGAINST('张');
<p>-- 方案2：使用Elasticsearch
-- 将搜索功能迁移到ES</p>
<p>-- 方案3：限制查询
-- 要求用户输入至少2个字符，使用'张%'而不是'%张%'
</code></pre></p>
<hr>
<h2>28.4 连接数耗尽故障排查 ⭐⭐⭐⭐⭐</h2>
<h3>28.4.1 问题现象</h3>
<pre><code>应用报错：Too many connections
</code></pre>
<h3>28.4.2 排查步骤</h3>
<p><strong>步骤1：查看连接数</strong></p>
<pre><code class="language-sql">SHOW STATUS LIKE 'Threads_connected';
-- Threads_connected: 600
<p>SHOW VARIABLES LIKE 'max_connections';
-- max_connections: 600</p>
<p>-- 问题：连接数已满
</code></pre></p>
<p><strong>步骤2：查看连接来源</strong></p>
<pre><code class="language-sql">SELECT 
    USER,
    HOST,
    COUNT(*) AS conn_count
FROM information_schema.PROCESSLIST
GROUP BY USER, HOST
ORDER BY conn_count DESC;
<p>-- 结果：
-- app_user  192.168.1.100  500
-- 问题：某个应用服务器占用大量连接
</code></pre></p>
<p><strong>步骤3：查看连接状态</strong></p>
<pre><code class="language-sql">SELECT 
    COMMAND,
    COUNT(*) AS count
FROM information_schema.PROCESSLIST
GROUP BY COMMAND;
<p>-- 结果：
-- Sleep  480
-- Query   20
-- 问题：大量Sleep连接（连接未释放）
</code></pre></p>
<p><strong>步骤4：解决方案</strong></p>
<pre><code class="language-sql">-- 临时方案：增加最大连接数
SET GLOBAL max_connections = 1000;
<p>-- 长期方案：
-- 1. 检查应用代码，确保连接正确释放
-- 2. 使用连接池
-- 3. 设置wait_timeout
SET GLOBAL wait_timeout = 600;  -- 10分钟
SET GLOBAL interactive_timeout = 600;</p>
<p>-- 4. 杀死长时间Sleep的连接
SELECT CONCAT('KILL ', id, ';')
FROM information_schema.PROCESSLIST
WHERE COMMAND = 'Sleep' AND TIME &gt; 600;
</code></pre></p>
<hr>
<h2>28.5 主从复制中断故障排查 ⭐⭐⭐⭐⭐</h2>
<h3>28.5.1 问题现象</h3>
<pre><code>监控告警：主从复制中断
</code></pre>
<h3>28.5.2 排查步骤</h3>
<p><strong>步骤1：查看复制状态</strong></p>
<pre><code class="language-sql">SHOW SLAVE STATUS\G
<p>-- 关注字段：
-- Slave_IO_Running: No
-- Slave_SQL_Running: Yes
-- Last_IO_Error: Got fatal error 1236 from master...
</code></pre></p>
<p><strong>步骤2：分析错误</strong></p>
<pre><code>错误类型1：binlog文件不存在
Last_IO_Error: Got fatal error 1236 from master when reading data from binary log: 'Could not find first log file name in binary log index file'
<p>原因：主库binlog被删除
解决：重新搭建主从复制
</code></pre></p>
<pre><code>错误类型2：SQL线程错误
Slave_SQL_Running: No
Last_SQL_Error: Error 'Duplicate entry '1' for key 'PRIMARY'' on query
<p>原因：主从数据不一致
解决：跳过错误或重新同步
</code></pre></p>
<p><strong>步骤3：解决方案</strong></p>
<p><strong>方案1：跳过错误（谨慎使用）</strong></p>
<pre><code class="language-sql">-- 跳过1个错误
STOP SLAVE;
SET GLOBAL sql_slave_skip_counter = 1;
START SLAVE;
</code></pre>
<p><strong>方案2：重新搭建主从</strong></p>
<pre><code class="language-bash"># 1. 主库备份
mysqldump -u root -p --single-transaction --master-data=2 --all-databases &gt; backup.sql
<h1>2. 从库导入</h1>
<p>mysql -u root -p &lt; backup.sql</p>
<h1>3. 配置复制</h1>
<p>CHANGE MASTER TO
MASTER_HOST='192.168.1.10',
MASTER_USER='repl',
MASTER_PASSWORD='repl_password',
MASTER_LOG_FILE='mysql-bin.000010',
MASTER_LOG_POS=154;</p>
<p>START SLAVE;
</code></pre></p>

<h2>28.6 磁盘空间满故障排查 ⭐⭐⭐⭐⭐</h2>
<h3>28.6.1 问题现象</h3>
<pre><code>MySQL无法写入数据
错误：ERROR 3 (HY000): Error writing file '/var/lib/mysql/...' (Errcode: 28 - No space left on device)
</code></pre>
<h3>28.6.2 排查步骤</h3>
<p><strong>步骤1：查看磁盘空间</strong></p>
<pre><code class="language-bash">df -h
<h1>结果：</h1>
<h1>/dev/sda1  100G  100G  0  100%  /var/lib/mysql</h1>
<h1>问题：磁盘已满</h1>
<p></code></pre></p>
<p><strong>步骤2：查找大文件</strong></p>
<pre><code class="language-bash"># 查看MySQL数据目录大小
du -sh /var/lib/mysql/*
<h1>结果：</h1>
<h1>50G  /var/lib/mysql/mydb</h1>
<h1>30G  /var/lib/mysql/mysql-bin.*</h1>
<h1>20G  /var/lib/mysql/ibdata1</h1>
<p></code></pre></p>
<p><strong>步骤3：清理空间</strong></p>
<p><strong>清理binlog：</strong></p>
<pre><code class="language-sql">-- 查看binlog
SHOW BINARY LOGS;
<p>-- 删除7天前的binlog
PURGE BINARY LOGS BEFORE DATE_SUB(NOW(), INTERVAL 7 DAY);</p>
<p>-- 或删除指定binlog之前的所有binlog
PURGE BINARY LOGS TO 'mysql-bin.000100';</p>
<p>-- 配置自动清理
SET GLOBAL expire_logs_days = 7;
</code></pre></p>
<p><strong>清理慢查询日志：</strong></p>
<pre><code class="language-bash"># 备份后清空
cp /var/lib/mysql/slow.log /backup/slow.log.bak
&gt; /var/lib/mysql/slow.log
</code></pre>
<p><strong>清理临时文件：</strong></p>
<pre><code class="language-bash"># 清理/tmp目录
rm -f /tmp/mysql*
</code></pre>

<p><strong>步骤3：解决方案</strong></p>

-- 2. 按相同顺序访问资源
-- 3. 使用乐观锁代替悲观锁
-- 4. 降低事务隔离级别（谨慎）
</code></pre>

<h2>28.8 数据恢复实战</h2>
<h3>28.8.1 误删除数据</h3>
<p><strong>场景：</strong></p>
<pre><code class="language-sql">-- 误删除
DELETE FROM users WHERE age &gt; 18;
-- 删除了100万条数据
</code></pre>
<p><strong>恢复步骤：</strong></p>
<pre><code class="language-bash"># 1. 停止应用，防止继续写入
<h1>2. 使用binlog恢复</h1>
<h1>找到误删除之前的binlog位置</h1>
<p>mysqlbinlog --start-datetime=&quot;2024-11-10 09:00:00&quot; <br>
--stop-datetime=&quot;2024-11-10 09:59:59&quot; <br>
/var/lib/mysql/mysql-bin.000010 &gt; recover.sql</p>
<h1>3. 过滤掉DELETE语句</h1>
<p>grep -v &quot;DELETE FROM users&quot; recover.sql &gt; recover_filtered.sql</p>
<h1>4. 恢复数据</h1>
<p>mysql -u root -p mydb &lt; recover_filtered.sql</p>
<h1>5. 验证数据</h1>
<p>SELECT COUNT(*) FROM users;
</code></pre></p>
