## 问题出在哪？顺序反了

传统的编程学习路径是这样的：**先学理论，再学概念，最后做项目**。

但大部分人卡就在「最后」这一步。理论都懂，概念都背，真要把这些碎片拼成一个能跑的东西，瞬间就不知道从哪下手。

这个仓库的核心逻辑非常简单：**反过来学**。

你想理解操作系统？别先啃 1000 页的《深入理解计算机系统》，动手写一个简单内核出来。

你想理解编译器？别先背《编译原理》那套术语，从写一个简单的 Lisp 解释器开始。

你想理解 React？别光看官方文档，自己写一个 160 行的 React 出来。

> "What I cannot create, I do not understand." —— 费曼

这其实就是费曼学习法的工程化落地：**不能创造，就不算真的理解**。

## 三个小项目，试了就懂

我自己试过几个小项目，最大的感受是：之前看书看不明白的地方，动手写一遍瞬间就通了。

举三个门槛不高、收获很大的例子：

### 1. 写一个自己的 Git

你天天用 Git，但你真的理解 Git 是怎么工作的吗？`git add` 到底干了什么？对象存储是什么结构？

这个仓库里有一个 Python 教程，不到 1000 行就能实现一个可工作的 Git，支持 `init`、`add`、`commit`、`log` 这些基础命令。

跟着写完一遍，你再看 Git 的底层文档，那些「对象」「树」「提交」这些概念瞬间就活了。之前是死记硬背，现在是你自己写出来的，想忘都忘不了。

### 2. 写一个自己的 Redis

Redis 为什么这么快？数据结构到底怎么实现的？

这里有一个简化版 Python 教程，不到 500 行就能跑起来一个基本的 KV 存储服务器。

我见过很多人简历上写「熟练使用 Redis」，问他「跳表是什么结构」都说不清楚。要是你自己实现过一遍，这个问题闭着眼都能答。

### 3. 写一个自己的 React

现在前端几乎人人用 React，但你真的理解 `Virtual DOM` 是什么吗？

这里有一个经典教程叫 Gooact，真的只用 **160 行 JavaScript** 就实现了 React 的核心逻辑。

写完你再去看 React 源码，脉络一下子就清晰了。原来那些高大上的概念，拆解之后也没那么神秘。

## 适合谁？不适合谁？一句话说清楚

| 适合 | 不适合 |
|------|--------|
| 已经入门，想突破瓶颈 | 纯零基础，第一门编程语言 |
| 知识点零散，想建立体系 | 赶项目，需要快速出活 |
| 准备面试，想吃透原理 | 只想了解概念，不打算深究 |

总结一下：如果你已经会写基本代码，但感觉遇到了瓶颈，对底层原理一知半解，想往上走又不知道从哪开始——这个仓库就是为你准备的。

## 怎么开始？给你一个可复制的四步走

我自己总结了一个比较舒服的节奏，分享给你：

**第一步：选一个你"天天用但不太懂"的小东西**

别一开始就选「写一个操作系统」这种大项目，找一个小的，能在一个周末做完的。推荐几个门槛不高收获很大的起点：

- 前端：写一个自己的 React（160 行）/ 简单 Promise 实现
- 后端：写一个自己的 Git（~1000 行 Python）/ 简单 Web 服务器
- 底层：写一个简单的哈希表 / 内存分配器

关键是**小而完整**，能快速拿到正反馈。

**第二步：别先看教程，自己先想 15 分钟**

拿到教程之后，先别急着跟着抄。花 15 分钟想一想：这个问题如果让我来做，我会分成几步？核心数据结构怎么设计？哪些地方可能会踩坑？

想一遍再看作者的实现，收获会大很多。

**第三步：一行一行敲，卡壳先自己想 10 分钟**

不要复制粘贴，真的一行一行敲。遇到报错了，先自己 Google 解决，解决不了再看教程是怎么处理的。

这个过程虽然慢，但解决问题的能力就是这么练出来的。

**第四步：做完写三句话总结**

不需要太长，就回答三个问题：

1. 这个东西核心解决了什么问题？
2. 哪个地方我卡了很久，最后是怎么想通的？
3. 有哪个设计技巧让你觉得「居然可以这样」？

写下来，不管是发博客还是存在自己笔记里，三个月后再看，印象还很深。

## 最后：它改变的不是你的知识库，是你的思维方式

你不需要等，不需要攒够了"足够多的知识"再开始，现在就能找一个小题目动手做。做着做着，你就会发现：原来那些看上去高大上的技术，拆解之后也没那么神秘。

> 「不能创造，就不算理解。」

这句话放在今天这个 AI 能帮你写代码的时代，反而更有意义。AI 能帮你写代码，但理解为什么这么写，只能靠你自己。
