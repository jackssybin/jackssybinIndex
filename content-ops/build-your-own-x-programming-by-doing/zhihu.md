> **TL;DR**：`build-your-own-x` 适合**已经入门但遇到瓶颈的开发者**通过「动手从零造轮子」来吃透底层原理，不适合纯零基础入门。它的核心价值不是收集了 500 份教程，而是把费曼学习法从一句口号变成了可执行的编程学习路径。

## 问题：为什么看完很多书还是不会写项目？

很多人学习编程都陷入过这样的怪圈：

- 视频刷完了，知识点都听懂了，自己动手做项目还是卡壳
- 厚书翻完了，合上书问你「这个组件到底怎么工作的」说不清楚
- 简历写「熟悉 XX 框架」，面试官一问底层原理直接哑口无言

这其实不是你能力不行，是学习顺序反了。传统路径是「先理论，后实践」，但大部分人卡在「最后一步」永远迈不出去。

GitHub 上这个 53 万星的仓库 `codecrafters-io/build-your-own-x` 给出了另一条路：**反过来学，在创造中理解**。

![项目领域覆盖](https://jackssybin.cn/images/build-your-own-x-programming-by-doing/01-category-grid.png)

## 这个仓库到底有什么？

它就是一个高质量教程合集，收集了 **500+ 份** step-by-step 的指南，教你从零开始构建各种各样的技术。覆盖范围非常广：

| 大类 | 代表内容 |
|------|----------|
| 图形/游戏 | 3D 渲染器（光线追踪/光栅化）、游戏引擎、物理引擎 |
| AI/神经网络 | LLM 从零实现、扩散模型、RAG、PyTorch 内核重写 |
| 系统底层 | 操作系统内核、内存分配器、网络栈、RISC-V 处理器 |
| 语言/编译 | 编程语言、编译器、解释器、正则表达式引擎 |
| 数据存储 | 数据库、Redis、B+树、搜索引擎、哈希表 |
| 区块链 | 区块链、加密货币、PoW/PoS 共识 |
| 工具应用 | Git、Docker、文本编辑器、Shell、Web 服务器 |
| 前端框架 | React、Redux、Virtual DOM、模块打包器 |

从最底层的硬件到最上层的前端框架，只要是你感兴趣的方向，基本都能找到对应的教程。而且这些教程都是社区筛选出来的，质量比网上搜索到的零散水文好很多。

## 为什么「动手造轮子」是更有效的学习方式？

我自己跟着做过几个小项目，最大的感受是：**看书一百遍，不如动手写一遍**。

举三个最受欢迎也最容易上手的例子：

### 例子一：自己写一个 Git

你天天用 Git，但 `git add` 到底干了什么？对象存储是什么结构？很多人用了五六年 Git，这些问题还是说不清楚。

这个仓库里有一个 Python 教程叫「Write yourself a Git!」，不到 1000 行就能实现一个支持 `init`/`add`/`commit`/`log` 的可工作 Git。

跟着敲完一遍，你再看 Git 的底层文档，「对象」「树」「提交」这些概念瞬间就活了。之前是死记硬背，现在是你自己写出来的，想忘都忘不了。

### 例子二：自己写一个 Redis

Redis 为什么这么快？跳跃表到底是怎么工作的？很多人简历写「熟练使用 Redis」，这个问题都答不清楚。

这里有一个简化版 Python 教程，不到 500 行就能跑起来一个基本的 KV 存储服务器。自己实现一遍基本的字典和跳跃表，比你看十篇讲解文章都管用。

### 例子三：自己写一个 React

现在前端人人用 React，但你真的理解 Virtual DOM 是什么吗？有一个经典项目叫 Gooact，**只用 160 行 JavaScript** 就实现了 React 的核心渲染逻辑。

写完你再去看 React 源码，原来那些高大上的概念，拆解之后也没那么神秘。

## 适合谁？不适合谁？给你一个清晰的判断

我觉得不是所有人都适合用这个方法，得分阶段看：

### ✅ 非常适合

1. **已经入门，想突破瓶颈** — 会写 CRUD，能熟练用框架搭项目，但对底层原理一知半解，想往上走又不知道从哪开始。
2. **知识点零散，希望建立体系** — 碎片化看了很多博客视频，知识点在脑子里是散的，通过完整造一个轮子能把这些碎片串起来。
3. **准备面试，想把知识点吃透** — 面试官特别喜欢问「讲讲 XX 的原理」，你自己实现过一遍，说出来的东西和背书完全不是一个 Level。

### ❌ 可能不适合

1. **纯零基础，第一门编程语言** — 完全没基础上来就造轮子，挫败感会很强，建议还是先过一遍基础语法再试。
2. **赶项目，需要快速出活** — 这个方法是深度学习，不是快速开发。如果目标是两周上线一个产品，老老实实拿成熟框架用就好。
3. **只是想了解概念，并不打算深究** — 如果只是想知道「Redis 大概是干什么的」，看一篇 10 分钟的概述文章就够了，不需要动手写。

## 如果你想尝试，给你一个可复制的四步路线

我自己实践下来，这个节奏比较舒服，分享给你：

### 第一步：选一个「天天用但不太懂」的小东西

**别一开始就选「写一个操作系统」这种大项目**，找一个小的，能在一个周末做完的。推荐几个门槛不高收获很大的起点：

- 前端：写一个自己的 React（160 行）/ 简单 Promise 实现
- 后端：写一个自己的 Git（~1000 行 Python）/ 简单 Web 服务器
- 底层：写一个简单的哈希表 / 内存分配器

关键是**小而完整**，能快速拿到正反馈。

### 第二步：别先看教程，自己先想 15 分钟

拿到教程之后，先别急着跟着抄。花 15 分钟想清楚：这个问题如果让我来做，我会分成几步？核心数据结构怎么设计？哪些地方可能会踩坑？

想一遍再看作者的实现，收获会大很多。

### 第三步：一行一行敲，卡壳先自己想 10 分钟

不要复制粘贴，真的一行一行敲。遇到报错了，先自己 Google 解决，解决不了再看教程是怎么处理的。

这个过程虽然慢，但**解决问题的能力就是这么练出来的**。

### 第四步：做完之后，写三句话总结

不需要太长，就回答三个问题：

1. 这个东西核心解决了什么问题？
2. 哪个地方我卡了很久，最后是怎么想通的？
3. 有哪个设计技巧让你觉得「居然可以这样」？

写下来，不管是发博客还是存在自己笔记里，三个月后再看，印象还很深。

## 总结：这个仓库最珍贵的不是教程，是思维方式

`build-your-own-x` 火到 53 万星不是没有道理的。它最珍贵的地方，不是收集了 500 份教程，而是它把「费曼学习法」从一句正确的废话，变成了一件**每个人都能上手做的具体事情**。

你不需要等，不需要攒够了「足够多的知识」再开始，现在就能找一个小题目动手做。做着做着，你就会发现：原来那些看上去高大上的技术，拆解之后也没那么神秘。

> "What I cannot create, I do not understand."

这句话放在今天这个 AI 能帮你写代码的时代，反而更有意义。AI 能帮你写代码，但**理解为什么这么写**，只能靠你自己。
