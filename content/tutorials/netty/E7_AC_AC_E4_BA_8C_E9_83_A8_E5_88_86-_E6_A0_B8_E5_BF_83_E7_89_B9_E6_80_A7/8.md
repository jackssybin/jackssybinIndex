<h1>第8章：粘包与拆包解决方案</h1>
<h2>本章导读</h2>
<p>粘包和拆包是 TCP 网络编程中的常见问题。本章将深入讲解问题产生的原因、多种解决方案的详细实现，以及在实际项目中的最佳实践。</p>

<h2>8.1 TCP 粘包/拆包问题原理</h2>
<h3>8.1.1 问题现象</h3>
<p><strong>正常情况</strong>：</p>
<pre><code>发送: [D1] [D2] [D3]
接收: [D1] [D2] [D3]
</code></pre>
<p><strong>粘包</strong>：</p>
<pre><code>发送: [D1] [D2] [D3]
接收: [D1D2D3]
</code></pre>
<p><strong>拆包</strong>：</p>
<pre><code>发送: [D1大包]
接收: [D1前半] [D1后半]
</code></pre>
<p><strong>粘包+拆包</strong>：</p>
<pre><code>发送: [D1] [D2大包] [D3]
接收: [D1D2前半] [D2后半D3]
</code></pre>
<h3>8.1.2 产生原因</h3>
<p><strong>发送端原因</strong>：</p>
<ol>
<li><strong>Nagle 算法</strong>：合并小包，减少网络传输次数</li>
<li><strong>TCP 缓冲区</strong>：数据先写入缓冲区，达到一定量才发送</li>
</ol>
<p><strong>接收端原因</strong>：</p>
<ol>
<li><strong>TCP 缓冲区</strong>：接收的数据先放入缓冲区</li>
<li><strong>应用层读取速度</strong>：读取速度慢导致数据堆积</li>
</ol>
<p><strong>网络原因</strong>：</p>
<ol>
<li><strong>MSS 限制</strong>：最大报文段大小限制，大包会被拆分</li>
<li><strong>MTU 限制</strong>：最大传输单元限制</li>
</ol>

<h2>8.2 固定长度解码器（FixedLengthFrameDecoder）</h2>
<h3>8.2.1 原理</h3>
<p>每个消息固定长度，不足的用空格或特殊字符填充。</p>
<pre><code>消息格式：
+----------+----------+----------+
| Message1 | Message2 | Message3 |
|  10字节  |  10字节  |  10字节  |
+----------+----------+----------+
</code></pre>
<h3>8.2.2 实现</h3>
<pre><code class="language-java">// 服务端

pipeline.addLast(new FixedLengthHandler());

<h2>8.4 长度字段解码器（LengthFieldBasedFrameDecoder）</h2>
<h3>8.4.1 基本用法</h3>
<pre><code class="language-java">/**
 * 消息格式：
 * +--------+----------+
 * | Length | Content  |
 * | 4 byte | N bytes  |
 * +--------+----------+
 */
pipeline.addLast(new LengthFieldBasedFrameDecoder(
    1024,    // maxFrameLength: 最大帧长度
    0,       // lengthFieldOffset: 长度字段偏移量
    4,       // lengthFieldLength: 长度字段长度
    0,       // lengthAdjustment: 长度调整值
    4        // initialBytesToStrip: 跳过的字节数
));
<p>// 编码器
pipeline.addLast(new LengthFieldPrepender(4));
</code></pre></p>
<h3>8.4.2 参数详解</h3>
<p><strong>示例1：长度字段在开头，不包含长度字段本身</strong></p>
<pre><code>消息格式：
+--------+----------+
| Length | Content  |  Length = Content的长度
| 2 byte | N bytes  |
+--------+----------+
<p>解码后：
+----------+
| Content  |
| N bytes  |
+----------+</p>
<p>配置：
new LengthFieldBasedFrameDecoder(1024, 0, 2, 0, 2);
↑  ↑  ↑  ↑
|  |  |  跳过2字节（长度字段）
|  |  长度调整值0
|  长度字段2字节
长度字段从0开始
</code></pre></p>
<p><strong>示例2：长度字段包含自身</strong></p>
<pre><code>消息格式：
+--------+----------+
| Length | Content  |  Length = Content长度 + 2
| 2 byte | N bytes  |
+--------+----------+
<p>配置：
new LengthFieldBasedFrameDecoder(1024, 0, 2, -2, 2);
↑  ↑  ↑
|  |  长度调整值-2（减去长度字段）
|  长度字段2字节
长度字段从0开始
</code></pre></p>
<p><strong>示例3：长度字段在中间</strong></p>
<pre><code>消息格式：
+------+--------+----------+
| Type | Length | Content  |
| 1 B  | 2 B    | N bytes  |
+------+--------+----------+
<p>解码后：
+----------+
| Content  |
| N bytes  |
+----------+</p>
<p>配置：
new LengthFieldBasedFrameDecoder(1024, 1, 2, 0, 3);
↑  ↑  ↑  ↑
|  |  |  跳过3字节（Type+Length）
|  |  长度调整值0
|  长度字段2字节
长度字段从第1字节开始
</code></pre></p>
<p><strong>示例4：长度字段包含整个消息</strong></p>
<pre><code>消息格式：
+------+--------+----------+
| Type | Length | Content  |  Length = Type + Length + Content
| 1 B  | 2 B    | N bytes  |
+------+--------+----------+
<p>配置：
new LengthFieldBasedFrameDecoder(1024, 1, 2, -3, 0);
↑  ↑  ↑  ↑
|  |  |  不跳过
|  |  长度调整值-3
|  长度字段2字节
长度字段从第1字节开始
</code></pre></p>

<h2>8.5 自定义解码器</h2>
<h3>8.5.1 自定义协议设计</h3>
<pre><code>协议格式：

| Magic  | Type | Length | Content  |
| 4 byte | 1 B  | 4 B    | N bytes  |
+--------+------+--------+----------+
<p>Magic: 0x12345678（魔数，用于识别协议）
Type: 消息类型
Length: Content的长度
Content: 消息内容
</code></pre></p>

<pre><code class="language-java">public class CustomProtocolDecoder extends ByteToMessageDecoder {

<pre><code>// 2. 标记读取位置

msg.setType(type);
msg.setContent(new String(data, CharsetUtil.UTF_8));

    ctx.close();
    throw new CorruptedFrameException(&amp;amp;quot;非法的长度: &amp;amp;quot; + length);
}

</code></pre>
<p>}

<h2>8.7 本章小结</h2>
<p>本章我们深入学习了粘包/拆包问题：</p>
<p>✅ <strong>问题原理</strong>：TCP 流式协议导致的消息边界问题<br>
✅ <strong>固定长度</strong>：FixedLengthFrameDecoder<br>
✅ <strong>分隔符</strong>：LineBasedFrameDecoder、DelimiterBasedFrameDecoder<br>
✅ <strong>长度字段</strong>：LengthFieldBasedFrameDecoder（推荐）<br>
✅ <strong>自定义协议</strong>：自定义解码器</p>
<h3>关键要点</h3>
<ol>
<li><strong>粘包/拆包</strong>是 TCP 的固有特性，应用层必须处理</li>
<li><strong>LengthFieldBasedFrameDecoder</strong> 是最常用和推荐的方案</li>
<li>自定义协议要考虑<strong>魔数</strong>、<strong>版本</strong>、<strong>长度限制</strong></li>
<li>要防止<strong>恶意超大包</strong>攻击</li>
<li>编解码器要放在 Pipeline 的<strong>最前面</strong></li>
