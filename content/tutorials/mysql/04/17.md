<h1>第17章：数据恢复实战</h1>
<blockquote>
<p>数据恢复是DBA的核心技能，关键时刻能救命</p>
</blockquote>
<h2>17.1 数据恢复概述</h2>
<h3>17.1.1 常见的数据丢失场景</h3>
<p><strong>人为误操作（最常见）：</strong></p>
<ul>
<li>❌ 误删数据：<code>DELETE FROM users WHERE ...</code></li>
<li>❌ 误删表：<code>DROP TABLE users</code></li>
<li>❌ 误删库：<code>DROP DATABASE mydb</code></li>
<li>❌ 误更新：<code>UPDATE users SET password = '123456'</code> （忘记WHERE）</li>
<li>❌ 误TRUNCATE：<code>TRUNCATE TABLE orders</code></li>
</ul>
<p><strong>系统故障：</strong></p>
<ul>
<li>❌ 硬件故障（磁盘损坏）</li>
<li>❌ 软件Bug</li>
<li>❌ 主从同步错误</li>
</ul>
<p><strong>恶意攻击：</strong></p>
<ul>
<li>❌ SQL注入</li>
<li>❌ 勒索病毒</li>
<li>❌ 删库跑路</li>
</ul>
<h3>17.1.2 恢复目标</h3>
<p><strong>RTO（Recovery Time Objective）：</strong></p>
<ul>
<li>恢复时间目标</li>
<li>系统可以容忍的最长停机时间</li>
<li>例如：RTO = 1小时</li>
</ul>
<p><strong>RPO（Recovery Point Objective）：</strong></p>
<ul>
<li>恢复点目标</li>
<li>系统可以容忍的最大数据丢失量</li>
<li>例如：RPO = 5分钟</li>
</ul>
<p><strong>恢复策略：</strong></p>
<ul>
<li>RTO越小，成本越高</li>
<li>RPO越小，备份频率越高</li>
<li>需要在成本和需求之间平衡</li>
</ul>

<h2>17.3 基于binlog的时间点恢复 ⭐⭐⭐⭐⭐</h2>
<h3>17.3.1 binlog恢复原理</h3>
<pre><code>完整备份 + binlog = 任意时间点恢复
<p>时间线：
[全量备份]----[binlog]----[误操作]----[现在]
2:00        2:00-10:00    10:00      10:30</p>
<p>恢复步骤：</p>
<ol>
<li>恢复全量备份（2:00的数据）</li>
<li>应用binlog（2:00-9:59的变更）</li>
<li>跳过误操作（10:00的DELETE）</li>
<li>继续应用binlog（10:01-10:30的变更）
</code></pre></li>
</ol>
<h3>17.3.2 基于时间点恢复</h3>
<pre><code class="language-bash"># 场景：10:00误删了数据，需要恢复到9:59
<h1>1. 停止MySQL（防止新的写入）</h1>
<p>systemctl stop mysqld</p>
<h1>2. 恢复全量备份</h1>
<p>mysql -u root -p mydb &lt; /backup/mydb_20241110_0200.sql</p>
<h1>3. 查看备份时的binlog位置</h1>
<h1>从备份文件中查找：</h1>
<h1>-- CHANGE MASTER TO MASTER_LOG_FILE='mysql-bin.000010', MASTER_LOG_POS=154;</h1>
<p>grep &quot;CHANGE MASTER&quot; /backup/mydb_20241110_0200.sql</p>
<h1>4. 应用binlog（从备份点到误操作前）</h1>
<p>mysqlbinlog --start-position=154 <br>
--stop-datetime=&quot;2024-11-10 09:59:59&quot; <br>
/var/lib/mysql/mysql-bin.000010 | mysql -u root -p mydb</p>
<h1>5. 跳过误操作，继续应用后续binlog</h1>
<p>mysqlbinlog --start-datetime=&quot;2024-11-10 10:01:00&quot; <br>
/var/lib/mysql/mysql-bin.000010 <br>
/var/lib/mysql/mysql-bin.000011 | mysql -u root -p mydb</p>
<h1>6. 启动MySQL</h1>
<p>systemctl start mysqld</p>
<h1>7. 验证数据</h1>
<p>mysql -u root -p mydb -e &quot;SELECT COUNT(*) FROM users&quot;
</code></pre></p>
<h3>17.3.3 基于位置点恢复</h3>
<pre><code class="language-bash"># 场景：知道误操作的binlog位置
<h1>1. 查找误操作的位置</h1>
<p>mysqlbinlog -vv /var/lib/mysql/mysql-bin.000010 | grep -A 10 &quot;DELETE FROM users&quot;</p>
<h1>输出示例：</h1>
<h1>#241110 10:00:00 server id 1  end_log_pos 1234</h1>
<h1>DELETE FROM users WHERE id = 100</h1>
<h1>2. 恢复全量备份</h1>
<p>mysql -u root -p mydb &lt; /backup/mydb_20241110_0200.sql</p>
<h1>3. 应用binlog到误操作前（position 1233）</h1>
<p>mysqlbinlog --start-position=154 <br>
--stop-position=1233 <br>
/var/lib/mysql/mysql-bin.000010 | mysql -u root -p mydb</p>
<h1>4. 跳过误操作，继续应用后续binlog（从position 1500开始）</h1>
<p>mysqlbinlog --start-position=1500 <br>
/var/lib/mysql/mysql-bin.000010 | mysql -u root -p mydb
</code></pre></p>
<h3>17.3.4 恢复误删除的数据</h3>
<pre><code class="language-bash"># 场景：误执行了 DELETE FROM users WHERE city = '北京'
<h1>方法1：从binlog中提取被删除的数据</h1>
<h1>1. 找到DELETE语句的binlog位置</h1>
<p>mysqlbinlog -vv --base64-output=DECODE-ROWS <br>
/var/lib/mysql/mysql-bin.000010 | grep -B 5 &quot;DELETE FROM users&quot;</p>
<h1>2. 提取DELETE语句前的数据（ROW格式的binlog）</h1>
<h1>binlog中记录了被删除行的完整数据</h1>
<h1>可以手动构造INSERT语句恢复</h1>
<h1>方法2：使用binlog2sql工具（推荐）</h1>
<h1>安装binlog2sql</h1>
<p>git clone <a href="https://github.com/danfengcao/binlog2sql.git">https://github.com/danfengcao/binlog2sql.git</a>
cd binlog2sql</p>
<h1>生成回滚SQL</h1>
<p>python <a href="http://binlog2sql.py">binlog2sql.py</a> <br>
-h127.0.0.1 -P3306 -uroot -p'password' <br>
--start-file='mysql-bin.000010' <br>
--start-datetime='2024-11-10 09:00:00' <br>
--stop-datetime='2024-11-10 11:00:00' <br>
-d mydb -t users <br>
--flashback &gt; rollback.sql</p>
<h1>执行回滚</h1>
<p>mysql -u root -p mydb &lt; rollback.sql
</code></pre></p>

<h2>17.4 基于XtraBackup的恢复</h2>
<h3>17.4.1 全量恢复</h3>
<pre><code class="language-bash"># 场景：服务器故障，需要完整恢复
<h1>1. 准备备份</h1>
<p>xtrabackup --prepare --target-dir=/backup/full_20241110</p>
<h1>2. 停止MySQL</h1>
<p>systemctl stop mysqld</p>
<h1>3. 清空数据目录</h1>
<p>rm -rf /var/lib/mysql/*</p>
<h1>4. 恢复数据</h1>
<p>xtrabackup --copy-back --target-dir=/backup/full_20241110</p>
<h1>5. 修改权限</h1>
<p>chown -R mysql:mysql /var/lib/mysql</p>
<h1>6. 启动MySQL</h1>
<p>systemctl start mysqld</p>
<h1>7. 验证</h1>
<p>mysql -u root -p -e &quot;SHOW DATABASES&quot;
</code></pre></p>
<h3>17.4.2 增量恢复</h3>
<pre><code class="language-bash"># 场景：恢复全量+增量备份
<h1>1. 准备全量备份（不回滚）</h1>
<p>xtrabackup --prepare --apply-log-only <br>
--target-dir=/backup/full</p>
<h1>2. 应用第一次增量</h1>
<p>xtrabackup --prepare --apply-log-only <br>
--target-dir=/backup/full <br>
--incremental-dir=/backup/inc1</p>
<h1>3. 应用第二次增量（最后一次）</h1>
<p>xtrabackup --prepare <br>
--target-dir=/backup/full <br>
--incremental-dir=/backup/inc2</p>
<h1>4. 停止MySQL并恢复</h1>
<p>systemctl stop mysqld
rm -rf /var/lib/mysql/*
xtrabackup --copy-back --target-dir=/backup/full
chown -R mysql:mysql /var/lib/mysql
systemctl start mysqld
</code></pre></p>
<h3>17.4.3 XtraBackup + binlog恢复</h3>
<pre><code class="language-bash"># 场景：恢复到最新状态
<h1>1. 恢复XtraBackup备份</h1>
<p>xtrabackup --prepare --target-dir=/backup/full
xtrabackup --copy-back --target-dir=/backup/full
chown -R mysql:mysql /var/lib/mysql</p>
<h1>2. 查看备份时的binlog位置</h1>
<p>cat /backup/full/xtrabackup_binlog_info</p>
<h1>输出：mysql-bin.000015  12345</h1>
<h1>3. 应用binlog（从备份点到现在）</h1>
<p>mysqlbinlog --start-position=12345 <br>
/var/lib/mysql/mysql-bin.000015 <br>
/var/lib/mysql/mysql-bin.000016 | mysql -u root -p</p>
<h1>4. 启动MySQL</h1>
<p>systemctl start mysqld
</code></pre></p>

<h3>17.5.5 案例5：主从同步错误导致数据不一致</h3>
<pre><code class="language-bash"># 场景：从库执行了写操作，导致主从数据不一致
<h1>恢复步骤：</h1>
<h1>1. 停止从库的复制</h1>
<p>mysql -u root -p -e &quot;STOP SLAVE&quot;</p>
<h1>2. 在从库上重新搭建（使用主库备份）</h1>
<h1>方法1：使用mysqldump</h1>
<h1>在主库上备份</h1>
<p>mysqldump -u root -p <br>
--all-databases <br>
--single-transaction <br>
--master-data=2 <br>
--flush-logs &gt; master_backup.sql</p>
<h1>传输到从库</h1>
<p>scp master_backup.sql slave_server:/backup/</p>
<h1>在从库上恢复</h1>
<p>mysql -u root -p &lt; /backup/master_backup.sql</p>
<h1>方法2：使用XtraBackup（推荐，大数据库）</h1>
<h1>在主库上备份</h1>
<p>xtrabackup --backup <br>
--user=root <br>
--password=your_password <br>
--target-dir=/backup/master_full</p>
<h1>传输到从库</h1>
<p>rsync -avz /backup/master_full slave_server:/backup/</p>
<h1>在从库上恢复</h1>
<p>xtrabackup --prepare --target-dir=/backup/master_full
systemctl stop mysqld
rm -rf /var/lib/mysql/*
xtrabackup --copy-back --target-dir=/backup/master_full
chown -R mysql:mysql /var/lib/mysql
systemctl start mysqld</p>
<h1>3. 重新配置主从复制</h1>
<h1>从备份中获取binlog位置</h1>
<p>grep &quot;CHANGE MASTER&quot; /backup/master_backup.sql</p>
<h1>或</h1>
<p>cat /backup/master_full/xtrabackup_binlog_info</p>
<h1>配置从库</h1>
<p>mysql -u root -p &lt;&lt;EOF
CHANGE MASTER TO
MASTER_HOST='master_ip',
MASTER_USER='repl',
MASTER_PASSWORD='repl_password',
MASTER_LOG_FILE='mysql-bin.000020',
MASTER_LOG_POS=12345;
START SLAVE;
EOF</p>
<h1>4. 检查主从状态</h1>
<p>mysql -u root -p -e &quot;SHOW SLAVE STATUS\G&quot;</p>
<h1>5. 设置从库为只读（防止再次写入）</h1>
<p>mysql -u root -p -e &quot;SET GLOBAL read_only = 1&quot;
</code></pre></p>

<h2>17.7 数据恢复最佳实践</h2>
<h3>17.7.1 预防措施</h3>
<pre><code class="language-sql">-- 1. 开启binlog
[mysqld]
log_bin = mysql-bin
binlog_format = ROW  -- ROW格式记录完整数据，便于恢复
expire_logs_days = 7
<p>-- 2. 开启慢查询日志
slow_query_log = 1
long_query_time = 1</p>
<p>-- 3. 设置从库为只读
SET GLOBAL read_only = 1;
SET GLOBAL super_read_only = 1;</p>
<p>-- 4. 使用安全模式（防止误操作）
SET sql_safe_updates = 1;</p>
<p>-- 开启后，没有WHERE或LIMIT的UPDATE/DELETE会报错
UPDATE users SET password = '123456';  -- 报错
UPDATE users SET password = '123456' WHERE id = 1;  -- 正常</p>
<p>-- 5. 重要操作前先备份
mysqldump -u root -p mydb users &gt; users_backup_before_update.sql</p>
<p>-- 6. 使用事务（可以回滚）
START TRANSACTION;
UPDATE users SET status = 0 WHERE city = '北京';
-- 检查影响的行数
SELECT ROW_COUNT();
-- 如果不对，回滚
ROLLBACK;
-- 如果正确，提交
COMMIT;</p>
<p>-- 7. 先SELECT再UPDATE/DELETE
SELECT * FROM users WHERE city = '北京';  -- 先查看
UPDATE users SET status = 0 WHERE city = '北京';  -- 再更新
</code></pre></p>
<h3>17.7.2 恢复流程规范</h3>
<pre><code>1. 发现问题
   ↓
2. 立即停止应用（防止新数据写入）
   ↓
3. 评估影响范围
   ↓
4. 制定恢复方案
   ↓
5. 在测试环境验证
   ↓
6. 执行恢复
   ↓
7. 验证数据完整性
   ↓
8. 恢复应用访问
   ↓
9. 监控系统状态
   ↓
10. 编写事故报告
</code></pre>
<h3>17.7.3 恢复检查清单</h3>
<pre><code>✅ 确认问题类型（误删、误更新、硬件故障等）
✅ 确认影响范围（哪些表、多少数据）
✅ 确认最新可用备份
✅ 确认binlog是否完整
✅ 停止应用访问
✅ 制定恢复方案
✅ 在测试环境验证
✅ 记录恢复步骤

</code></pre>
