<h1>第07章：存储引擎深入理解</h1>
<blockquote>
<p>存储引擎是MySQL的核心组件，决定了数据的存储方式和性能特性</p>
</blockquote>
<h2>7.1 存储引擎概述</h2>
<h3>7.1.1 什么是存储引擎？</h3>
<p><strong>存储引擎（Storage Engine）：</strong></p>
<ul>
<li>MySQL的数据存储和管理机制</li>
<li>负责数据的存储、索引、事务等功能</li>
<li>不同的存储引擎有不同的特性和适用场景</li>
</ul>
<p><strong>MySQL的插件式存储引擎架构：</strong></p>
<pre><code>应用层
    ↓
SQL层（解析、优化）
    ↓
存储引擎层（InnoDB、MyISAM等）
    ↓
文件系统
</code></pre>
<h3>7.1.2 查看存储引擎</h3>
<pre><code class="language-sql">-- 查看MySQL支持的所有存储引擎
SHOW ENGINES;
<p>-- 查看当前默认存储引擎
SHOW VARIABLES LIKE 'default_storage_engine';</p>
<p>-- 查看表使用的存储引擎
SHOW TABLE STATUS FROM database_name;
SHOW CREATE TABLE table_name;
</code></pre></p>

<h2>7.2 InnoDB存储引擎 ⭐⭐⭐⭐⭐</h2>
<h3>7.2.1 InnoDB概述</h3>
<p><strong>InnoDB特点：</strong></p>
<ul>
<li>✅ <strong>支持事务</strong>（ACID特性）</li>
<li>✅ <strong>支持外键</strong></li>
<li>✅ <strong>行级锁</strong>（并发性能好）</li>
<li>✅ <strong>MVCC</strong>（多版本并发控制）</li>
<li>✅ <strong>崩溃恢复</strong>（crash-safe）</li>
<li>✅ <strong>聚簇索引</strong>（数据和索引存储在一起）</li>
</ul>
<p><strong>适用场景：</strong></p>
<ul>
<li>需要事务支持的应用（转账、订单等）</li>
<li>高并发读写场景</li>
<li>需要崩溃恢复的场景</li>
<li>大部分OLTP（在线事务处理）应用</li>
</ul>
<p><strong>MySQL 5.7默认存储引擎：InnoDB</strong></p>
<h3>7.2.2 InnoDB文件结构</h3>
<pre><code class="language-bash"># InnoDB数据文件
/var/lib/mysql/
├── ibdata1              # 系统表空间（共享表空间）
├── ib_logfile0          # redo log文件
├── ib_logfile1          # redo log文件
└── database_name/
    ├── table_name.frm   # 表结构定义文件
    └── table_name.ibd   # 表数据和索引文件（独立表空间）
</code></pre>
<p><strong>文件说明：</strong></p>
<ul>
<li><strong>ibdata1</strong>：系统表空间，存储数据字典、undo log等</li>
<li><strong>ib_logfile</strong>：redo log，用于崩溃恢复</li>
<li><strong>.frm</strong>：表结构定义文件（所有引擎都有）</li>
<li><strong>.ibd</strong>：独立表空间，存储表数据和索引</li>
</ul>
<h3>7.2.3 InnoDB核心特性</h3>
<h4>1. 事务支持</h4>
<pre><code class="language-sql">-- InnoDB支持完整的ACID事务
START TRANSACTION;
<p>UPDATE accounts SET balance = balance - 100 WHERE user_id = 1;
UPDATE accounts SET balance = balance + 100 WHERE user_id = 2;</p>
<p>COMMIT;  -- 或 ROLLBACK
</code></pre></p>
<h4>2. 行级锁</h4>
<pre><code class="language-sql">-- InnoDB使用行级锁，并发性能好
-- 会话1
START TRANSACTION;
UPDATE users SET age = 26 WHERE id = 1;  -- 锁定id=1的行
<p>-- 会话2（不会被阻塞）
UPDATE users SET age = 31 WHERE id = 2;  -- 可以正常执行</p>
<p>-- 会话3（会被阻塞）
UPDATE users SET age = 27 WHERE id = 1;  -- 等待会话1释放锁
</code></pre></p>
<h4>3. 外键约束</h4>
<pre><code class="language-sql">-- InnoDB支持外键
CREATE TABLE orders (
    id INT PRIMARY KEY AUTO_INCREMENT,
    user_id INT,
    amount DECIMAL(10,2),
    FOREIGN KEY (user_id) REFERENCES users(id)
        ON DELETE CASCADE
        ON UPDATE CASCADE
) ENGINE=InnoDB;
<p>-- 外键约束：
-- 1. 保证数据完整性
-- 2. 级联删除/更新
-- 3. 性能开销较大
</code></pre></p>
<h4>4. MVCC多版本并发控制</h4>
<pre><code class="language-sql">-- InnoDB通过MVCC实现高并发
-- 读不加锁，写不阻塞读
-- 详见第08章：事务与并发控制
</code></pre>
<h4>5. 崩溃恢复</h4>
<pre><code class="language-sql">-- InnoDB通过redo log实现崩溃恢复
-- MySQL异常关闭后，重启时自动恢复未提交的事务
-- 详见第15章：MySQL日志系统
</code></pre>
<h3>7.2.4 InnoDB配置参数</h3>
<pre><code class="language-ini">[mysqld]
# InnoDB缓冲池大小（最重要的参数）
# 建议设置为物理内存的50%-70%
innodb_buffer_pool_size = 1G
<h1>InnoDB日志文件大小</h1>
<p>innodb_log_file_size = 512M</p>
<h1>InnoDB刷新日志策略</h1>
<h1>0: 每秒刷新（性能最好，可能丢失1秒数据）</h1>
<h1>1: 每次事务提交刷新（最安全，性能较差）⭐ 推荐</h1>
<h1>2: 每次事务提交写入OS缓存，每秒刷新</h1>
<p>innodb_flush_log_at_trx_commit = 1</p>
<h1>InnoDB文件格式</h1>
<p>innodb_file_format = Barracuda</p>
<h1>独立表空间（推荐开启）</h1>
<p>innodb_file_per_table = 1</p>
<h1>InnoDB IO线程数</h1>
<p>innodb_read_io_threads = 4
innodb_write_io_threads = 4</p>
<h1>InnoDB锁等待超时时间（秒）</h1>
<p>innodb_lock_wait_timeout = 50
</code></pre></p>
<h3>7.2.5 InnoDB索引结构</h3>
<pre><code class="language-sql">-- InnoDB使用聚簇索引（Clustered Index）
-- 特点：
-- 1. 主键索引的叶子节点存储完整的行数据
-- 2. 二级索引的叶子节点存储主键值
-- 3. 通过二级索引查询需要回表
<p>-- 示例
CREATE TABLE users (
id INT PRIMARY KEY,        -- 聚簇索引
username VARCHAR(50),
age INT,
INDEX idx_age (age)        -- 二级索引
) ENGINE=InnoDB;</p>
<p>-- 查询过程：
-- SELECT * FROM users WHERE age = 25;
-- 1. 通过idx_age找到age=25的主键id
-- 2. 通过主键id回表查询完整数据
</code></pre></p>

<h2>7.3 MyISAM存储引擎</h2>
<h3>7.3.1 MyISAM概述</h3>
<p><strong>MyISAM特点：</strong></p>
<ul>
<li>❌ <strong>不支持事务</strong></li>
<li>❌ <strong>不支持外键</strong></li>
<li>✅ <strong>表级锁</strong>（并发性能差）</li>
<li>✅ <strong>支持全文索引</strong>（MySQL 5.6之前）</li>
<li>✅ <strong>压缩表</strong>（节省空间）</li>
<li>✅ <strong>查询速度快</strong>（简单查询）</li>
</ul>
<p><strong>适用场景：</strong></p>
<ul>
<li>只读或读多写少的应用</li>
<li>不需要事务的场景</li>
<li>日志、历史数据等</li>
<li>MySQL 5.7已不推荐使用</li>
</ul>
<h3>7.3.2 MyISAM文件结构</h3>
<pre><code class="language-bash"># MyISAM数据文件
/var/lib/mysql/database_name/
├── table_name.frm   # 表结构定义文件
├── table_name.MYD   # 数据文件（MYData）
└── table_name.MYI   # 索引文件（MYIndex）
</code></pre>
<p><strong>文件说明：</strong></p>
<ul>
<li><strong>.frm</strong>：表结构定义</li>
<li><strong>.MYD</strong>：表数据</li>
<li><strong>.MYI</strong>：表索引</li>
</ul>
<h3>7.3.3 MyISAM核心特性</h3>
<h4>1. 表级锁</h4>
<pre><code class="language-sql">-- MyISAM使用表级锁，并发性能差
-- 会话1
UPDATE users SET age = 26 WHERE id = 1;  -- 锁定整个表
<p>-- 会话2（被阻塞）
UPDATE users SET age = 31 WHERE id = 2;  -- 等待表锁释放
SELECT * FROM users WHERE id = 3;        -- 读操作也可能被阻塞
</code></pre></p>
<h4>2. 不支持事务</h4>
<pre><code class="language-sql">-- MyISAM不支持事务
START TRANSACTION;  -- 无效
UPDATE users SET age = 26 WHERE id = 1;
ROLLBACK;  -- 无法回滚，数据已经修改
</code></pre>
<h4>3. 不支持崩溃恢复</h4>
<pre><code class="language-sql">-- MyISAM没有redo log
-- MySQL异常关闭可能导致数据损坏
-- 需要使用REPAIR TABLE修复
REPAIR TABLE table_name;
</code></pre>
<h4>4. 支持压缩表</h4>
<pre><code class="language-sql">-- MyISAM支持压缩表（只读）
-- 使用myisampack工具压缩
-- 可以节省50%-80%的磁盘空间
</code></pre>
<h3>7.3.4 MyISAM索引结构</h3>
<pre><code class="language-sql">-- MyISAM使用非聚簇索引（Non-Clustered Index）
-- 特点：
-- 1. 索引和数据分开存储
-- 2. 索引的叶子节点存储数据的物理地址
-- 3. 主键索引和二级索引结构相同
<p>-- 示例
CREATE TABLE users (
id INT PRIMARY KEY,
username VARCHAR(50),
age INT,
INDEX idx_age (age)
) ENGINE=MyISAM;</p>
<p>-- 查询过程：
-- SELECT * FROM users WHERE age = 25;
-- 1. 通过idx_age找到age=25的数据物理地址
-- 2. 通过物理地址直接读取数据
</code></pre></p>

<h3>7.4.2 性能对比</h3>
<pre><code class="language-sql">-- 读性能对比
-- 简单查询：MyISAM略快（无事务开销）
-- 复杂查询：InnoDB更快（行锁、MVCC）
<p>-- 写性能对比
-- 单线程写入：MyISAM可能更快
-- 多线程写入：InnoDB远超MyISAM（行锁）</p>
<p>-- 并发性能对比
-- 读写混合：InnoDB远超MyISAM（MVCC）
-- 只读场景：MyISAM略有优势
</code></pre></p>
<h3>7.4.3 存储空间对比</h3>
<pre><code class="language-sql">-- 创建测试表
CREATE TABLE test_innodb (

    data VARCHAR(100)
) ENGINE=InnoDB;
<p>CREATE TABLE test_myisam (
id INT PRIMARY KEY AUTO_INCREMENT,
data VARCHAR(100)
) ENGINE=MyISAM;</p>
<p>-- 插入相同数据
-- InnoDB文件大小：约1.2倍MyISAM
-- 原因：InnoDB存储事务信息、MVCC版本等
</code></pre></p>
<h3>7.4.4 如何选择？</h3>
<p><strong>选择InnoDB的场景（推荐）：</strong></p>
<ul>
<li>✅ 需要事务支持（转账、订单等）</li>
<li>✅ 高并发读写</li>
<li>✅ 需要崩溃恢复</li>
<li>✅ 需要外键约束</li>
<li>✅ 大部分应用场景</li>
</ul>
<p><strong>选择MyISAM的场景（不推荐）：</strong></p>
<ul>
<li>✅ 只读或读多写少</li>
<li>✅ 不需要事务</li>
<li>✅ 历史数据、日志表</li>
<li>⚠️ MySQL 5.7已不推荐使用</li>
</ul>
<p><strong>建议：</strong></p>
<ul>
<li><strong>MySQL 5.7默认使用InnoDB</strong></li>
<li><strong>99%的场景都应该使用InnoDB</strong></li>
<li><strong>除非有特殊需求，否则不要使用MyISAM</strong></li>
</ul>

<h2>7.5 其他存储引擎</h2>
<h3>7.5.1 Memory引擎</h3>
<pre><code class="language-sql">-- Memory引擎：数据存储在内存中
CREATE TABLE cache_table (
    id INT PRIMARY KEY,
    data VARCHAR(100)
) ENGINE=Memory;
<p>-- 特点：
-- ✅ 速度极快（内存访问）
-- ❌ 数据易丢失（重启后数据消失）
-- ❌ 不支持TEXT/BLOB类型
-- ❌ 表级锁</p>
<p>-- 适用场景：
-- 临时数据、缓存
-- 会话数据
-- 中间结果集
</code></pre></p>
<h3>7.5.2 Archive引擎</h3>
<pre><code class="language-sql">-- Archive引擎：压缩存储，适合归档
CREATE TABLE log_archive (
    id INT AUTO_INCREMENT,
    log_time TIMESTAMP,
    log_content TEXT,
    KEY(id)
) ENGINE=Archive;
<p>-- 特点：
-- ✅ 高压缩比（1:10）
-- ✅ 支持INSERT和SELECT
-- ❌ 不支持UPDATE和DELETE
-- ❌ 不支持索引（除了AUTO_INCREMENT）</p>
<p>-- 适用场景：
-- 历史数据归档
-- 日志存储
-- 只增不改的数据
</code></pre></p>
<h3>7.5.3 CSV引擎</h3>
<pre><code class="language-sql">-- CSV引擎：以CSV格式存储数据
CREATE TABLE csv_table (
    id INT,
    name VARCHAR(50),
    age INT
) ENGINE=CSV;
<p>-- 特点：
-- ✅ 数据以CSV格式存储，可直接编辑
-- ❌ 不支持索引
-- ❌ 不支持NULL值</p>
<p>-- 适用场景：
-- 数据导入导出
-- 与Excel等工具交互
</code></pre></p>
<h3>7.5.4 Blackhole引擎</h3>
<pre><code class="language-sql">-- Blackhole引擎：黑洞引擎，不存储数据
CREATE TABLE blackhole_table (
    id INT,
    data VARCHAR(100)
) ENGINE=Blackhole;
<p>-- 特点：
-- 写入的数据会被丢弃
-- 但会记录binlog</p>
<p>-- 适用场景：
-- 主从复制中的中继服务器
-- 性能测试
-- 过滤不需要的数据
</code></pre></p>

<h2>7.6 存储引擎的使用</h2>
<h3>7.6.1 指定存储引擎</h3>
<pre><code class="language-sql">-- 方法1：创建表时指定
CREATE TABLE users (
    id INT PRIMARY KEY,
    username VARCHAR(50)
) ENGINE=InnoDB;
<p>-- 方法2：修改表的存储引擎
ALTER TABLE users ENGINE=InnoDB;</p>
<p>-- 方法3：设置默认存储引擎
SET default_storage_engine=InnoDB;</p>
<p>-- 方法4：配置文件中设置
-- my.cnf
[mysqld]
default-storage-engine=InnoDB
</code></pre></p>
<h3>7.6.2 查看存储引擎</h3>
<pre><code class="language-sql">-- 查看表的存储引擎
SHOW TABLE STATUS LIKE 'users';
SHOW CREATE TABLE users;
<p>-- 查看所有表的存储引擎
SELECT
TABLE_NAME,
ENGINE,
TABLE_ROWS,
DATA_LENGTH,
INDEX_LENGTH
FROM information_schema.TABLES
WHERE TABLE_SCHEMA = 'database_name';
</code></pre></p>
<h3>7.6.3 转换存储引擎</h3>
<pre><code class="language-sql">-- 方法1：ALTER TABLE（推荐）
ALTER TABLE users ENGINE=InnoDB;
<p>-- 方法2：导出导入
-- 1. 导出数据
mysqldump -u root -p database_name table_name &gt; table_name.sql</p>
<p>-- 2. 修改SQL文件中的ENGINE
-- 3. 导入数据
mysql -u root -p database_name &lt; table_name.sql</p>
<p>-- 方法3：CREATE...SELECT
CREATE TABLE users_innodb LIKE users;
ALTER TABLE users_innodb ENGINE=InnoDB;
INSERT INTO users_innodb SELECT * FROM users;</p>
<p>-- 注意事项：
-- 1. 转换过程会锁表
-- 2. 大表转换时间较长
-- 3. 建议在业务低峰期操作
</code></pre></p>

<h2>7.7 存储引擎最佳实践</h2>
<h3>7.7.1 选择建议</h3>
<pre><code class="language-sql">-- ✅ 推荐：默认使用InnoDB
CREATE TABLE users (
    id INT PRIMARY KEY AUTO_INCREMENT,
    username VARCHAR(50)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
<p>-- ✅ 特殊场景：临时数据使用Memory
CREATE TABLE session_data (
session_id VARCHAR(64) PRIMARY KEY,
data TEXT
) ENGINE=Memory;</p>
<p>-- ✅ 特殊场景：归档数据使用Archive
CREATE TABLE access_log_archive (
id BIGINT AUTO_INCREMENT,
access_time TIMESTAMP,
log_content TEXT,
KEY(id)
) ENGINE=Archive;
</code></pre></p>
