<h1>第25章：分库分表</h1>
<blockquote>
<p>分库分表是解决海量数据存储和高并发访问的核心方案</p>
</blockquote>
<h2>25.1 分库分表概述</h2>
<h3>25.1.1 为什么需要分库分表？</h3>
<p><strong>问题场景：</strong></p>
<ul>
<li>❌ 单表数据量过大（&gt;1000万）</li>
<li>❌ 查询慢，索引失效</li>
<li>❌ 单库连接数不够</li>
<li>❌ 单库I/O瓶颈</li>
</ul>
<p><strong>示例：</strong></p>
<pre><code class="language-sql">-- 订单表：1亿条数据
SELECT * FROM orders WHERE user_id = 12345;
-- 即使有索引，查询也很慢
<p>-- 问题：
-- 1. 表太大，索引树深度增加
-- 2. Buffer Pool无法缓存所有数据
-- 3. 单表锁竞争激烈
</code></pre></p>
<h3>25.1.2 分库分表的优势</h3>
<p><strong>分表优势：</strong></p>
<ul>
<li>✅ 减小单表数据量</li>
<li>✅ 提高查询性能</li>
<li>✅ 减少锁竞争</li>
<li>✅ 提高并发能力</li>
</ul>
<p><strong>分库优势：</strong></p>
<ul>
<li>✅ 分散连接数</li>
<li>✅ 分散I/O压力</li>
<li>✅ 提高并发能力</li>
<li>✅ 突破单机性能瓶颈</li>
</ul>
<h3>25.1.3 分库分表类型</h3>
<p><strong>垂直分库：</strong></p>
<pre><code>原来：
单库：用户表、订单表、商品表
<p>分库后：
用户库：用户表
订单库：订单表
商品库：商品表
</code></pre></p>
<p><strong>垂直分表：</strong></p>
<pre><code>原来：
用户表：id, name, age, address, avatar, description
<p>分表后：
用户基本表：id, name, age
用户详情表：id, address, avatar, description
</code></pre></p>
<p><strong>水平分库：</strong></p>
<pre><code>原来：
单库：订单表（1亿条）
<p>分库后：
订单库1：订单表（2500万条）
订单库2：订单表（2500万条）
订单库3：订单表（2500万条）
订单库4：订单表（2500万条）
</code></pre></p>
<p><strong>水平分表：</strong></p>
<pre><code>原来：
订单表：1亿条
<p>分表后：
订单表_0：2500万条
订单表_1：2500万条
订单表_2：2500万条
订单表_3：2500万条
</code></pre></p>

<h2>25.2 分片策略 ⭐⭐⭐⭐⭐</h2>
<h3>25.2.1 范围分片（Range）</h3>
<p><strong>原理：</strong></p>
<ul>
<li>按照某个字段的范围分片</li>
<li>例如：按ID、时间范围</li>
</ul>
<p><strong>示例：</strong></p>
<pre><code class="language-sql">-- 按ID范围分片
订单表_0：id &lt; 2500万
订单表_1：2500万 &lt;= id &lt; 5000万
订单表_2：5000万 &lt;= id &lt; 7500万
订单表_3：id &gt;= 7500万
<p>-- 按时间范围分片
订单表_2023：2023年的订单
订单表_2024：2024年的订单
订单表_2025：2025年的订单
</code></pre></p>
<p><strong>优点：</strong></p>
<ul>
<li>✅ 简单易理解</li>
<li>✅ 范围查询性能好</li>
<li>✅ 扩容方便</li>
</ul>
<p><strong>缺点：</strong></p>
<ul>
<li>❌ 数据分布可能不均匀</li>
<li>❌ 热点问题（最新数据访问频繁）</li>
</ul>
<p><strong>适用场景：</strong></p>
<ul>
<li>按时间查询的业务</li>
<li>有明显范围的数据</li>
</ul>
<hr>
<h3>25.2.2 哈希分片（Hash）</h3>
<p><strong>原理：</strong></p>
<ul>
<li>对分片键进行哈希运算</li>
<li>根据哈希值确定分片</li>
</ul>
<p><strong>示例：</strong></p>
<pre><code class="language-java">// 按用户ID哈希分片
int shardIndex = userId % 4;  // 4个分片
<p>// 用户1001 → 1001 % 4 = 1 → 订单表_1
// 用户1002 → 1002 % 4 = 2 → 订单表_2
// 用户1003 → 1003 % 4 = 3 → 订单表_3
// 用户1004 → 1004 % 4 = 0 → 订单表_0
</code></pre></p>
<p><strong>优点：</strong></p>
<ul>
<li>✅ 数据分布均匀</li>
<li>✅ 避免热点问题</li>
</ul>
<p><strong>缺点：</strong></p>
<ul>
<li>❌ 范围查询性能差</li>
<li>❌ 扩容困难（需要重新哈希）</li>
</ul>
<p><strong>适用场景：</strong></p>
<ul>
<li>按ID查询的业务</li>
<li>数据分布要求均匀</li>
</ul>
<hr>
<h3>25.2.3 一致性哈希（Consistent Hash）</h3>
<p><strong>原理：</strong></p>
<ul>
<li>解决哈希分片扩容问题</li>
<li>使用哈希环</li>
</ul>
<p><strong>示例：</strong></p>
<pre><code>哈希环：0 ~ 2^32-1
<p>节点分布：
节点1：hash(node1) = 100
节点2：hash(node2) = 200
节点3：hash(node3) = 300</p>
<p>数据分布：
数据A：hash(A) = 50  → 节点1（顺时针最近）
数据B：hash(B) = 150 → 节点2
数据C：hash(C) = 250 → 节点3</p>
<p>扩容（增加节点4）：
节点4：hash(node4) = 150
只需迁移部分数据（150-200之间的数据）
</code></pre></p>
<p><strong>优点：</strong></p>
<ul>
<li>✅ 扩容时只需迁移部分数据</li>
<li>✅ 数据分布相对均匀</li>
</ul>
<p><strong>缺点：</strong></p>
<ul>
<li>❌ 实现复杂</li>
<li>❌ 可能出现数据倾斜</li>
</ul>
<hr>
<h3>25.2.4 地理位置分片（Geo）</h3>
<p><strong>原理：</strong></p>
<ul>
<li>按照地理位置分片</li>
<li>例如：按省份、城市</li>
</ul>
<p><strong>示例：</strong></p>
<pre><code class="language-sql">-- 按地区分片
订单库_华北：北京、天津、河北的订单
订单库_华东：上海、江苏、浙江的订单
订单库_华南：广东、福建、海南的订单
订单库_华中：湖北、湖南、河南的订单
</code></pre>
<p><strong>优点：</strong></p>
<ul>
<li>✅ 就近访问，延迟低</li>
<li>✅ 符合业务逻辑</li>
</ul>
<p><strong>缺点：</strong></p>
<ul>
<li>❌ 数据分布可能不均匀</li>
<li>❌ 跨地区查询复杂</li>
</ul>

<h2>25.3 分片键选择 ⭐⭐⭐⭐⭐</h2>
<h3>25.3.1 分片键选择原则</h3>
<p><strong>原则1：高频查询字段</strong></p>
<pre><code class="language-sql">-- ✅ 好的分片键
SELECT * FROM orders WHERE user_id = 12345;
-- user_id是高频查询字段，适合作为分片键
<p>-- ❌ 不好的分片键
SELECT * FROM orders WHERE order_status = 1;
-- order_status不是唯一标识，不适合作为分片键
</code></pre></p>
<p><strong>原则2：数据分布均匀</strong></p>
<pre><code class="language-sql">-- ✅ 好的分片键：user_id
-- 每个用户的订单数量相对均匀
<p>-- ❌ 不好的分片键：order_date
-- 某些日期（如双11）订单量特别大
</code></pre></p>
<p><strong>原则3：避免跨分片查询</strong></p>
<pre><code class="language-sql">-- ✅ 好的分片键：user_id
SELECT * FROM orders WHERE user_id = 12345;
-- 只需查询一个分片
<p>-- ❌ 不好的分片键：order_id
SELECT * FROM orders WHERE user_id = 12345;
-- 需要查询所有分片（user_id不是分片键）
</code></pre></p>
<p><strong>原则4：业务相关性</strong></p>
<pre><code class="language-sql">-- ✅ 好的分片键：user_id
-- 订单表按user_id分片
-- 用户表也按user_id分片
-- 可以进行分片内JOIN
<p>-- ❌ 不好的分片键：order_id
-- 订单表按order_id分片
-- 用户表按user_id分片
-- 无法进行分片内JOIN
</code></pre></p>
<h3>25.3.2 常见分片键</h3>
<p><strong>订单表：</strong></p>
<ul>
<li>推荐：user_id（用户维度查询多）</li>
<li>备选：order_id（订单维度查询）</li>
</ul>
<p><strong>用户表：</strong></p>
<ul>
<li>推荐：user_id</li>
</ul>
<p><strong>商品表：</strong></p>
<ul>
<li>推荐：product_id</li>
</ul>
<p><strong>日志表：</strong></p>
<ul>
<li>推荐：时间（按天或月分片）</li>
</ul>

<h2>25.5 分库分表问题与解决方案</h2>
<h3>25.5.1 跨分片查询 ⭐⭐⭐⭐⭐</h3>
<p><strong>问题：</strong></p>
<pre><code class="language-sql">-- 按user_id分片
-- 查询某个用户的订单（单分片查询）
SELECT * FROM orders WHERE user_id = 12345;  -- ✅ 性能好
<p>-- 查询所有待支付订单（跨分片查询）
SELECT * FROM orders WHERE status = 'PENDING';  -- ❌ 性能差
-- 需要查询所有分片，然后合并结果
</code></pre></p>
<p><strong>解决方案1：冗余数据</strong></p>
<pre><code class="language-sql">-- 订单表按user_id分片
-- 订单状态表按status分片
-- 两个表冗余存储
<p>-- 查询用户订单
SELECT * FROM orders WHERE user_id = 12345;</p>
<p>-- 查询待支付订单
SELECT * FROM order_status WHERE status = 'PENDING';
</code></pre></p>
<p><strong>解决方案2：使用ES等搜索引擎</strong></p>
<pre><code>MySQL：存储数据
Elasticsearch：搜索数据
<p>-- 复杂查询走ES
SELECT * FROM orders WHERE status = 'PENDING' AND amount &gt; 100;
</code></pre></p>
<p><strong>解决方案3：汇总表</strong></p>
<pre><code class="language-sql">-- 定期将分片数据汇总到一个表
-- 用于统计和报表查询
<p>-- 汇总表
CREATE TABLE orders_summary (
date DATE,
status VARCHAR(20),
count INT,
total_amount DECIMAL(10,2)
);
</code></pre></p>
<hr>
<h3>25.5.2 分布式事务 ⭐⭐⭐⭐⭐</h3>
<p><strong>问题：</strong></p>
<pre><code class="language-java">// 跨分片事务
@Transactional
public void transfer(Long fromUserId, Long toUserId, BigDecimal amount) {
    // fromUserId和toUserId可能在不同分片
    accountMapper.deduct(fromUserId, amount);  // 分片1
    accountMapper.add(toUserId, amount);       // 分片2
    // 如何保证事务一致性？
}
</code></pre>
<p><strong>解决方案1：避免跨分片事务</strong></p>
<pre><code class="language-java">// 设计时避免跨分片事务
// 例如：账户表和订单表都按user_id分片
// 同一个用户的数据在同一个分片
</code></pre>
<p><strong>解决方案2：两阶段提交（2PC）</strong></p>
<pre><code class="language-java">// ShardingSphere支持XA事务
@ShardingSphereTransactionType(TransactionType.XA)
@Transactional
public void transfer(Long fromUserId, Long toUserId, BigDecimal amount) {
    accountMapper.deduct(fromUserId, amount);
    accountMapper.add(toUserId, amount);
}
<p>// 缺点：性能差，不推荐
</code></pre></p>
<p><strong>解决方案3：柔性事务（Saga/TCC）</strong></p>
<pre><code class="language-java">// 使用Seata等分布式事务框架
@GlobalTransactional
public void transfer(Long fromUserId, Long toUserId, BigDecimal amount) {
    accountService.deduct(fromUserId, amount);  // 本地事务
    accountService.add(toUserId, amount);       // 本地事务
}
<p>// 优点：性能好
// 缺点：实现复杂
</code></pre></p>
<p><strong>解决方案4：最终一致性</strong></p>
<pre><code class="language-java">// 使用消息队列保证最终一致性
public void transfer(Long fromUserId, Long toUserId, BigDecimal amount) {
    // 1. 扣款
    accountMapper.deduct(fromUserId, amount);
<pre><code>// 2. 发送消息
mqProducer.send(new TransferMessage(toUserId, amount));
</code></pre>
<p>}</p>
<p>// 消费者
@RabbitListener(queues = &quot;transfer&quot;)
public void handleTransfer(TransferMessage msg) {
// 3. 加款
accountMapper.add(msg.getUserId(), msg.getAmount());
}
</code></pre></p>

<h3>25.5.3 全局唯一ID ⭐⭐⭐⭐⭐</h3>
<p><strong>问题：</strong></p>
<pre><code class="language-sql">-- 分库分表后，自增ID会重复
-- 分片1：id = 1, 2, 3, ...
-- 分片2：id = 1, 2, 3, ...  -- 重复！
</code></pre>
<p><strong>解决方案1：UUID</strong></p>
<pre><code class="language-java">String id = UUID.randomUUID().toString();
// 优点：简单
// 缺点：无序、占用空间大、索引性能差
</code></pre>
<p><strong>解决方案2：雪花算法（Snowflake）</strong></p>
<pre><code class="language-java">// 64位ID
// 1位符号位 + 41位时间戳 + 10位机器ID + 12位序列号
<p>SnowflakeIdWorker idWorker = new SnowflakeIdWorker(1, 1);
long id = idWorker.nextId();</p>
<p>// 优点：有序、性能好
// 缺点：依赖时钟
</code></pre></p>
<p><strong>解决方案3：数据库号段模式</strong></p>
<pre><code class="language-sql">-- 创建ID生成表
CREATE TABLE id_generator (
    biz_type VARCHAR(50) PRIMARY KEY,
    max_id BIGINT NOT NULL,
    step INT NOT NULL
);
<p>-- 获取ID
UPDATE id_generator SET max_id = max_id + step WHERE biz_type = 'order';

<h2>25.6 分库分表最佳实践</h2>
<h3>25.6.1 设计原则</h3>
<pre><code>1. 能不分就不分
   - 单表&lt;1000万，不需要分表
   - 优化SQL、添加索引、升级硬件
<ol start="2">
<li>
<p>优先垂直拆分</p>
<ul>
<li>按业务拆分（用户库、订单库、商品库）</li>
<li>简单、易维护</li>
</ul>
</li>
<li>
<p>再考虑水平拆分</p>
<ul>
<li>单表&gt;1000万时考虑</li>
<li>选择合适的分片键</li>
</ul>
</li>
<li>
<p>避免跨分片查询</p>
<ul>
<li>设计时考虑查询场景</li>
<li>冗余数据或使用ES</li>
</ul>
</li>
<li>
<p>避免跨分片事务</p>
<ul>
<li>设计时避免</li>
<li>或使用分布式事务</li>
</ul>
</li>
<li>
<p>预留扩容空间</p>
<ul>
<li>分片数量设置为2的幂次（2、4、8、16）</li>
<li>方便扩容
</code></pre></li>
</ul>
</li>
</ol>
<h3>25.6.2 常见错误</h3>
<pre><code class="language-sql">-- ❌ 错误1：分片键选择不当
-- 问题：导致大量跨分片查询
-- 解决：选择高频查询字段作为分片键
<p>-- ❌ 错误2：分片数量过多
-- 问题：管理复杂、连接数过多
-- 解决：根据数据量合理规划</p>
<p>-- ❌ 错误3：不考虑扩容
-- 问题：扩容时需要重新分片
-- 解决：使用一致性哈希或预留分片</p>
<p>-- ❌ 错误4：忽略分布式事务
-- 问题：数据不一致
-- 解决：避免跨分片事务或使用分布式事务</p>
<p>-- ❌ 错误5：不监控数据倾斜
-- 问题：某些分片数据过多
-- 解决：监控各分片数据量，及时调整
</code></pre></p>
