<h2>32.1 基础知识</h2>
<h3>32.1.1 MySQL架构</h3>
<p><strong>Q1：MySQL的架构分为哪几层？</strong></p>
<pre><code>A：MySQL架构分为4层：
1. 连接层：处理客户端连接、认证
2. 服务层：SQL解析、优化、缓存
3. 引擎层：存储引擎（InnoDB、MyISAM等）
4. 存储层：数据存储在磁盘
</code></pre>
<p><strong>Q2：InnoDB和MyISAM的区别？</strong></p>
<pre><code>A：主要区别：
1. 事务：InnoDB支持，MyISAM不支持
2. 锁：InnoDB行锁，MyISAM表锁
3. 外键：InnoDB支持，MyISAM不支持
4. 崩溃恢复：InnoDB支持，MyISAM不支持
5. MVCC：InnoDB支持，MyISAM不支持
6. 性能：InnoDB适合写多，MyISAM适合读多
<p>推荐：统一使用InnoDB
</code></pre></p>
<hr>
<h2>32.2 索引相关</h2>
<h3>32.2.1 索引基础</h3>
<p><strong>Q3：什么是索引？有什么作用？</strong></p>
<pre><code>A：索引是帮助MySQL高效获取数据的数据结构。
作用：
1. 加快查询速度
2. 减少磁盘I/O
3. 加速排序和分组
<p>缺点：</p>
<ol>
<li>占用空间</li>
<li>降低写入性能</li>
<li>需要维护
</code></pre></li>
</ol>
<p><strong>Q4：MySQL索引使用什么数据结构？为什么？</strong></p>
<pre><code>A：B+树
<p>原因：</p>
<ol>
<li>B+树高度低（3-4层），减少磁盘I/O</li>
<li>叶子节点存储所有数据，范围查询效率高</li>
<li>叶子节点有指针，支持顺序访问</li>
<li>非叶子节点只存储索引，可以存储更多索引</li>
</ol>
<p>为什么不用B树？</p>
<ul>
<li>B树非叶子节点也存储数据，树高更高</li>
</ul>
<p>为什么不用哈希？</p>
<ul>
<li>哈希不支持范围查询</li>
</ul>
<p>为什么不用红黑树？</p>
<ul>
<li>红黑树高度高，磁盘I/O多
</code></pre></li>
</ul>
<p><strong>Q5：什么是聚簇索引和非聚簇索引？</strong></p>
<pre><code>A：
聚簇索引（Clustered Index）：
- 索引和数据存储在一起
- InnoDB的主键索引是聚簇索引
- 一个表只有一个聚簇索引
<p>非聚簇索引（Secondary Index）：</p>
<ul>
<li>索引和数据分开存储</li>
<li>叶子节点存储主键值</li>
<li>需要回表查询</li>
</ul>
<p>回表：通过非聚簇索引查到主键，再通过主键查数据
</code></pre></p>
<p><strong>Q6：什么是覆盖索引？</strong></p>
<pre><code>A：查询的列都在索引中，不需要回表。
<p>示例：
CREATE INDEX idx_name_age ON users(name, age);
SELECT name, age FROM users WHERE name = '张三';  -- 覆盖索引，不需要回表</p>
<p>SELECT name, age, email FROM users WHERE name = '张三';  -- 需要回表
</code></pre></p>
<p><strong>Q7：什么是最左前缀原则？</strong></p>
<pre><code>A：复合索引从最左边开始匹配。
<p>示例：
CREATE INDEX idx_abc ON table(a, b, c);</p>
<p>✅ 可以使用索引：
WHERE a = 1
WHERE a = 1 AND b = 2
WHERE a = 1 AND b = 2 AND c = 3</p>
<p>❌ 不能使用索引：
WHERE b = 2
WHERE c = 3
WHERE b = 2 AND c = 3
</code></pre></p>
<h3>32.2.2 索引失效</h3>
<p><strong>Q8：哪些情况会导致索引失效？</strong></p>
<pre><code>A：
1. 在索引列上使用函数
   WHERE YEAR(create_time) = 2024  ❌
   WHERE create_time &gt;= '2024-01-01' AND create_time &lt; '2025-01-01'  ✅
<ol start="2">
<li>
<p>隐式类型转换
WHERE phone = 123456789  ❌（phone是VARCHAR）
WHERE phone = '123456789'  ✅</p>
</li>
<li>
<p>LIKE以%开头
WHERE name LIKE '%张%'  ❌
WHERE name LIKE '张%'  ✅</p>
</li>
<li>
<p>OR条件（其中一个列没有索引）
WHERE name = '张三' OR age = 18  ❌（age没有索引）</p>
</li>
<li>
<p>不等于（!=、&lt;&gt;）
WHERE status != 1  ❌</p>
</li>
<li>
<p>IS NULL / IS NOT NULL（视情况）
WHERE name IS NULL  ❌（可能失效）</p>
</li>
<li>
<p>NOT IN、NOT EXISTS
WHERE id NOT IN (1, 2, 3)  ❌
</code></pre></p>
</li>
</ol>

<h2>32.3 事务相关</h2>
<h3>32.3.1 事务基础</h3>
<p><strong>Q9：什么是事务？ACID是什么？</strong></p>
<pre><code>A：事务是一组SQL操作，要么全部成功，要么全部失败。
<p>ACID：</p>
<ol>
<li>Atomicity（原子性）：事务不可分割</li>
<li>Consistency（一致性）：事务前后数据一致</li>
<li>Isolation（隔离性）：事务之间互不影响</li>
<li>Durability（持久性）：事务提交后永久保存
</code></pre></li>
</ol>
<p><strong>Q10：MySQL的事务隔离级别有哪些？</strong></p>
<pre><code>A：4个隔离级别（从低到高）：
<ol>
<li>
<p>READ UNCOMMITTED（读未提交）</p>
<ul>
<li>可以读到未提交的数据</li>
<li>问题：脏读、不可重复读、幻读</li>
</ul>
</li>
<li>
<p>READ COMMITTED（读已提交）</p>
<ul>
<li>只能读到已提交的数据</li>
<li>问题：不可重复读、幻读</li>
<li>Oracle默认</li>
</ul>
</li>
<li>
<p>REPEATABLE READ（可重复读）</p>
<ul>
<li>同一事务中多次读取结果一致</li>
<li>问题：幻读（InnoDB通过MVCC+间隙锁解决）</li>
<li>MySQL默认 ✅</li>
</ul>
</li>
<li>
<p>SERIALIZABLE（串行化）</p>
<ul>
<li>最高隔离级别，完全串行执行</li>
<li>问题：性能差</li>
</ul>
</li>
</ol>
<p>推荐：REPEATABLE READ
</code></pre></p>
<p><strong>Q11：什么是脏读、不可重复读、幻读？</strong></p>
<pre><code>A：
脏读（Dirty Read）：
- 读到其他事务未提交的数据
- 示例：事务A修改但未提交，事务B读到了修改后的数据
<p>不可重复读（Non-Repeatable Read）：</p>
<ul>
<li>同一事务中多次读取结果不一致</li>
<li>示例：事务A读取两次，中间事务B修改并提交，两次结果不同</li>
</ul>
<p>幻读（Phantom Read）：</p>
<ul>
<li>同一事务中多次查询，记录数不一致</li>
<li>示例：事务A查询两次，中间事务B插入新记录，第二次多了记录
</code></pre></li>
</ul>
<p><strong>Q12：什么是MVCC？</strong></p>
<pre><code>A：MVCC（Multi-Version Concurrency Control）多版本并发控制
<p>原理：</p>
<ol>
<li>每行记录有两个隐藏列：trx_id（事务ID）、roll_pointer（回滚指针）</li>
<li>每次修改生成新版本，旧版本通过undo log保存</li>
<li>读取时根据事务ID判断可见性</li>
</ol>
<p>优点：</p>
<ul>
<li>读不加锁，提高并发性能</li>
<li>解决不可重复读问题</li>
</ul>
<p>InnoDB通过MVCC+间隙锁解决幻读
</code></pre></p>
<hr>
<h2>32.4 锁相关</h2>
<h3>32.4.1 锁类型</h3>
<p><strong>Q13：MySQL有哪些锁？</strong></p>
<pre><code>A：
按粒度分：
1. 表锁：锁整个表（MyISAM）
2. 行锁：锁单行（InnoDB）
3. 间隙锁：锁范围（InnoDB）
<p>按类型分：</p>
<ol>
<li>共享锁（S锁）：读锁，多个事务可以同时持有</li>
<li>排他锁（X锁）：写锁，只有一个事务可以持有</li>
</ol>
<p>按算法分：</p>
<ol>
<li>Record Lock：记录锁</li>
<li>Gap Lock：间隙锁</li>
<li>Next-Key Lock：记录锁+间隙锁
</code></pre></li>
</ol>
<p><strong>Q14：什么是死锁？如何避免？</strong></p>
<pre><code>A：
死锁：两个或多个事务互相等待对方释放锁。
<p>示例：
事务A：锁住行1，等待行2
事务B：锁住行2，等待行1</p>
<p>避免方法：</p>
<ol>
<li>按相同顺序访问资源</li>
<li>减少锁持有时间</li>
<li>使用乐观锁</li>
<li>降低隔离级别</li>
<li>设置锁等待超时：innodb_lock_wait_timeout
</code></pre></li>
</ol>

<h2>32.5 性能优化</h2>
<h3>32.5.1 SQL优化</h3>
<p><strong>Q15：如何优化慢查询？</strong></p>
<pre><code>A：
1. 分析执行计划（EXPLAIN）
2. 添加索引
3. 避免SELECT *
4. 避免索引失效
5. 优化JOIN
6. 分页优化
7. 使用覆盖索引
8. 避免子查询，改用JOIN
</code></pre>
<p><strong>Q16：如何优化深分页？</strong></p>
<pre><code>A：
问题SQL：
SELECT * FROM orders ORDER BY id LIMIT 1000000, 10;
<p>优化方案：</p>
<ol>
<li>
<p>子查询优化
SELECT * FROM orders
WHERE id &gt;= (SELECT id FROM orders ORDER BY id LIMIT 1000000, 1)
LIMIT 10;</p>
</li>
<li>
<p>延迟关联
SELECT o.* FROM orders o
INNER JOIN (SELECT id FROM orders ORDER BY id LIMIT 1000000, 10) t
ON <a href="http://o.id">o.id</a> = <a href="http://t.id">t.id</a>;</p>
</li>
<li>
<p>记录上次最大ID
SELECT * FROM orders WHERE id &gt; @last_id ORDER BY id LIMIT 10;
</code></pre></p>
</li>
</ol>
<h3>32.5.2 参数优化</h3>
<p><strong>Q17：重要的MySQL参数有哪些？</strong></p>
<pre><code>A：
1. innodb_buffer_pool_size：缓冲池大小（50-70%内存）
2. innodb_log_file_size：redo log大小
3. innodb_flush_log_at_trx_commit：刷盘策略（1:最安全，2:性能好）
4. sync_binlog：binlog刷盘策略
5. max_connections：最大连接数
6. query_cache_size：查询缓存（MySQL 8.0已移除）
</code></pre>

<h2>32.6 高可用</h2>
<h3>32.6.1 主从复制</h3>
<p><strong>Q18：MySQL主从复制原理？</strong></p>
<pre><code>A：
1. 主库写入binlog
2. 从库I/O线程读取binlog，写入relay log
3. 从库SQL线程执行relay log
<p>三个线程：</p>
<ul>
<li>主库：binlog dump线程</li>
<li>从库：I/O线程、SQL线程
</code></pre></li>
</ul>
<p><strong>Q19：主从延迟如何解决？</strong></p>
<pre><code>A：
原因：
1. 从库性能差
2. 主库写入量大
3. 大事务
4. 网络延迟
<p>解决方案：</p>
<ol>
<li>升级从库硬件</li>
<li>并行复制（MySQL 5.7+）</li>
<li>拆分大事务</li>
<li>读写分离，读从库时容忍延迟</li>
<li>重要查询读主库
</code></pre></li>
</ol>
