# 看完 20 本编程书还是不会写项目，这个 53 万星仓库救了我

> "What I cannot create, I do not understand." —— Richard Feynman

你有没有过这种体验：

- 跟着视频课刷完了 100 集，讲的知识点都听得懂，自己动手写项目还是卡壳？
- 厚厚的编程大部头翻完了，合上书脑子里还是一团浆糊，问你「这个组件到底怎么工作的」说不清楚？
- 简历上写「熟悉 XX 框架」，面试官追问一句「你能说说它底层是怎么实现的吗」直接哑口无言？

我曾经也陷在这个怪圈里出不来。看了很多书，收藏了很多教程，真到上手写东西的时候，还是会被各种细节卡住。直到朋友甩给我这个 GitHub 仓库，才突然想通了问题出在哪。

![build-your-own-x 覆盖哪些技术领域](/images/build-your-own-x-programming-by-doing/01-category-grid.png)

## 为什么说「看书一百遍，不如动手造一遍」？

传统的编程学习路径是这样的：**先学理论，再学概念，最后做项目**。但大部分人卡就在「最后」这一步。理论都懂，概念都背，真要把这些碎片拼成一个能跑的东西，瞬间就不知道从哪下手了。

这个仓库的核心逻辑非常简单：**反过来学**。

你想理解操作系统？别先啃 1000 页的《深入理解计算机系统》，动手写一个简单内核出来。你想理解编译器？别先背《编译原理》的龙书那套术语，从写一个简单的 Lisp 解释器开始。你想理解 React？别光看官方文档，自己写一个 160 行的 React 出来。

这其实就是费曼学习法的工程化落地：**不能创造，就不算真的理解**。

`codecrafters-io/build-your-own-x` 现在已经有 **530,000+ stars**，收集了 **500+ 份** step-by-step 的教程，教你从零开始构建各种各样的技术。

## 这个学习方法真的有效吗？我用三个例子给你算笔账

我自己试过几个小项目，最大的感受是：**之前看书看不明白的地方，动手写一遍瞬间就通了**。

举三个最热门的例子：

### 例子一：写一个你自己的 Git

你天天用 Git，但你真的理解 Git 是怎么工作的吗？`git add` 到底干了什么？对象存储是什么结构？这个仓库里有好几个教程：

- **Python 版**：[Write yourself a Git!](https://wyag.thb.lt/) — 不到 1000 行 Python 就能实现一个可工作的 Git，支持 `init`、`add`、`commit`、`log` 这些基础命令。
- **JavaScript 版**：[Gitlet](http://gitlet.maryrosecook.com/docs/gitlet.html) — 浏览器里就能跑的 Git 实现。

跟着写完一遍，你再看 Git 的底层文档，那些「对象」「树」「提交」这些概念瞬间就活了。之前是死记硬背，现在是你自己写出来的，想忘都忘不了。

### 例子二：写一个你自己的 Redis

Redis 为什么这么快？数据结构到底怎么实现的？这个仓库里也有教程：

- **C 语言版**：[Build Your Own Redis from Scratch](https://build-your-own.org/redis) — 完整的系列教程，从网络编程到数据结构，一步步教你写一个 Redis 服务器。
- **Python 版**：[Write your own miniature Redis with Python](http://charlesleifer.com/blog/building-a-simple-redis-server-with-python/) — 更短的入门版，不到 500 行就能跑起来。

我见过很多人简历上写「熟练使用 Redis」，问他「跳表是什么结构」都说不清楚。要是你自己实现过一遍基本的 KV 存储，这个问题闭着眼都能答。

### 例子三：写一个你自己的 React

现在前端几乎人人用 React，但你真的理解 `Virtual DOM` 是什么吗？`setState` 为什么是异步的？这个仓库里有：

- **Didact**：[A DIY guide to build your own React](https://github.com/hexacta/didact) — 非常经典的教程，一步步教你从零实现 React 的核心逻辑。
- **Gooact**：[React in 160 lines of JavaScript](https://medium.com/@sweetpalma/gooact-react-in-160-lines-of-javascript-44e0742ad60f) — 更短的版本，真的只有 160 行。

看完源码不如写一遍 160 行的最小 React，写完你再去看 React 源码，脉络一下子就清晰了。

## 适合谁？不适合谁？给你一个清晰的判断标准

我觉得不是所有人都适合用这个方法，得看你处于什么阶段：

### ✅ 非常适合这些人

1. **已经入门，但想突破瓶颈** — 你会写 CRUD，也能用框架搭项目，但对底层原理一知半解，想往上走又不知道从哪开始。
2. **基础知识零散，希望建立体系** — 碎片化看了很多博客视频，知识点在脑子里是散的，通过完整造一个轮子能把这些碎片串起来。
3. **面试准备，想把知识点吃透** — 面试的时候面试官特别喜欢问「讲讲 XX 的原理」，你自己实现过一遍，说出来的东西和背书完全不是一个Level。

### ❌ 可能不适合这些人

1. **纯零基础，第一门编程语言** — 完全没基础上来就造轮子，挫败感会很强，建议还是先过一遍基础语法再试。
2. **赶项目，需要快速出活** — 这个方法是**深度学习**，不是快速开发。如果你的目标是两周上线一个产品，老老实实拿成熟框架用就好。
3. **只是想了解概念，并不打算深究** — 如果你只是想知道「Redis 大概是干什么的」，看一篇 10 分钟的概述文章就够了，不需要动手写。

## 怎么开始？给你一个可直接复制的四步走路线

我自己总结了一个比较舒服的节奏，分享给你：

### 第一步：选一个你"天天用但不太懂"的小东西

别一开始就选「写一个操作系统」这种大项目，找一个你天天接触但原理不太清楚的小工具入手。推荐几个门槛不高收获很大的起点：

- 前端：写一个自己的 React（160 行） / Virtual DOM / 简单的 Promise 实现
- 后端：写一个自己的 Git（~1000 行 Python） / 简单 Web 服务器
- 底层：写一个简单的哈希表 / 内存分配器 / Lisp 解释器

关键是**小而完整**，能在一个周末做完，拿到正反馈。

### 第二步：别先看源码，自己先想"如果我来写，我会怎么做"

拿到教程之后，先别急着跟着抄。花 15 分钟想一想：这个问题如果让我来做，我会分成几步？核心数据结构怎么设计？哪些地方可能会踩坑？

比如写 Git，你先想一想：Git 存文件到底是存在哪？commit 信息怎么串起来？想一遍再看作者的实现，收获会大很多。

### 第三步：跟着教程敲，遇到卡壳先自己想 10 分钟

不要复制粘贴，真的一行一行敲。遇到报错了，先自己 Google 解决，解决不了再看教程是怎么处理的。这个过程虽然慢，但**解决问题的能力就是这么练出来的**。

### 第四步：做完之后，写一篇总结

不需要太长，就回答三个问题：

1. 这个东西核心解决了什么问题？
2. 哪个地方我卡了很久，最后是怎么想通的？
3. 有哪个设计技巧让你觉得"居然可以这样"？

写下来，不管是发博客还是存在自己笔记里，三个月后再看，印象还很深。

## 最后：这个仓库最珍贵的不是教程，是思维方式

我觉得 `build-your-own-x` 最珍贵的地方，不是它收集了 500 份教程，而是它把「费曼学习法」从一句正确的废话，变成了一件**每个人都能上手做的具体事情**。

你不需要等，不需要攒够了"足够多的知识"再开始，现在就能找一个小题目动手做。做着做着，你就会发现：**原来那些看上去高大上的技术，拆解之后也没那么神秘**。

> 「不能创造，就不算理解。」

这句话放在今天这个 AI 能帮你写代码的时代，反而更有意义。AI 能帮你写代码，但理解为什么这么写，只能靠你自己。
